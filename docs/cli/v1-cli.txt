V1 CLI — Reduced Scope
=======================

V1 proves one thing: the tool can index one TypeScript repo, build a correct
structural graph, and answer deterministic queries over it.

Nothing beyond this scope ships in v1. If the graph is wrong, everything built
on top of it is wrong.

Every command supports --json for machine consumption.
Default output is human-readable tables.

Tool name: TBD (using "tool" as placeholder)


================================================================================
REPO MANAGEMENT
================================================================================

tool repo add <path> [--name <alias>]

  Register a local repository for indexing.
  Stores the absolute path, assigns a stable repo ID.
  Does NOT index automatically — indexing is explicit.

  Example:
    tool repo add /home/dev/payments-service --name payments
    tool repo add .

  Output:
    repo_id, name, path, status (registered)


tool repo index <repo>

  Full parse of the repository. Creates snapshot S0.
  Walks all source files (respecting .gitignore and exclusion rules).
  Runs the language extractor (TS for v1) on each file.
  Runs any detected framework extractors.
  Stores all nodes and edges in SQLite attached to the new snapshot.

  <repo> can be the repo name, repo ID, or path.

  Flags:
    --exclude <glob>        Additional exclusion patterns (e.g., --exclude "*.test.ts")
    --include <glob>        Only include matching files
    --verbose               Show progress (files parsed, nodes/edges found)

  Example:
    tool repo index payments
    tool repo index payments --exclude "**/__tests__/**" --verbose

  Output:
    snapshot_id, files_parsed, symbols_found, edges_found, duration_ms


tool repo status <repo>

  Show current state of a repo: last snapshot, stale files, pending refresh.

  Example:
    tool repo status payments

  Output:
    repo_id, name, path, last_snapshot, snapshot_commit, stale_files_count,
    total_files, total_symbols, total_edges


tool repo refresh <repo>

  Re-index the repository and create a new refresh snapshot linked to the
  previous one. In v1 this performs a full re-extraction (same as index) but
  records the result as a refresh snapshot with a parent link, enabling
  snapshot comparison in future versions.

  True incremental refresh (only re-parsing changed files based on content
  hash comparison) is deferred to v2.

  Example:
    tool repo refresh payments

  Output:
    snapshot_uid, files_total, nodes_total, edges_total,
    edges_unresolved, duration_ms


tool repo list

  List all registered repositories.

  Flags:
    --json                  Machine-readable output

  Example:
    tool repo list
    tool repo list --json

  Output per repo:
    name, path (human) or repo_uid, name, path (JSON)


tool repo remove <repo>

  Unregister a repository. Does not delete the SQLite data unless
  explicitly removed.

  <repo> can be the repo name, repo ID, or path.

  Flags:
    --json                  Machine-readable output

  Example:
    tool repo remove payments


================================================================================
GRAPH QUERIES
================================================================================

All graph queries return results attached to the current snapshot.
All support --json.
All return evidence sources and edge resolution (static/dynamic/inferred).


tool graph callers <repo> <symbol-id> [--depth <n>] [--edge-types <types>]

  Find all direct callers of a symbol. With --depth, find transitive callers
  up to N levels deep.

  <symbol-id> is a qualified name (e.g., "PaymentService.retry") or a
  node ID from a previous query result.

  Flags:
    --depth <n>             Transitive callers up to depth N (default: 1)
    --edge-types <types>    Comma-separated edge types to follow
                            (default: CALLS)
                            Common values: CALLS, INSTANTIATES, IMPLEMENTS,
                            IMPORTS, OWNS
    --json                  Machine-readable output

  Example:
    tool graph callers payments PaymentService.retry
    tool graph callers payments PaymentService.retry --depth 3 --json
    tool graph callers payments TreeStore --edge-types CALLS,INSTANTIATES

  Output per result:
    node_id, symbol_name, file, line, edge_type, resolution, evidence[], depth


tool graph callees <repo> <symbol-id> [--depth <n>] [--edge-types <types>]

  Find all symbols called by this symbol. With --depth, find transitive callees.

  Flags:
    --depth <n>             Transitive callers up to depth N (default: 1)
    --edge-types <types>    Comma-separated edge types (default: CALLS)
    --json                  Machine-readable output

  Example:
    tool graph callees payments CheckoutService.process --depth 2
    tool graph callees payments UserService --edge-types CALLS,INSTANTIATES

  Output: same shape as callers


tool graph imports <repo> <file-or-module> [--depth <n>]

  Find all modules/files imported by the target. With --depth, find transitive
  import chains.

  Example:
    tool graph imports payments src/services/payment.ts
    tool graph imports payments src/services/ --depth 3

  Output per result:
    node_id, file, imported_from, edge_type, resolution, evidence[]


tool graph path <repo> <from-id> <to-id> [--max-depth <n>]

  Find the shortest path between two nodes in the call/import graph.
  Returns the full edge path if one exists.

  Flags:
    --max-depth <n>         Maximum traversal depth (default: 8)
    --edge-types <types>    Comma-separated edge types to follow
                            (default: CALLS,IMPORTS)

  Example:
    tool graph path payments CheckoutController.create PaymentGateway.charge
    tool graph path payments CheckoutController.create PaymentGateway.charge --edge-types CALLS

  Output:
    found (bool), path_length, path (array of edges with node_id, symbol, file, line)


tool graph dead <repo>

  Find all nodes with zero incoming edges that are NOT declared entrypoints.
  These are dead code candidates.

  Flags:
    --kind <node-kind>      Filter by node kind (FUNCTION, CLASS, etc.)
    --min-lines <n>         Only show nodes with at least N lines of code

  Example:
    tool graph dead payments
    tool graph dead payments --kind FUNCTION --min-lines 10

  Output per result:
    node_id, symbol_name, file, line, kind, lines_of_code, evidence[]


tool graph cycles <repo>

  Detect dependency cycles at the module or file level.
  Returns each cycle as an ordered list of nodes.

  Flags:
    --level <file|module>   Detection level (default: module)

  Example:
    tool graph cycles payments
    tool graph cycles payments --level file --json

  Output per cycle:
    cycle_id, length, nodes (array of node_id, file/module name)


tool graph stats <repo>

  Module structural metrics. Computes fan-in, fan-out, instability,
  abstractness, and distance from main sequence for each module that
  contains at least one source file.

  Definitions:
    fan_in       — number of modules that import this module
    fan_out      — number of modules this module imports
    instability  — fan_out / (fan_in + fan_out). 0 = stable, 1 = unstable
    abstractness — interfaces and type aliases / total type declarations
    distance     — |abstractness + instability - 1|. 0 = on main sequence

  Flags:
    --json                  Machine-readable output

  Example:
    tool graph stats payments
    tool graph stats payments --json

  Output per module:
    module, fan_in, fan_out, instability, abstractness,
    distance_from_main_sequence, file_count, symbol_count

  Summary (human output only):
    module count, avg instability, avg distance, max fan-in, max fan-out


tool arch violations <repo>

  Find IMPORTS edges that cross declared forbidden boundaries.
  Scoped to IMPORTS edges only (call graph is not used for policy
  enforcement). Returns each violation with source file, target file,
  line, and the boundary rule that was violated.

  Flags:
    --json                  Machine-readable output

  Example:
    tool arch violations payments
    tool arch violations payments --json

  Output per violation:
    boundary_module, forbidden_module, reason, source_file, target_file, line


tool graph metrics <repo>

  Function-level complexity metrics from stored measurements.
  Queries the measurements table for cyclomatic complexity, parameter
  count, and max nesting depth computed during indexing.

  Counting rules:
    cyclomatic_complexity — base 1, +1 per decision point:
      if, else if, for, for-of/in, while, do-while, catch,
      switch case (not default), ternary, &&, ||, ??
    parameter_count — required + optional + rest parameters
    max_nesting_depth — deepest control flow nesting:
      if, for, while, do, switch, try, catch
      else-if chains count as same level (not stacking)
      nested function bodies are excluded

  Flags:
    --sort <field>          Sort by: cc (default), params, nesting
                            (function mode only; module mode sorts by max CC)
    --limit <n>             Max results to return (applies in both modes)
    --module                Aggregate per module instead of per function
    --json                  Machine-readable output

  Example:
    tool graph metrics payments --limit 20
    tool graph metrics payments --sort nesting --limit 10
    tool graph metrics payments --module
    tool graph metrics payments --json

  Output per function (default):
    symbol, file, line, cyclomatic_complexity, parameter_count,
    max_nesting_depth

  Output per module (--module):
    module, function_count, avg_cyclomatic_complexity,
    max_cyclomatic_complexity, avg_nesting_depth, max_nesting_depth

  Summary (human output only, function mode):
    function count, avg cyclomatic complexity, max cyclomatic complexity


tool graph versions <repo>

  Extracted domain versions from manifest files.
  Currently extracts package name and version from package.json
  at the repository root.

  These are domain versions — what the system claims to be — distinct
  from repo-graph's internal toolchain provenance.

  Flags:
    --json                  Machine-readable output

  Example:
    tool graph versions payments
    tool graph versions payments --json

  Output per version:
    kind (package_name | package_version), value, source_file


tool graph churn <repo>

  Import and display per-file git churn measurements.
  Reads git commit history for the specified time window and stores
  change_frequency (commit count) and churn_lines (lines added + deleted)
  as measurements attached to FILE nodes.

  Measurement definitions:
    change_frequency — number of commits that modified the file in the window
    churn_lines — total lines added + deleted in the window

  Flags:
    --since <period>        Time window in git date format (default: 90.days.ago)
    --limit <n>             Max results to display
    --json                  Machine-readable output

  Example:
    tool graph churn payments --since 30.days.ago
    tool graph churn payments --since 365.days.ago --limit 20
    tool graph churn payments --json

  Output per file:
    file, commit_count, lines_changed

  Note: requires the repo root to be a git repository.
  Non-git directories return empty results.


tool graph coverage <repo> <report>

  Import coverage from a supported report format (auto-detected).
  Currently supports Istanbul/c8 JSON (coverage-final.json).
  Additional formats (JaCoCo, lcov, Cobertura) can be added as adapters
  behind the CoverageImporterPort without changing this command.
  Reads per-file statement, function, and branch coverage from the report,
  rebases paths to repo-relative, filters to indexed files, and stores
  as measurements. Re-running is idempotent.

  Measurement kinds:
    line_coverage     — statements covered / total (0.0 to 1.0)
    function_coverage — functions covered / total (0.0 to 1.0)
    branch_coverage   — branches covered / total (0.0 to 1.0)

  Each measurement value includes covered count and total for transparency.

  Flags:
    --json                  Machine-readable output

  Example:
    tool graph coverage payments ./coverage/coverage-final.json
    tool graph coverage payments ./coverage/coverage-final.json --json

  Output per file:
    file, line%, func%, branch%

  Note: only files that exist in the indexed snapshot are imported.
  Files in the report but not in the graph are silently skipped.


tool trend <repo>

  Compare health metrics between the latest snapshot and its parent.
  Shows whether the codebase is improving or decaying across measured
  dimensions.

  Checks toolchain comparability before computing deltas. Reports
  NOT_COMPARABLE when measurement semantics, extraction semantics,
  or stable key format differ between snapshots.

  Metrics tracked:
    files, nodes, edges (neutral — no judgment)
    cycles (lower = better)
    violations (lower = better)
    avg_complexity, max_complexity (lower = better)
    functions_measured (neutral)
    avg_coverage (higher = better)
    hotspot_files (lower = better)

  Delta directions: improved, worsened, unchanged, new, no_data

  Flags:
    --json                  Machine-readable output

  Example:
    tool trend payments
    tool trend payments --json

  Prerequisites:
    At least two snapshots must exist (run `repo index` then `repo refresh`).

  Known limitation:
    Coverage and churn measurements are snapshot-scoped but use
    repo-level file inventory for filtering. Fine-grained historical
    deltas may be imprecise until snapshot-scoped FILE matching is
    implemented.


tool change impact <repo> [--against-snapshot | --staged | --since <ref>] [--max-depth <n>]

  Compute the set of modules affected by a file-change set, propagating
  over REVERSE MODULE IMPORTS edges from the modules owning the changed
  files. Output includes seed modules, impacted modules with distance,
  and honest trust metadata.

  Scope (mutually exclusive):
    --against-snapshot    Diff working tree vs the snapshot's
                          basis_commit. DEFAULT. Fails if the snapshot
                          has no basis_commit (repo indexed from a
                          non-git directory).
    --staged              Diff staged changes (git diff --cached).
    --since <ref>         Diff working tree vs the given git ref
                          (commit, branch, tag).

  Flags:
    --max-depth <n>       Cap reverse-IMPORTS traversal depth
                          (default: unbounded)
    --json                Machine-readable output

  Graph basis: reverse MODULE IMPORTS only. This is a deliberate
  conservative first slice. CALLS edges, dynamic dispatch, and
  registry/framework liveness are NOT modeled. See the trust.caveats
  array in output for the full honesty boundary.

  Example:
    tool change impact payments
    tool change impact payments --staged --json
    tool change impact payments --since main --max-depth 3

  JSON output includes:
    command, repo, scope (echoed), snapshot_uid, snapshot_basis_commit,
    trust (graph_basis, calls_included, caveats[]),
    changed_files[] (path, matched_to_index, owning_module,
                     unmatched_reason),
    seed_modules[] (owning modules of matched changed files),
    impacted_modules[] (module, distance, reason = seed | reverse_import),
    counts (changed_files, changed_files_matched,
            changed_files_unmatched, seed_modules, impacted_modules,
            max_distance)

  Design principle: impact analysis is easy to make dishonest. This
  command emits trust metadata as first-class output so consumers
  (CI, agents) can tell what was and was NOT considered. Extending
  the propagation basis (e.g. adding CALLS) will require updating
  trust.graph_basis and trust.calls_included explicitly.


tool evidence <repo>

  Verification status report. Produces a snapshot-level view of all
  requirements, their obligations, computed verdicts, waiver overlays,
  and effective verdicts. This is the Tier 1 read surface that makes
  the requirement-to-evidence closure loop visible.

  Recomputes verdicts on demand and applies active waivers via the
  effective-verdict resolver. Includes snapshot and toolchain metadata
  for traceability.

  Verdict model: five-state (PASS, FAIL, MISSING_EVIDENCE, UNSUPPORTED,
  WAIVED). WAIVED is only an EFFECTIVE state. Every obligation emits
  both computed_verdict (truth) and effective_verdict (policy-adjusted).
  Normative contract: docs/architecture/gate-contract.txt.

  Requirement-level status (derived from EFFECTIVE verdicts):
    SATISFIED        — all obligations are PASS or WAIVED
    INCOMPLETE       — at least one effective FAIL
    PARTIAL          — no FAIL, at least one MISSING_EVIDENCE
    UNVERIFIABLE     — no FAIL/MISSING, at least one UNSUPPORTED
    NO OBLIGATIONS   — no verification entries defined

  Flags:
    --req-id <id>           Filter to a specific requirement
    --json                  Machine-readable output

  Example:
    tool evidence payments
    tool evidence payments --req-id REQ-001
    tool evidence payments --json

  JSON output includes:
    command, repo, snapshot, toolchain (full provenance),
    requirements (array with per-obligation computed_verdict,
      effective_verdict, evidence, waiver_basis),
    summary (aggregate pass/fail/waived/missing_evidence/unsupported counts)


tool graph risk <repo>

  Under-tested hotspots: files that are complex, frequently changed,
  AND poorly covered. Requires prior `graph hotspots` and optionally
  `graph coverage`. Produces the strongest risk signal the tool offers.

  Formula (v1):
    risk_score = hotspot_score * (1 - line_coverage)

  Files without coverage data are treated as 0% covered (maximum risk
  multiplier). A fully covered hotspot has risk 0. An uncovered
  non-hotspot also has low risk (hotspot score near 0).

  Each result includes all component values for transparency:
    file, risk_score, hotspot_score, line_coverage, churn_lines,
    sum_complexity

  Scores are stored in the inferences table (kind = "under_tested_hotspot").
  Re-running is idempotent.

  Flags:
    --limit <n>             Max results (default: 20)
    --json                  Machine-readable output

  Example:
    tool graph risk payments
    tool graph risk payments --limit 10
    tool graph risk payments --json


tool graph hotspots <repo>

  File-level hotspots: churn x complexity. Requires prior churn import
  via `graph churn`. Reads churn and complexity measurements from the
  database, computes hotspot scores, and stores them as inferences.

  Formula (v1):
    raw_score = churn_lines * sum_cyclomatic_complexity_in_file
    normalized_score = raw_score / max(raw_score)

  Each result includes all component values for transparency:
    file, normalized_score, raw_score, churn_lines, change_frequency,
    sum_complexity

  Scores are stored in the inferences table (kind = "hotspot_score")
  for later use in risk assessment (see `graph risk`)
  for later use in risk assessment and trend analysis. Re-running is
  idempotent (previous hotspot inferences are deleted first).

  Flags:
    --limit <n>             Max results (default: 20)
    --json                  Machine-readable output

  Example:
    tool graph hotspots payments
    tool graph hotspots payments --limit 10
    tool graph hotspots payments --json


================================================================================
DECLARATIONS
================================================================================

Human-supplied facts. Persistent until explicitly changed.
Stored separately from extracted facts.
Used by graph queries (e.g., dead node detection excludes declared entrypoints).


tool declare module <repo> <module-path> [flags]

  Declare metadata about a module.

  Flags:
    --purpose <text>        What this module is for
    --maturity <level>      PROTOTYPE | MATURE | PRODUCTION
    --owner <text>          Team or person responsible

  Example:
    tool declare module payments src/core/payments \
      --purpose "payment invariants and orchestration" \
      --maturity MATURE \
      --owner payments-team


tool declare boundary <repo> <module-a> --forbids <module-b>

  Declare that module-a must NOT depend on module-b.
  Used by arch violations queries (v2) and can be checked against
  the extracted graph immediately.

  Example:
    tool declare boundary payments src/core --forbids src/infrastructure
    tool declare boundary payments src/domain --forbids src/api


tool declare entrypoint <repo> <symbol-id>

  Declare that a symbol is a known entrypoint (route handler, event listener,
  cron job, public API export). Declared entrypoints are excluded from dead
  code detection.

  Example:
    tool declare entrypoint payments CheckoutController.create
    tool declare entrypoint payments PaymentEventHandler.onPaymentReceived


tool declare invariant <repo> <symbol-id> <text>

  Attach a business invariant to a symbol. Informational for now. Used by
  agent skills to assess risk during change impact analysis.

  Example:
    tool declare invariant payments PaymentService.charge \
      "duplicate settlement must not occur"


tool declare requirement <repo> <module>

  Declare a structured requirement object. First Tier 1 primitive:
  versioned engineering intent with explicit constraints and non-goals.
  Verification obligations are added via `declare obligation` (each
  obligation carries a stable obligation_id for waiver reference).

  Flags:
    --req-id <id>           Requirement identifier (e.g. REQ-001) [required]
    --objective <text>      What must be achieved [required]
    --constraint <text...>  What must not happen (repeatable)
    --non-goal <text...>    Explicitly out of scope (repeatable)
    --json                  JSON output

  Example:
    tool declare requirement payments src/core \
      --req-id REQ-001 \
      --objective "Core domain must have zero external dependencies" \
      --constraint "No imports from adapters/" \
      --constraint "No imports from cli/" \
      --non-goal "Performance optimization"

  Stored value shape:
    {
      "req_id": "REQ-001",
      "version": 1,
      "objective": "...",
      "constraints": ["..."],
      "non_goals": ["..."],
      "verification": [...],
      "status": "active"
    }


tool declare obligation <repo> <reqId>

  Add a verification obligation to an existing requirement.
  Supersedes the old requirement declaration with an updated version
  containing the new obligation.

  A stable obligation_id (UUID) is generated on creation. This id is
  the reference target for waiver declarations. Identity is frozen by
  the semantic-match tuple (method, target, threshold, operator) —
  see docs/architecture/gate-contract.txt for the full identity contract.

  Supported methods:
    arch_violations       — check that boundary violations = 0
    coverage_threshold    — check average line coverage against threshold
    complexity_threshold  — check max cyclomatic complexity against threshold
    hotspot_threshold     — check max hotspot score against threshold

  Flags:
    --obligation <text>     What must be proven [required]
    --method <method>       Verification method [required]
    --target <path>         Target module or file path
    --threshold <n>         Numeric threshold
    --operator <op>         Comparison operator (>= | <= | == | > | <)
    --json                  JSON output

  Example:
    tool declare obligation payments REQ-001 \
      --obligation "Zero boundary violations for core" \
      --method arch_violations \
      --target src/core

    tool declare obligation payments REQ-001 \
      --obligation "Line coverage >= 80% on core" \
      --method coverage_threshold \
      --target src/core \
      --threshold 0.8

  JSON output includes declaration_uid, req_id, obligation_id,
  obligation, method, version.


tool declare waiver <repo> <reqId>

  Record a waiver against a specific obligation of a requirement version.
  Waivers are VERSION-SCOPED: they reference exactly
  (req_id, requirement_version, obligation_id). When the requirement
  is superseded, the waiver remains in the audit trail but stops
  applying to the new version; a new waiver must be explicitly issued.

  Waivers suppress gate failure. They do NOT erase the underlying
  computed verdict — both computed_verdict and effective_verdict
  remain visible in all read surfaces. Normative contract:
  docs/architecture/gate-contract.txt.

  Validations:
    - referenced requirement must exist and be active
    - --requirement-version must match the current active version
      (waivers cannot be created against superseded versions)
    - --obligation-id must exist in that requirement version

  Expiry semantics: a waiver with expires_at <= now stops suppressing
  verdict (the computed verdict resumes as effective). The expired
  waiver remains queryable via `declare list --kind waiver`.

  Flags:
    --obligation-id <id>        Target obligation's obligation_id [required]
    --requirement-version <n>   Requirement version to waive against [required]
    --reason <text>             Explanation of why exception is acceptable [required]
    --expires-at <iso>          ISO 8601 expiry timestamp
    --review-by <iso>           ISO 8601 review deadline
    --rationale <category>      Categorization (e.g. tech_debt, pilot)
    --policy-basis <text>       Reference to policy/gate version
    --created-by <text>         Authorship marker (defaults to 'cli')
    --json                      JSON output

  Example:
    tool declare waiver payments REQ-001 \
      --obligation-id 9f3e-... \
      --requirement-version 2 \
      --reason "Pilot phase, tracked in ADR-003" \
      --expires-at 2026-07-01T00:00:00Z \
      --rationale pilot

  JSON output: declaration_uid, req_id, requirement_version,
  obligation_id, reason.


tool graph obligations <repo>

  Evaluate verification obligations from all active requirements
  against current measurements, then apply active waivers to produce
  effective verdicts. Emits BOTH computed_verdict (truth) and
  effective_verdict (policy-adjusted) for every obligation.

  Verdict model: five-state (PASS, FAIL, MISSING_EVIDENCE, UNSUPPORTED,
  WAIVED). WAIVED is only an EFFECTIVE state. Consistent with `gate`
  and `evidence`. Normative: docs/architecture/gate-contract.txt.

  Each verdict includes the exact evidence used for the evaluation:
  measurement values, thresholds, operators, file counts. When a
  waiver applies, waiver_basis carries the full exception audit trail.

  Flags:
    --json                  Machine-readable output

  Example:
    tool graph obligations payments
    tool graph obligations payments --json

  Output per obligation:
    req_id, req_version, obligation_id, obligation, method, target,
    threshold, operator, computed_verdict, effective_verdict,
    evidence, waiver_basis

  Counts (all against EFFECTIVE verdicts):
    pass, fail, waived, missing_evidence, unsupported


tool declare list <repo>

  List all declarations for a repo.

  Flags:
    --kind <kind>           Filter by declaration kind:
                            module | boundary | entrypoint | invariant |
                            requirement | waiver | owner | maturity

  Example:
    tool declare list payments
    tool declare list payments --kind boundary --json
    tool declare list payments --kind waiver --json


tool declare remove <repo> <declaration-id>

  Remove a declaration by ID.


================================================================================
GATE
================================================================================

tool gate <repo> [--strict | --advisory]

  CI integration surface. Evaluates all active obligations, applies
  waiver overlays, reduces to a single gate outcome, and exits with
  a code that CI pipelines can branch on.

  This is the external integration point. Full normative contract:
  docs/architecture/gate-contract.txt. Summary below.

  Verdict model (five-state):
    computed:   PASS | FAIL | MISSING_EVIDENCE | UNSUPPORTED
    effective:  computed OR WAIVED (when an active waiver applies)

  Computed verdict is never erased. Both are visible in JSON output.

  Gate modes and exit codes:

    default (no flag):
      exit 0 — all effective verdicts are PASS or WAIVED
      exit 1 — at least one effective FAIL
      exit 2 — no FAIL, but at least one MISSING_EVIDENCE or UNSUPPORTED

    --strict:
      exit 0 — all PASS or WAIVED
      exit 1 — any FAIL, MISSING_EVIDENCE, or UNSUPPORTED
      (escalates incomplete evaluability to hard fail)

    --advisory:
      exit 0 — no FAIL (MISSING/UNSUPPORTED are informational)
      exit 1 — any FAIL
      (for early-adoption CI while evidence infrastructure is built)

  The three-exit default preserves the distinction between
  "judged and failed" (1) and "could not reach a verdict" (2).

  Flags:
    --strict                Escalate MISSING/UNSUPPORTED to fail
    --advisory              Treat MISSING/UNSUPPORTED as informational
    --json                  Machine-readable output

  Example:
    tool gate payments
    tool gate payments --strict --json
    tool gate payments --advisory

  JSON output:
    command, repo, snapshot, toolchain,
    obligations[] (each with req_id, req_version, obligation_id,
      obligation, method, target, threshold, operator,
      computed_verdict, effective_verdict, evidence, waiver_basis),
    gate { outcome, exit_code, mode, counts }

  Process exit code = gate.exit_code.


================================================================================
RESPONSE CONTRACT
================================================================================

Every graph query --json response follows this envelope.
Repo management and declare commands return operation-specific JSON
structures (not wrapped in this envelope) because they are CRUD
operations, not result-set queries.

{
  "command": "graph callers",
  "repo": "payments",

================================================================================
WHAT IS EXPLICITLY NOT IN V1
================================================================================

These are planned but do NOT ship until the graph is proven correct:

  tool repo watch            (v2 — file watcher for live refresh)
  tool repo snapshot         (v2 — named snapshots for comparison)
  tool arch *                (v2 — architecture boundary queries)
  tool flow *                (v2 — execution and data flow tracing)
  tool change *              (v2 — impact analysis, effect propagation)
  tool legacy *              (v2 — hotspots, clones, seams, monster methods)
  tool fleet *               (v3 — cross-repo operations)
  tool ask *                 (never in core — separate NL wrapper if ever)

No inference layer in v1. Only deterministic facts and human declarations.
No Python pipelines in v1. Only TypeScript extractors and SQLite storage.
No framework extractor plugin interface in v1. Hardcoded extractors only.

Resolution semantics in v1:

  V1 uses direct AST walking in the TypeScript extractor plus a strict second
  pass in the indexer.

  The extractor emits unresolved symbolic targets:
    IMPORTS        -> extensionless file keys
    CALLS          -> names like generateId or this.repo.findById
    INSTANTIATES   -> class names
    IMPLEMENTS     -> interface names

  The indexer upgrades an unresolved edge to a stored graph edge only when
  the target resolves to exactly one node in the current repo snapshot.
  If resolution is ambiguous or missing, the edge is not persisted and is
  counted as unresolved.

  This gives v1 high precision but lower recall for inter-file call graphs.
  IMPORTS usually resolve well. CALLS / INSTANTIATES / IMPLEMENTS are more
  conservative because v1 does not perform local type binding, scope-aware
  variable tracing, or compiler-assisted semantic resolution.

  Potential v2 improvements:
    local binding analysis inside the extractor
    stronger symbol-to-import linking before the indexer pass
    compiler-assisted adapters (for example TypeScript compiler data or SCIP)
    richer unresolved-edge diagnostics and breakdowns


================================================================================
V1 ACCEPTANCE CRITERIA
================================================================================

The tool is v1-complete when:

1. It can index a real TypeScript codebase (>100 files) in under 30 seconds.
2. tool graph callers returns the correct set of callers for any symbol,
   verified against manual inspection.
3. tool graph callees returns resolved callees with high precision.
   v1 uses syntax-only resolution, so recall is lower for this.method()
   and obj.method() calls. See Resolution Semantics above.
4. tool graph dead returns provably unreachable functions (verified by
   cross-referencing with Knip output or manual inspection).
5. tool graph cycles detects known circular dependencies (verified against
   dependency-cruiser or Madge output).
6. tool graph path finds the correct shortest path between two nodes
   (limited to resolved edges).
7. tool repo refresh creates a refresh snapshot linked to the previous one.
   In v1, refresh performs a full re-extraction. True incremental refresh
   (only re-parsing changed files) is v2.
8. tool declare entrypoint causes declared entrypoints to be excluded from
   tool graph dead results.
9. tool declare boundary records are stored and retrievable.
10. All of the above work with --json output that an agent can parse.
