V1 Validation Report
=====================

Date: 2026-04-01
Codebases tested:
  - FRAKTAG engine (53 TS files, clean architecture, Express API)
  - AMODX (395 TS files, AWS Lambda monorepo, React frontend, CDK infra)


================================================================================
ACCEPTANCE CRITERIA RESULTS
================================================================================

1. Index >100-file TS codebase in <30 seconds
   PASS.  AMODX: 395 files indexed in 648ms.
          FRAKTAG: 53 files indexed in 174ms.

2. graph callers returns correct callers
   PARTIAL.  Works for CALLS edges. Does not find callers via
             INSTANTIATES or IMPORTS edges by default (callers/callees
             default to CALLS edge type). Correct behavior per the query
             contract, but a usability gap for "who uses this class?"

3. graph callees returns correct callees
   PARTIAL.  Arborist.execute shows 3 of 4 dispatched methods.
             The 4th (moveNode) is missed because this.moveNode() cannot
             be resolved without class-context binding. This is the
             expected v1 resolution gap — v1 has high precision but
             lower recall for call edges, as documented in v1-cli.txt.

4. graph dead returns unreachable symbols
   PASS.  Returns exported types, interfaces, constants, and classes
          that are never referenced internally. Results are correct.
          OWNS edges do not contaminate dead detection (verified).

5. graph cycles detects circular dependencies
   PASS.  FRAKTAG: 6 cycles detected (core<->prompts, core<->nuggets,
          core<->chunking, llm<->prompts<->core). All verified correct.
          AMODX: 21 cycles detected (renderer component modules cycling
          through the shared src directory). Architecturally accurate.

6. graph path finds shortest paths
   PARTIAL.  Works when edges are resolved. Fails for paths that traverse
             unresolved this.property.method() chains (expected for v1).
             IMPORTS-based paths work correctly.

7. repo refresh works
   PARTIAL.  Creates a refresh snapshot with parent link, which is
             correct. But the underlying implementation still does a full
             re-extraction rather than diffing changed files. The v1-cli.txt
             acceptance criterion has been updated to match this reality.
             True incremental refresh is v2.

8. declare entrypoint excludes from graph dead
   PASS.  Verified in unit tests (28 storage tests cover this).

9. declare boundary stored and retrievable
   PASS.  Verified via CLI: declare boundary, declare list, --json output.

10. All commands support --json
    PASS.  Graph commands use the universal envelope with snake_case.
           Repo/declare commands use operation-specific JSON.

================================================================================
EXTRACTION CAPABILITY BOUNDARY (v1.5, as of ts-core:0.2.0)
================================================================================

Updated: 2026-04-02

Measured against: repo-graph (34 files), FRAKTAG (81 files), AMODX (397 files)

Strong (high precision, high recall):
  - IMPORTS edges: near-perfect. 0-2 failures across all validation codebases.
  - MODULE nodes and OWNS edges: accurate.
  - Module-level cycle detection: accurate and immediately actionable.
  - Interface/class member extraction: complete for method signatures
    and property signatures. Overloaded methods merged into one node.

Bounded (high precision, architecture-sensitive recall):
  - Receiver type binding for this.property.method() via class-scope
    bindings (field type annotations + constructor parameter properties).
  - Receiver type binding for variable.method() via file-scope, parameter,
    and function-local bindings (explicit type annotations + new ClassName()).
  - 3-part chain resolution (ctx.storage.method()) via memberTypes map
    for types defined in the same file.
  - Lexical scope model: sequential accumulation with TDZ pre-scan,
    var hoisting as shadow-only with statement-position typed upgrade,
    loop-header scoping, initializer self-reference blocked.
  - Best results on class-heavy architectures with explicit in-project
    typing (ports, DI via constructor). Minimal impact on functional-style
    codebases dominated by SDK client calls.

Weak (known gaps, not fixable by more local binding rules):
  - External SDK types (node_modules): not indexed. obj.method() where
    obj is typed by an external package stays unresolved.
  - Cross-file member type propagation: memberTypes map is per-file.
    Imported types cannot contribute to 3-part chain resolution.
  - Destructured bindings: const { storage } = ctx not tracked.
  - Reassignment: let x = a; x = b; x.method() uses first binding only.
  - Dynamic patterns: computed property access, bracket notation,
    Proxy, string-template method names.

Known precision regression (Horizon 1 unfinished):
  - Interface member extraction introduced new same-named nodes that
    increase ambiguity in the resolver's name-matching pool.
    FRAKTAG: this.method (needs class context) went from 0 to 109
    because methods like "save", "findById" now exist on both class
    AND interface nodes.
  - This is a self-inflicted regression, not a natural ceiling.
    The fix is to narrow the resolver's name matching to prefer nodes
    on the class being called from (requires propagating class context
    into the resolver, not just the extractor). Tracked as remaining
    Horizon 1 precision work.

Measurement (2026-04-02):
                      v1 baseline    v1.5 current
  repo-graph:           285 edges     351 edges    (+23%)
  FRAKTAG:              836 edges     979 edges    (+17%)
  AMODX:               1685 edges    1699 edges    (+1%)


================================================================================
KNOWN V1 LIMITATIONS (UPDATED)
================================================================================

1. [PARTIALLY ADDRESSED] this.method() calls now resolve when the method
   name is globally unique. When ambiguous (same method name on multiple
   classes/interfaces), resolution fails. The interface member extraction
   increased ambiguity — see precision regression above.

2. [PARTIALLY ADDRESSED] this.property.method() resolves via class-scope
   bindings when the property has an explicit type annotation. Untyped
   properties or inferred types are not resolved.

3. [PARTIALLY ADDRESSED] obj.method() resolves when obj is a typed local
   variable, typed parameter, or typed file-scope const. SDK clients and
   cross-file types remain unresolved.

4. [UNCHANGED] External dependencies (node_modules) are not indexed.

5. [DONE] --edge-types flag on callers/callees is implemented.

6. [UNCHANGED] graph path fails when the path traverses unresolved edges.

7. [UNCHANGED] No framework-specific extraction yet.


================================================================================
RECOMMENDED IMPROVEMENTS (UPDATED PRIORITY ORDER)
================================================================================

Horizon 1 remaining:
  1. Fix the this.method() ambiguity regression: when resolving a bare
     method name from inside a class, prefer methods on the same class
     (or its supertypes) over same-named methods on unrelated types.

Horizon 2:
  2. arch violations over IMPORTS edges (trustworthy substrate).
  3. Express framework extractor for ROUTES_TO edges.
  4. True incremental refresh using content hash comparison.
  5. Nested .gitignore support.
  6. Cross-file member type propagation for 3-part chains.
  7. Compiler-assisted resolution (TypeScript API or SCIP) for the
     external SDK and inferred-type cases.


================================================================================
VERDICT (UPDATED)
================================================================================

v1.5 is usable. The import/module graph is the trustworthy substrate —
high precision, high recall, suitable for policy enforcement. The call
graph is improved but architecture-sensitive: strong on explicitly-typed
class-heavy codebases, bounded on functional/SDK-heavy patterns, and
carrying one self-inflicted ambiguity regression from interface member
extraction. Arch violations should build on IMPORTS, not CALLS.
Module-level cycle detection is accurate and immediately actionable.
Dead code detection works and produces real, verifiable results.

The tool is ready for agent consumption on TS/JS codebases. The main
gap is call graph completeness, which requires type-aware resolution
in v2.
