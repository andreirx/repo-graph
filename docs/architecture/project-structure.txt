Project Structure — Clean Architecture
=======================================

Dependency rule: source code dependencies point inward.
CLI -> Use Cases -> Core Model. Adapters implement Core Ports.
Nothing in core/ imports anything from adapters/ or cli/.
main.ts is the composition root — the only place that wires concrete adapters to ports.

================================================================================
DESIGN PRINCIPLES
================================================================================

1. Core model is language-neutral.
   NodeKind includes FUNCTION, CLASS, INTERFACE, MODULE, FILE, ENDPOINT, EVENT,
   TABLE, CONFIG_KEY — not TypeScriptFunction or JavaClass. The language is a
   property on the node, not a type in the domain model.

2. Extractors are evidence sources, not truth sources.
   Every edge produced by an extractor carries an evidence field saying which
   extractor produced it and what resolution was achieved (STATIC, DYNAMIC, INFERRED).
   Multiple extractors can produce edges for the same relationship — the storage
   layer deduplicates and records all evidence sources.

3. Four data categories, stored separately.
   - Facts: nodes and edges extracted from code. Attached to snapshots.
   - Measurements: deterministic values from graph, AST, or imported evidence.
     No confidence score. See measurement-model.txt.
   - Declarations: human-supplied via `rgr declare`. Persistent until superseded.
     Includes requirements with verification obligations.
   - Inferences: derived assessments (hotspot scores, risk scores).
     Carry confidence scores. Attached to snapshots.

4. The SQLite database is the integration point.
   The core tool writes facts, measurements, and declarations.
   Coverage and churn are imported from external tools into the same DB.
   The CLI queries all four data categories. The database schema IS the contract.

5. Framework extractors are the planned differentiation (v2+).
   Base extractors (Tree-sitter symbol + import extraction) are commodity.
   Framework extractors (Express routes, NestJS decorators, Spring annotations)
   will capture edges invisible to import analysis. Not yet implemented.

6. Skills are agent workflows, not tool features.
   A skill is a markdown file describing a sequence of CLI commands an agent should
   run to accomplish a specific analysis task. The tool stays deterministic.
   The skill encodes the engineering judgment of how to chain queries.
   Skills are updated independently of tool code.

7. Tests use fixture codebases with known graphs.
   Every test fixture is a minimal project where the expected nodes, edges, and
   query results are known in advance. Extractor tests assert against these
   known-good graphs. This prevents "testing the parser with the parser."

                               coverage, risk, obligations (5-state verdicts)

                                 verdicts, waiver basis
  src/cli/commands/trend.ts   — snapshot-to-snapshot health delta
  src/cli/commands/gate.ts    — CI gate surface (composes evaluator +
                                effective-verdict + reducer; see
                                docs/architecture/gate-contract.txt)
