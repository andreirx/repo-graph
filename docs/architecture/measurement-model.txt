Measurement Model — Four Truth Layers
======================================

Date: 2026-04-03

Repo-graph handles four distinct kinds of truth. Collapsing them into
one storage shape makes query semantics muddy and the product harder
to trust.


================================================================================
THE FOUR LAYERS
================================================================================

1. Evidence
   Raw artifacts from external tools. Not interpreted by repo-graph.
   Examples: Istanbul coverage JSON, git log output, ESLint report,
   bundle stats, trace dumps.
   Storage: artifacts table + filesystem. Evidence links connect
   artifacts to the nodes/edges/measurements they support.

2. Measurements
   Deterministic values computed from graph, AST, or imported evidence.
   Reproducible: same input -> same value. No confidence score needed
   because they are facts, not guesses.
   Examples: cyclomatic complexity of a function, fan-out of a module,
   line coverage percentage of a file, parameter count.
   Storage: NEW measurements table (see schema below).

3. Policies
   Thresholds, architectural rules, and quality gates. Human-declared.
   Examples: "cyclomatic complexity must not exceed 15",
   "src/core must not import src/adapters",
   "coverage on boundary modules must be >= 80%".
   Storage: declarations table (existing) with new declaration kinds.

4. Assessments
   Derived judgments that combine measurements with policies or
   heuristics. Carry confidence or severity. May be probabilistic.
   Examples: hotspot score, risk level, "unhealthy module" verdict,
   "low-test-confidence" warning.
   Storage: inferences table (existing). This is the correct use of
   inferences — derived, judgment-bearing, confidence-scored.


================================================================================
WAIVERS: POLICY OVERLAY ON ASSESSMENTS
================================================================================

Waivers are a distinct form of policy (layer 3) that overlays on
evaluation results. They do NOT belong to any of the four truth layers
above; they modify how assessments are consumed by gates and reports.

Waivers are declarations (kind = 'waiver'). A waiver references a
specific (req_id, requirement_version, obligation_id) tuple and
suppresses that obligation's failure for gate purposes.

Core principle: a waiver suppresses gate failure, NOT the underlying
computed verdict. Every consumer (graph obligations, evidence, gate)
exposes BOTH computed_verdict (truth) and effective_verdict (policy-
adjusted). The computed verdict is never erased.

Five-state effective verdict: PASS, FAIL, MISSING_EVIDENCE,
UNSUPPORTED, WAIVED. WAIVED is ONLY an effective state — never a
computed state.

Waivers are version-scoped: when a requirement is superseded, active
waivers against the old version stop applying to the new version
(the waiver remains in the audit trail). Expired waivers (expires_at
<= now) stop suppressing; the computed verdict resumes as effective.

Full contract: docs/architecture/gate-contract.txt (normative).


================================================================================
WHY MEASUREMENTS ARE NOT INFERENCES
================================================================================

Cyclomatic complexity = 12 is a measured fact. It is not a guess.
It does not have a confidence score. It does not need a basis_json
explaining what evidence supports it. It IS the evidence.

Hotspot score = 0.87 is a derived assessment. It combines churn rate
(measured) with complexity (measured) using a formula (policy/heuristic).
The formula could change. The score is a judgment, not a fact.

If both live in the inferences table, a consumer cannot distinguish
"this is measured" from "this is estimated." That distinction matters
for trust. A system of record must make it queryable.


================================================================================
MEASUREMENTS TABLE (NEW)
================================================================================

CREATE TABLE measurements (
  measurement_uid     TEXT PRIMARY KEY,
  snapshot_uid        TEXT NOT NULL REFERENCES snapshots(snapshot_uid),
  repo_uid            TEXT NOT NULL REFERENCES repos(repo_uid),
  target_stable_key   TEXT NOT NULL,        -- what this measures
  kind                TEXT NOT NULL,        -- metric kind (see below)
  value_json          TEXT NOT NULL,        -- structured value
  source              TEXT NOT NULL,        -- "extractor:ts-core:0.2.0"
                                            -- | "graph-stats:0.1.0"
                                            -- | "coverage-istanbul:0.1.0"
  created_at          TEXT NOT NULL
);

CREATE INDEX idx_measurements_target
  ON measurements(snapshot_uid, target_stable_key, kind);
CREATE INDEX idx_measurements_kind
  ON measurements(snapshot_uid, kind);

No confidence field. No basis_json. These are facts.


================================================================================
MEASUREMENT KINDS
================================================================================

Module structural metrics (computed from graph):

  fan_in              — number of modules that import this module
  fan_out             — number of modules this module imports
  instability         — fan_out / (fan_in + fan_out)
  abstractness        — interfaces and abstract types / total types
  distance_main_seq   — |abstractness + instability - 1|
  cycle_count         — number of dependency cycles this module participates in
  boundary_violations — number of forbidden imports from this module
  coupling_density    — cross-module edge count / module pair count

Function/symbol metrics (computed from AST by extractor):

  cyclomatic_complexity — decision point count
  cognitive_complexity  — weighted complexity penalizing nesting
  parameter_count       — number of formal parameters
  nesting_depth         — deepest control flow nesting level
  function_length       — line count (line_end - line_start + 1)

File/module size metrics:

  module_size           — total LOC across all files in the module
  symbol_count          — number of exported symbols
  file_count            — number of source files in the module

Coverage metrics (imported from external tools):

  line_coverage         — lines covered / total lines (0.0 to 1.0)
  function_coverage     — functions covered / total functions
  branch_coverage       — branches covered / total branches

Churn metrics (imported from git):

  change_frequency      — commits touching this file in time window
  churn_lines           — lines added + deleted in time window
  author_count          — distinct authors in time window


================================================================================
METRIC FAMILIES AND THEIR SOURCES
================================================================================

Family          Source               Depends On
------          ------               ----------
Modularity      graph queries        existing edges + module nodes
Code Quality    extractor AST walk   tree-sitter parse (during indexing)
Coverage        external import      Istanbul/c8/nyc JSON artifacts
Churn           external import      git log artifacts
Hotspot         computation          churn + complexity measurements
Risk            computation          hotspot + coverage + violations


================================================================================
ASSESSMENT KINDS (USE INFERENCES TABLE)
================================================================================

Assessments are derived from measurements + policies. They belong in
the inferences table because they carry confidence and subjective
judgment.

  hotspot_score       — churn_lines * cyclomatic_complexity (normalized)
  risk_score          — hotspot * (1 - coverage) * violation_weight
  module_health       — composite of structure + complexity + coverage
  dead_likelihood     — graph orphan confidence (existing)
  arch_role           — inferred architectural role (existing)

Each assessment cites the measurements it was derived from via
evidence_links or basis_json. The formula is documented and versioned
as part of the assessment source identifier.


================================================================================
POLICY DECLARATION KINDS (USE DECLARATIONS TABLE)
================================================================================

New declaration kinds for quality policies:

  complexity_threshold — max cyclomatic/cognitive complexity for a module
  coverage_threshold   — min coverage percentage for a module
  size_threshold       — max LOC or symbol count for a module
  quality_gate         — composite pass/fail rule for pre-merge checks

These are human-declared, persistent across snapshots, and used by
assessment queries to produce pass/fail verdicts.

Waiver declarations (kind = 'waiver', shipped) are a policy overlay,
not a threshold policy. They reference a specific
(req_id, requirement_version, obligation_id) tuple and suppress gate
failure for that obligation. See WAIVERS section above and
docs/architecture/gate-contract.txt.


================================================================================
THE HEALTH VECTOR
================================================================================

The strongest product move is not a single "quality" number. It is a
health vector with trend across snapshots:

  Module Health = {
    structure:  { instability, abstractness, distance, cycles, violations }
    complexity: { max_cc, avg_cc, monster_methods, avg_nesting }
    coverage:   { line, branch, function, risk_weighted }
    churn:      { frequency, lines, hotspot_pressure }
    policy:     { violations_count, gates_passing }
  }

Each dimension is measured. The combination is assessed. The trend is
queryable across snapshots. That is what makes the tool credible.

    - benchmark artifact kind in artifacts table

Phase 9: Source locator hygiene (lightweight)
  Principle: rgr owns identity and structure. Agents own live
  line lookup via rg/awk/tree-sitter/LSP from the working tree.
  rgr should NOT build a live line database or track mutable
  editor coordinates.

  What rgr already stores (adequate for most uses):
    - file_path, line_start, col_start, line_end, col_end per node
    - file_content_hash per file_version
    - snapshot_uid binding

  What could be added later as lightweight traceability aids:
    - stale flag: compare indexed file hash vs current workspace
    - expose file hash in query JSON so agents know if coordinates
      are from the current version
    - byte offsets for snapshot-accurate spans (lower priority)

  What NOT to build:
    - per-line database
    - continuously updated line rows
    - branch-sensitive mutable coordinates
    - live editing integration

  The agent resolves final live positions on demand from the
  working tree. rgr provides the identity (what symbol), the
  agent resolves the location (which line right now).


================================================================================
EXTRACTED DOMAIN VERSIONS
================================================================================

Extracted domain versions are a distinct class of measurement.
They represent what the system claims to be, not what repo-graph
observed structurally.

Three version classes (must not be conflated):

  Provenance versions (repo-graph internal):
    What toolchain produced this snapshot. Answers reproducibility.
    Stored in: snapshots.toolchain_json, declarations.authored_basis_json

  Extracted domain versions (from the codebase):
    What the system claims to be. Answers compatibility and intent.
    Stored in: measurements table with source = "version-extractor:X.Y.Z"

  Declared versions (human-authored):
    Version facts where extraction is impossible or incomplete.
    Stored in: declarations table with appropriate kinds.

Measurement kinds for extracted versions:

  package_version       — from package.json, Cargo.toml, pyproject.toml, etc.
  api_version           — from route prefixes, OpenAPI specs, protobuf packages
  migration_version     — from migration filenames, ORM metadata
  schema_version        — from schema registries, JSON Schema, AsyncAPI
  dependency_version    — from lockfiles (third-party compatibility context)
  release_tag           — from git tags and release branches

Each carries:
  - target_stable_key: the repo, module, or file this version describes
  - value_json: { "value": "2.4.1", "source_file": "package.json" }
  - source: "manifest-extractor:0.1.0" (or "declaration" for human-supplied)

Evidence model:
  Source file provenance is embedded in value_json (source_file field),
  not via evidence_links rows. This is adequate for manifest-extracted
  facts where the source is a single known file. The evidence_links
  table is reserved for external artifacts (coverage reports, git
  metrics) where the evidence is a separate file or output.

Extraction priority:
  1. Manifest files (package.json, etc.) — highest signal, structured
  2. OpenAPI/AsyncAPI/protobuf specs — contract-level versions
  3. Migration filenames — database schema versions
  4. Git tags — release-level versions
  5. Route prefixes / URL patterns — API version conventions

The extraction is deterministic (parsing structured files), so these
are measurements (confidence 1.0), not inferences.
