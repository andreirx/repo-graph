Missing Links in Current Tooling

Yes. That is still an open problem.

What you are describing is not ordinary code RAG. It is repository intelligence: a system that can answer not just “find relevant code” but “place this code in the architecture and execution reality of the system.” Vector search alone does not do that.

The core distinction:

1. Retrieval answers “what text or code looks related to this query?”
2. Code intelligence answers “what symbol references what?”
3. Architecture understanding answers “what role does this component play, what boundary is it on, what depends on it, and what runtime path reaches it?”
4. Execution understanding answers “when does this actually run, under which entrypoint, event, request, job, or scheduler?”

Only the first of those is RAG.

Why this remains open:

1. Code meaning is relational, not chunk-local.
A function only “means” something through imports, callers, interfaces, registrations, config wiring, DI containers, routes, schemas, queues, tests, and runtime traces. Embeddings flatten code into vectors. They lose graph structure unless you rebuild that structure separately.

2. Architecture is mostly implicit.
Most repos do not contain explicit machine-readable facts saying “this is an application service,” “this is an adapter,” or “this sits on the write path after validation but before persistence.” Humans infer that from naming, folder structure, dependency direction, framework conventions, and docs. Machines can infer some of it, but not reliably across codebases.

3. Execution path is not the same as call graph.
A call graph is static. Real execution also depends on routing, async queues, reflection, codegen, config flags, feature gates, plugin systems, dynamic imports, ORM hooks, middleware chains, retry systems, and external events.

4. Data path and event path are cross-cutting.
A single business flow may traverse HTTP controller -> use case -> domain service -> repository -> outbox -> queue consumer -> webhook handler -> projection updater. No single chunk captures that. A vector DB of chunks will miss the path unless the system explicitly models edges.

5. Exact understanding is impossible in the general case.
For unrestricted programs, full semantic behavior cannot be decided mechanically in the general case. Practical tools always work by approximation: parsers, heuristics, type systems, framework-specific rules, traces, and confidence scoring.

What is solved reasonably well:

1. Symbol lookup, references, definitions, imports.
2. Semantic search when you do not know the keyword.
3. Localized impact analysis in well-behaved typed codebases.
4. Repo maps and token-budget summaries.
5. Basic call graph extraction in some languages and frameworks.

What is still weak:

1. “Where does this sit in the architecture?”
2. “What runtime path activates this?”
3. “What data contracts pass through this?”
4. “What events eventually reach this handler?”
5. “What business capability does this cluster of files implement?”
6. “Which invariants or failure modes does this component protect?”

The correct mental model is not “slap a vector database on a repo.” The correct model is a layered repository knowledge system:

1. Structural index.
Files, symbols, definitions, references, imports, inheritance, implementations, routes, schemas, configs.

2. Semantic index.
Embeddings over symbols, modules, comments, docs, commit messages, ADRs, test names.

3. Graph model.
Nodes: repo, package, module, file, symbol, endpoint, event, table, queue, job, config key.
Edges: imports, calls, implements, instantiates, reads, writes, emits, consumes, routes_to, registered_by, guarded_by, tested_by.

4. Runtime evidence.
Logs, traces, coverage, integration tests, sampled execution, event recordings.

5. Architectural inference layer.
Rules such as dependency direction, folder conventions, interface ownership, port/adapter detection, entrypoint classification, boundary crossing detection.

6. Query planner.
For “who calls this?” use structural search.
For “what handles payment retries?” use hybrid semantic + graph.
For “how does request X reach table Y?” use graph expansion plus runtime evidence.

That is why current tools feel partial. Most of them are good at one or two layers, not all six.

If the target is “understands and maps code components better,” the central artifact should be a graph with evidence, not a vector store. RAG should sit on top as a natural-language retrieval interface into that graph. Without the graph, the system will sound insightful while being structurally wrong.

The strongest product direction is:

1. Build a code graph first.
2. Attach architectural labels and confidence scores.
3. Add semantic retrieval over graph nodes.
4. Answer every query with evidence paths, not just chunks.

That is the open problem: not retrieval of code, but faithful synthesis of structure, intent, and runtime reality from incomplete evidence.

**What We Actually Learned Across Domains**

From product management:
- Requirements must separate outcomes from proposed solutions.
- Constraints and non-goals matter as much as goals.
- Stakeholders change requirements for different reasons; that must stay visible.
- Ambiguity deferred becomes rework later.

From systems engineering:
- Interfaces and failure modes must be explicit.
- Verification must be planned before implementation.
- Traceability from requirement to evidence is not optional in complex systems.
- Operational context matters as much as nominal behavior.

From clean architecture:
- Stable policy must not depend on volatile mechanisms.
- Boundaries must be deliberate and enforced.
- Data crossing boundaries must be simplified and normalized.
- Reuse works only when abstractions align with real axes of change.

From component/module engineering:
- Reuse is expensive when module boundaries are wrong.
- Premature shared modules create coupling and release paralysis.
- Mature reusable modules need versioning, ownership, compatibility rules, and test contracts.

From agile and scaled planning:
- Work must be decomposed into independently valuable, independently testable slices.
- Cross-team coordination requires shared dependency visibility.
- Planning artifacts exist to manage sequencing, not to satisfy management theater.
- Fast feedback loops beat speculative detail, but only when scope and evidence are controlled.

From DevOps and release engineering:
- Build, test, deploy, rollback, observe.
- If it is not reproducible, it is not engineered.
- Progressive delivery beats big-bang release.
- Production telemetry is part of the specification loop.

From safety-critical and high-reliability engineering:
- Hazards, invariants, and failure containment must be explicit.
- Evidence matters more than confidence.
- Human review should focus on irreducible judgment, not clerical translation.
- Change impact must be bounded before execution.

All of those are the same lesson in different clothes: preserve intent while reducing ambiguity and proving correctness at every boundary.

**What The Tool Should Understand**

Not just syntax.

It should extract and represent:

- file/module/package containment
- symbols, definitions, references
- imports and dependency direction
- route registrations
- DI registrations and bindings
- event emitters and handlers
- queue producers and consumers
- ORM models and repository edges
- config-gated execution paths
- test-to-code coverage links
- migrations and deployment coupling

That is already enough to create major cognitive savings.

**What Must Be Automatic vs Human-Supplied**

Automatic first:
- AST parsing
- import/reference graph
- framework-specific edges
- dependency cycles
- approximate boundary inference
- impacted tests
- likely entrypoints
- architectural violations from existing code

Human-supplied later, selectively:
- module purpose
- critical invariants
- maturity level
- approved exceptions
- non-obvious architectural intent
- change-specific constraints for major work

This is the correct compression of past process lessons: automate the clerical parts, preserve human judgment only where judgment is actually needed.

**What To Add In Your Repo Right Now**

You already have:

- ADRs

That is enough to support the next step.

Add only the low-friction structured artifacts that a graph-based tool can consume or enrich.

First add:

4. Lessons-learned registry tied to modules and rules

Do not start with heavyweight requirements bureaucracy.

**1. Module Contracts**
One small machine-readable file per major module/package.

Fields:
- purpose
- maturity
- owned abstractions
- allowed dependencies
- forbidden dependencies
- public interfaces
- critical invariants
- test entrypoints

This is high leverage because it gives the graph semantic anchors.

**2. Executable Architecture Rules**
Turn key architectural constraints into checks.

Examples:
- core must not import adapters
- API layer cannot bypass application services
- DTO-only across boundary
- no cyclic dependencies between bounded contexts
- certain modules are read-only except through approved ports

This converts wiki rules into machine enforcement.

**3. Structured Evidence Summaries**
Not raw logs. Summaries keyed to code or change scope.

Fields:
- target module or symbol
- test command
- pass/fail
- last run timestamp
- what obligation it proves

This lets agents answer “what evidence exists for this area” without reading CI sludge.

**4. Lessons-Learned Registry**
Short records with:
- trigger
- failure mode
- affected area
- permanent rule or heuristic
- test obligation added
- ADR link if promoted

A lesson that is not attached to a module, rule, or test obligation is dead text.

**What Not To Add Yet**

Do not add:
- full formal requirements specs for every task
- work packets for every change
- manual system graph curation
- giant YAML programs nobody will maintain
- separate process artifacts that duplicate docs

That is where adoption collapses.

**The Real Compression Of 50 Years Of Engineering**

The past 50 years did not teach “write more process documents”.

They taught:
- separate concerns
- preserve traceability
- constrain changes
- verify behavior
- encode lessons
- automate repetition
- keep humans on the judgment path, not the clerical path

So the compressed AI-native form is:

1. auto-extract structure
2. let humans declare a few critical truths
3. enforce boundaries mechanically
4. attach evidence to structure
5. let agents query and operate against that substrate

That is the intermediary natural step.

**The Correct Near-Term Strategy**

For your own repo and for a general tool, the sequence is:

1. Keep `AGENTS.md`, ADRs, architecture docs.
2. Add module contracts in machine-readable form.
3. Add executable architecture rules.
4. Add structured evidence summaries.
