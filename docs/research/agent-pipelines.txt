Working With Legacy Code With AI Agents

1. Automated Comprehension and Structural Mapping
* Dependency Graphing: AI agents process repository structures to map execution paths, mathematically identifying structural bottlenecks ("pinch points") to optimize test placement.

Recursive folder summaries are just a localized text index. They create a "table of contents," but they completely fail to capture the execution reality of the system. If you want true repository intelligence, the agent cannot just read files and summarize them; it must act as an orchestrator for deterministic parsing tools to build a queryable graph.
Here is how you break down the task of Dependency Graphing for an AI agent, shifting the substrate from "LLM reading text" to "LLM driving deterministic extraction."
1. Existing Tools the Agent Should Use (Substrate Efficiency)
Do not force the LLM to read raw source code and guess the imports using regex or context windows. LLMs hallucinate edges. Instead, prompt the agent to write scripts that execute deterministic tools, parse their outputs, and map them to your canonical graph.
* Tree-sitter: The gold standard. It provides fast, incremental parsing into an Abstract Syntax Tree (AST) for almost any language. The agent should use Tree-sitter bindings (e.g., in Python) to extract symbols, function calls, and import statements with 100% accuracy.
* SCIP (Code Intelligence Protocol): Developed by Sourcegraph. It is a standardized protobuf schema for indexing source code (definitions, references, and hover documentation). The agent can trigger a SCIP indexer and translate the output.
* Language-Specific Dependency Analyzers:
    * TypeScript/Node: dependency-cruiser (outputs JSON graphs of module dependencies).
    * Python: pydeps or standard library ast module.
    * Java/C#: Bytecode/IL analyzers or static analysis tools (e.g., ArchUnit) that can export dependency matrices.
* Graph Storage: Instruct the agent to output the final mapped graph into a lightweight, queryable format like SQLite (with a nodes/edges table), a local Neo4j instance, or a standardized JSON graph schema.

2. The Inputs and Outputs
To prevent the agent from wandering, you must strictly define the data contract for the graphing operation.
Inputs provided to the Agent:
1. Target Directory: The root of the module or repository.
2. Exclusion Rules: .gitignore, test directories, vendor/node_modules.
3. Language Profile: Instructions on how dependencies are declared in this specific language (e.g., import X from Y vs. require(Y)).
4. Module Contracts (if available): The MODULE.yaml files you mentioned, defining the intended boundaries.
Outputs expected from the Agent:
A structured JSON (or SQLite database) containing two primary arrays: Nodes and Edges.
* Nodes: Files, Classes, Functions, Interfaces, Endpoints, Database Tables.
* Edges: IMPORTS, CALLS, IMPLEMENTS, INSTANTIATES, EMITS, CONSUMES.

3. The Generic Prompts for the Agent
Break the graphing process into three distinct execution phases.
Phase 1: Local Structural Extraction
Goal: Extract undeniable facts from individual files using deterministic parsing.
Prompt:
"You are a Structural Code Indexer. Your task is to extract the deterministic dependency graph of the provided codebase.
Do not attempt to read and summarize the logic. Instead, write and execute a Python script utilizing [Tree-sitter / native AST parser] to scan all files in the target directory, respecting exclusion rules.
For each file, extract the following into a structured JSON format:
1. Nodes: Every defined Class, Function, Interface, and exported constant.
2. External Edges: Every external import, require, or inclusion statement.
3. Internal Edges: Every explicit function invocation or class instantiation within the file.
Output the result as a local graph_manifest.json for each directory containing the raw nodes and edges."
Phase 2: Global Graph Resolution
Goal: Connect the local manifests into a repository-wide web and resolve relative imports to absolute node IDs.
Prompt:
"You are a Graph Resolution Engine. Your task is to aggregate the local graph_manifest.json files generated in the previous step into a single, global repository graph.
Execute the following operations:
1. Namespace Resolution: Resolve all relative imports (e.g., ../services/payment) to their absolute file paths and specific exported Node IDs.
2. Edge Linking: Convert raw string references into formal directed edges (e.g., Node: A [CALLS] Node: B).
3. Dead Node Detection: Flag any Nodes (functions/classes) that have an in-degree of 0 (no edges pointing to them) and are not designated entry points (like route handlers or public API exports).
Output the finalized global graph as a single repository_graph.json containing a unified list of all Nodes and Edges."
Phase 3: Architectural Inference & Policy Checking
Goal: Overlay the human-defined boundaries (from your MODULE.yaml files) onto the raw machine graph to find violations.
Prompt:
"You are an Architecture Compliance Verifier. You have been provided with the repository_graph.json and a set of human-authored Module Contracts (MODULE.yaml).
Perform an impact and violation analysis by running queries against the graph:
1. Boundary Violations: Identify any [IMPORTS] or [CALLS] edges where a node in a module explicitly violates the forbidden_dependencies listed in its Module Contract.
2. Cycle Detection: Identify any circular dependency chains at the module/package level.
3. Entrypoint Mapping: For the public interfaces defined in the Module Contracts, trace the execution graph downward and list all internal modules they depend upon.
Output an architecture_report.md detailing any structural violations, mapping the discrepancies between the intended architecture (the YAML) and the physical execution reality (the Graph)."
Tying it to the Execution Pipeline
Once this graph exists, your agents no longer need to "guess" what files to modify. When you hand the agent a Change Spec (e.g., "Add retry-state guard to payment flow"), the agent's very first move is to query the graph:
"Find all nodes that call PaymentService.retry, list the files they reside in, and list the MODULE.yaml constraints for those files." This forces the agent to operate within the engineered constraints of the system, rather than treating the codebase as a flat directory of text files.


* Semantic Grouping: Agents analyze monolithic, procedural "God classes" or legacy modules, grouping methods and variables by domain responsibility to propose target object-oriented boundaries (CRC modeling).


Semantic grouping is the process of taking a massive, unstructured block of code (like a 3,000-line God Class or a directory of procedural scripts) and discovering the hidden bounded contexts inside it.
If structural graphing (our previous step) answers "who talks to whom?", semantic grouping answers "what concepts belong together, even if they aren't currently talking?"
If you just hand a 3,000-line file to an LLM and ask it to "group this into classes," it will hallucinate, lose context, or provide superficial groupings based only on function names.
Instead, the agent must orchestrate a pipeline that combines Natural Language Processing (NLP) with Structural Cohesion Metrics. Here is how to break down the Semantic Grouping task.

1. Existing Tools to Leverage (The Substrate)
Do not ask the LLM to guess relationships. Use deterministic data science and machine learning libraries to mathematically calculate cohesion.
* Tree-sitter: (Again) for flawless Abstract Syntax Tree (AST) extraction. You need to pull out functions, signatures, docstrings, and class-level variables as discrete chunks.
* Embedding Models: To turn code semantics into math.
    * Local/Open Source: all-MiniLM-L6-v2 or jina-embeddings-v2-base-code.
    * API-based: OpenAI's text-embedding-3-small.
* scikit-learn / HDBSCAN: For clustering the dense vectors generated by the embedding models. HDBSCAN is particularly good because it doesn't force you to guess the number of clusters (unlike K-Means) and handles "noise" (utility functions that don't belong to a core domain) very well.
* NetworkX (Python): For running Community Detection algorithms (like the Louvain method). This is crucial because true semantic grouping is a hybrid of what the code says (embeddings) and how it executes (structural edges).

2. Tools to Build Ourselves in Python (The Pipeline)
Your agent should be prompted to write or execute these specific Python utilities to form the grouping pipeline:
1. The AST Shredder: A script that takes a massive file and shreds it into a JSON array of CodeChunks. Each chunk contains: id, type (function/variable), name, raw_code, and extracted_text (variable names, comments, string literals stripped of syntax).
2. The Hybrid Cohesion Matrix: A script that builds a graph where nodes are CodeChunks. It calculates edge weights between nodes based on two factors:
    * Semantic Weight: Cosine similarity between the embeddings of Chunk A and Chunk B.
    * Structural Weight: +1 if Chunk A calls Chunk B, or if both read/write to the same global/instance variable (LCOM - Lack of Cohesion of Methods).
3. The Cluster Extractor: A script that applies NetworkX community detection over the Hybrid Matrix to group the nodes into distinct proposed boundaries.

3. Inputs and Outputs
Inputs to the Pipeline:
* Target File/Directory: The monolith to be analyzed.
* Threshold Parameters: Minimum cosine similarity score to consider an edge valid (e.g., 0.75).
Outputs from the Pipeline:
* A semantic_clusters.json file containing arrays of Node IDs grouped by their calculated communities.

4. How to Prompt the Agent
You split the prompting into the Mechanical Phase (where the agent writes/runs the Python math) and the Synthesis Phase (where the agent finally reads the code to label it).
Phase 1: The Mechanical Extraction
Goal: Force the agent to rely on math, not LLM intuition, to find the groupings.
Prompt:
"You are a Semantic Code Analyst. Your task is to mathematically group the cohesive blocks within the target file [filename].
Write and execute a Python script that performs the following:
1. Use tree-sitter to parse the file and extract every function, method, and class-level variable into discrete text chunks.
2. Generate vector embeddings for the raw text of each chunk using [Embedding Model].
3. Calculate a similarity matrix using Cosine Similarity across all chunks.
4. Apply HDBSCAN clustering (or NetworkX Louvain community detection) to this matrix to group highly related chunks.
Output the raw groupings as clusters.json, where each cluster contains the names of the functions and variables assigned to it."
Phase 2: The Semantic Synthesis (The LLM Native Task)
Goal: Now that the math has grouped the nodes, use the LLM's reasoning to give those groups business meaning and propose the new architecture.
Prompt:
"You are an Architecture Modeler. I am providing you with clusters.json, which contains mathematically grouped functions and variables from our legacy monolith.
For each cluster, perform the following:
1. Analyze: Read the names, signatures, and docstrings of the items in this cluster.
2. Label: Deduce the single, unified business responsibility of this group. Name the proposed new Class or Module (e.g., PaymentRetryService, UserSessionState).
3. Interface Design: Based on the functions within this cluster, draft the public Interface/API that this new module should expose to the rest of the system.
4. Outliers: Identify any function in this cluster that seems mathematically grouped but logically violates the Single Responsibility Principle for your proposed label.
Output a CRC (Class, Responsibility, Collaborator) model for each proposed cluster."

By doing this, you prevent the AI from drowning in a 100,000-token context window trying to play architect. You use the LLM to drive the Python scripts that reduce the dimensionality of the code, and then you use the LLM to read only the highly-correlated clusters to assign them meaning.
If we have the structural graph (from your previous step) and the semantic groups (from this step), the next logical phase is mapping how data flows through these newly discovered boundaries. Shall we break down how to prompt an agent to build an Event & Data Flow Mapper?

1. The Substrate: Existing Tools to Leverage
Data and event flow mapping cannot be solved by simple text search. It requires constructing a Code Property Graph (CPG) or performing static taint analysis to track how variables move through the execution stack, combined with identifying asynchronous publish/subscribe linkages.
* Joern: An open-source tool that generates Code Property Graphs (combining AST, Control Flow Graph, and Program Dependence Graph). It allows querying data flow paths (e.g., "show me all paths from an HTTP parameter to a SQL query").
* CodeQL: A semantic code analysis engine that treats code as data. It excels at static data flow analysis and taint tracking.
* Tree-sitter & ripgrep: Used for high-speed extraction of specific architectural markers (e.g., @Controller annotations, amqp.publish calls, Kafka topic strings).
* OpenTelemetry / Distributed Tracing (Dynamic): If the system is running, static analysis is secondary to observing actual traffic. The agent parses exported OpenTelemetry spans or Jaeger JSON dumps to map real execution.
2. The Pipeline: Tools to Build in Python
When enterprise tools like CodeQL are unavailable or too heavy, the agent must build a specialized static analysis pipeline using AST manipulation.
1. The Entrypoint Extractor: A script utilizing Tree-sitter to scan the repository and identify all ingress boundaries: HTTP route definitions, gRPC service methods, GraphQL resolvers, queue consumers, and scheduled cron jobs.
2. The Event Registry Builder: A script that scans for message broker invocations (e.g., emit(), publish(), send()). It extracts the event topic/channel name and the structure of the payload being dispatched.
3. The Payload Tracer: A lightweight taint-analysis script. Given an entrypoint and an initial parameter (the "source"), it recursively traverses the AST to identify where that data is passed, mutated, or persisted (the "sink").
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository/Directory.
* Framework Profile: A strict definition of how the target framework handles routing and events (e.g., "In this Spring Boot app, entrypoints are marked by @RestController and KafkaListener. Database sinks extend JpaRepository.").
Outputs from the Pipeline:
* A data_flow_map.json detailing exact paths.
* Schema: Entrypoint -> Intermediary Modules -> Emitted Events -> Database Sinks.
4. The Prompts
Divide the operation into distinct, mechanical phases. Do not ask the LLM to "understand the flow." Ask it to orchestrate deterministic extraction.
Phase 1: Entrypoint & Sink Identification
Goal: Map the absolute edges of the system. Where does data enter, and where does it leave?
Prompt:
"You are a Boundary Extraction Engine. Your task is to identify all ingress and egress points in the target codebase.
1. Write a Python script using Tree-sitter configured for the target language.
2. Extract all Entrypoints: Identify functions or methods mapped to HTTP routes, RPC handlers, or message queue listeners based on the provided Framework Profile. Output their file paths, method names, and expected input schemas (DTOs).
3. Extract all Sinks: Identify all interfaces or classes responsible for database persistence (ORMs, SQL execution) or external API dispatch.
Output a boundaries.json file containing arrays of Entrypoints and Sinks."
Phase 2: Static Path Tracing
Goal: Connect the Entrypoints to the Sinks through the intermediate structural graph.
Prompt:
"You are a Static Flow Analyzer. Your task is to trace the execution path from a known entrypoint to a known sink.
1. Read boundaries.json and the previously generated repository_graph.json.
2. Write a recursive traversal script. For each Entrypoint, follow its outgoing [CALLS] edges in the repository graph.
3. Document the sequence of module transitions. Stop traversal when the path terminates at a defined Sink, or when the call stack depth exceeds the specified limit.
4. Record the specific data entities (variable names, types) passed across each module boundary.
Output a call_chains.json file where each record represents a complete vertical slice of execution from ingress to persistence."
Phase 3: Event Channel Resolution (The Asynchronous Gap)
Goal: Connect distributed modules that do not share direct function calls, but communicate via events or queues.
Prompt:
"You are an Asynchronous Edge Resolver. Your task is to bridge the gaps in the execution graph caused by decoupled event architecture.
1. Scan the codebase using regex or AST tools to find all producer invocations (e.g., publishing to Kafka, RabbitMQ, Redis, or internal EventEmitters). Extract the exact string literal or constant representing the topic/channel name.
2. Scan the codebase for all consumer definitions. Extract the exact string literal or constant they are subscribed to.
3. Generate new virtual edges in the graph labeled [EMITS_TO] and [CONSUMES_FROM], linking the producer function to the consumer function via the shared topic string.
Output event_mesh.json detailing the publishers, topics, and subscribers."



* Duplication Detection: Agents perform semantic scans across disparate files, identifying duplicated logical operations despite variations in variable naming or loop syntax, highlighting targets for extraction.


Duplication Detection, specifically addressing the "Shotgun Surgery" anti-pattern from Chapter 21, requires moving beyond standard copy-paste detection. Traditional tools (like PMD or SonarQube) use token matching; they fail when a developer copies a function, renames the variables, and changes a for loop to a while loop.
An AI agent pipeline must perform Semantic Clone Detection—finding code that executes the same logical intent regardless of syntax variations.
Here is the breakdown for architecting an agent-driven Duplication Detection pipeline.
1. The Substrate: Existing Tools to Leverage
Do not use LLM context windows to cross-reference thousands of files. Use deterministic normalization combined with high-dimensional vector math.
* Tree-sitter: Mandatory for Abstract Syntax Tree (AST) extraction. You must strip out the human elements (comments, formatting, variable names) to expose the raw logical skeleton.
* GraphCodeBERT or text-embedding-3-small: Embedding models specifically trained on code semantics, capable of understanding that a switch statement and a chain of if/else statements may represent identical logic.
* FAISS (Facebook AI Similarity Search): A library for efficient similarity search and clustering of dense vectors. Essential when comparing thousands of function embeddings against each other.
* MinHash / LSH (Locality Sensitive Hashing): An algorithmic fallback for rapidly detecting structural near-duplicates before spending compute on semantic embeddings.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that normalizes the code, maps it to a vector space, and calculates mathematical collisions.
1. The AST Normalizer: A script utilizing Tree-sitter to parse every function and method. It strips all comments, removes whitespace, and replaces all user-defined identifiers (variable names, function names) with generic positional tokens (e.g., VAR_1, VAR_2, METHOD_CALL_A).
2. The Vector Indexer: A script that takes the normalized text of every function, passes it through the embedding model, and loads the resulting vectors into a FAISS index.
3. The Collision Detector: A script that queries the FAISS index to calculate the Cosine Similarity of all vectors against each other. It groups vectors that exceed a strict mathematical threshold (e.g., > 0.92 similarity) into discrete duplicate clusters.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository/Directory.
* Similarity Threshold: A strict float value (e.g., 0.95 for near-exact logical matches, 0.85 for conceptual similarity).
* Minimum Token Count: Ignore trivial functions (e.g., getters/setters under 5 lines) to prevent false-positive noise.
Outputs from the Pipeline:
* A semantic_clones.json file.
* Schema: An array of Clusters. Each cluster contains the Logical_Intent_Summary and an array of Instances (absolute file paths, line ranges, and the raw code blocks that collided).
4. The Prompts
Divide the operation into mechanical extraction and architectural synthesis.
Phase 1: Mechanical Normalization and Indexing
Goal: Force the agent to strip away syntax variations and generate a mathematical collision map of the codebase.
Prompt:
"You are a Structural Clone Detector. Your task is to identify semantic duplication across the target codebase.
1. Write a Python script using Tree-sitter. Parse all files and extract every function exceeding 5 lines of code.
2. Normalize each extracted function: strip all comments, and replace all variable and method names with generic sequential tokens (ID_1, ID_2, etc.) to expose the raw operational structure.
3. Generate embeddings for these normalized AST strings using [Embedding Model].
4. Load the embeddings into a FAISS index and calculate pairwise cosine similarity. Group any functions with a similarity score >= [Threshold] into clusters.
Output semantic_clones.json containing the grouped file paths, line ranges, and similarity scores."
Phase 2: The Abstraction Synthesis (LLM Native Task)
Goal: Now that the math has proven that five disparate files contain the exact same logic, use the LLM to design the architectural fix.
Prompt:
"You are a Refactoring Architect. I am providing you with one cluster from semantic_clones.json. This cluster contains 4 distinct code blocks from different files that execute identical semantic logic despite superficial syntax differences.
Perform the following operations:
1. Intent Analysis: Define the exact business rule or operational logic that these 4 blocks are attempting to execute.
2. Parameterization: Identify the variant data—the specific variables or external calls that differ between the 4 implementations.
3. Abstraction Design: Generate a single, authoritative 'Extract Method' or 'Extract Class' implementation that encompasses the shared logic. Ensure the variant data identified in step 2 is passed in as explicit parameters or injected dependencies.
4. Integration Plan: Provide the exact method signature to be used when replacing the duplicated blocks in the original 4 files.
Output the centralized code block and the refactoring instructions."



* Hotspot Prioritization: Agents analyze version control metadata (e.g., git churn) and cyclomatic complexity to identify the most volatile, risk-prone modules, directing opportunistic refactoring efforts.


Hotspot Prioritization is the mechanical implementation of Chapter 24 ("We Feel Overwhelmed"). It operates on a fundamental economic truth of legacy code: terrible code that is never modified costs nothing. Technical debt only incurs interest when a developer is forced to read it or change it. Therefore, an AI agent must not waste compute or human attention refactoring dormant code. It must mathematically identify the intersection of High Structural Complexity and High Code Churn (Volatility).
Here is how to architect an agent-driven Hotspot Prioritization pipeline.
1. The Substrate: Existing Tools to Leverage
Do not ask the LLM to read files and guess their complexity. Use established static analysis and version control metrics.
* Git / Version Control Commands: The ultimate source of truth for volatility. git log --numstat reveals exactly which files are changed most frequently and how many lines are altered per commit.
* Lizard / Radon: Standard, language-agnostic (Lizard) or Python-specific (Radon) static analysis tools. They compute Cyclomatic Complexity (CCN), parameter counts, and Lines of Code (LOC) without needing to execute the code.
* Pandas (Python): For merging the Git volatility dataset with the Static Analysis complexity dataset and calculating the composite "Hotspot Score."
2. The Pipeline: Tools to Build in Python
The agent must construct a data pipeline that treats the codebase's history as a telemetry stream, merging temporal data with structural data.
1. The Volatility Extractor: A script that executes git log over a specified time horizon (e.g., the last 12 months). It calculates the "Commit Frequency" (how many times a file was in a PR) and "Line Churn" (total lines added/deleted) for every file.
2. The Complexity Scanner: A script that runs lizard over the current working tree. It outputs the average and maximum cyclomatic complexity for every file, flagging methods that exceed human working memory (e.g., CCN > 15).
3. The Hotspot Matrix Engine: A script using pandas that joins the Volatility and Complexity datasets on the file path. It calculates a normalized Hotspot_Score (e.g., Normalized_Churn * Normalized_Max_Complexity).
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository Directory.
* Time Horizon: e.g., since="1 year ago".
* Exclusion Rules: Ignore configuration files, package lock files, and documentation (e.g., *.json, *.md, package-lock.json).
Outputs from the Pipeline:
* A hotspot_matrix.json (and ideally a CSV for human viewing).
* Schema: An ordered array of files, ranked by Hotspot_Score descending, containing File_Path, Commit_Count, Max_Cyclomatic_Complexity, and Top_Offending_Functions.
4. The Prompts
Divide the operation into mechanical telemetry gathering and architectural intervention planning.
Phase 1: Mechanical Telemetry Gathering
Goal: Force the agent to extract undeniable historical and structural metrics rather than relying on developer intuition or complaints.
Prompt:
"You are a Codebase Telemetry Engineer. Your task is to identify the architectural hotspots in the target repository—files that are both highly complex and frequently modified.
1. Write a Python script to parse the git log for the last 12 months. Calculate the total number of commits for each source code file.
2. Write a script to execute the lizard static analysis tool against the repository to calculate the maximum Cyclomatic Complexity for each file.
3. Merge these two datasets using Pandas. Calculate a Hotspot_Score by multiplying the normalized commit frequency by the normalized maximum complexity.
4. Filter out any file with a commit frequency of less than 3 (ignore dormant code, regardless of how ugly it is).
Output hotspots.json containing the top 20 files sorted by their Hotspot_Score."
Phase 2: The Opportunistic Intervention Plan (LLM Native Task)
Goal: Now that the math has proven where the bleeding is, use the LLM to design a surgical intervention that aligns with the current sprint or feature request.
Prompt:
"You are a Refactoring Strategist. I am providing you with the top 5 files from hotspots.json and the Change Spec for our next planned feature.
Perform the following operations:
1. Intersection Analysis: Identify which, if any, of these top 5 historical hotspots lie on the execution path of our upcoming feature change (using the repository_graph.json generated previously).
2. Refactoring Scope: If a hotspot must be modified for the new feature, analyze its highest-complexity function. Propose a strict 'Opportunistic Refactoring' plan. Do NOT propose a total rewrite.
3. Sprout/Wrap Strategy: Specify exactly where to use a 'Sprout Method' or 'Sprout Class' to add the new feature safely without increasing the cyclomatic complexity of the existing hotspot.
4. Test Vise Design: Define the exact Characterization Tests required to clamp the existing logic in place before applying the Sprout techniques.
Output an intervention_plan.md detailing the strict, limited refactoring allowed during this feature ticket."



* Effect Analysis / Change Impact Propagation: Given a specific planned edit, agents trace forward through the dependency graph to identify every downstream method, test, and module whose observable behavior could be altered, producing the minimal mandatory test scope.


Effect Analysis is the mechanical implementation of Chapter 11 ("I Need to Make a Change. What Methods Should I Test?"). It answers the most dangerous question in legacy code: "If I change this line, what else breaks?"
This is fundamentally different from Hotspot Prioritization. Hotspots answer "what is generally risky in this codebase." Effect Analysis answers "what is specifically at risk from this exact edit." The distinction matters because a high-quality, low-churn module might never appear on a hotspot list, yet a single change to a method it depends on could silently corrupt its output.
Feathers introduces "effect sketches"—hand-drawn diagrams showing how a change propagates through return values, modified parameters, and shared state. An AI agent must not hand-draw anything. It must compute the effect propagation graph mechanically, using the repository graph and the variable lifecycle data from previous pipeline stages.
Here is how to architect an agent-driven Effect Analysis pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM guess what might be affected. Use graph traversal over proven structural data.
* The Repository Graph (Built Previously): The repository_graph.json with [CALLS], [READS], [WRITES], [IMPLEMENTS], and [INSTANTIATES] edges provides the structural backbone. Forward traversal from the change point along [CALLS] edges reveals all downstream consumers.
* The Variable Lifecycle Map (Built Previously): The variable_lifecycle_map.json reveals which variables are mutated by the changed method and read by others. A method that reads a shared instance variable mutated by the changed method is affected even if there is no direct [CALLS] edge between them.
* Coverage Tools: coverage.py, Istanbul, JaCoCo. Existing test coverage data tells the agent which tests currently exercise the changed method. If coverage is zero, the effect analysis output becomes the test obligation list.
* Tree-sitter: For extracting the exact return type and parameter mutation behavior of the changed method, which determines the propagation vector (does it propagate via return value, via mutated parameter, or via shared state?).
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that computes forward and backward effect propagation from a specific change point.
1. The Change Point Classifier: A script that analyzes the target method and classifies the propagation vectors. It determines: Does this method return a value consumed by callers? Does it mutate any parameter passed by reference? Does it write to any instance variable, class variable, or global variable?
2. The Forward Effect Walker: A script that traverses the repository graph forward from the change point. Starting at the changed method, it follows [CALLS] edges to find all direct callers. For each caller, it checks whether the caller uses the return value or the mutated parameter. If so, that caller is marked "affected" and the traversal continues recursively from that caller. Traversal stops when a caller ignores the return value and does not propagate the mutation.
3. The Shared State Radiator: A script that handles the non-call-graph propagation vector. If the changed method writes to an instance or global variable, this script queries the variable lifecycle map to find every other method that reads that variable. Those methods are affected regardless of whether they appear in the call graph.
4. The Test Scope Calculator: A script that cross-references the full "affected set" against the coverage data. It classifies each affected node as COVERED (existing tests exercise it), UNCOVERED (no tests exist—mandatory characterization test obligation), or TRANSITIVELY_COVERED (not directly tested, but a tested caller would catch regressions).
3. Inputs and Outputs
Inputs to the Pipeline:
* Target File, Method Name, and Line Range of the planned change.
* repository_graph.json.
* variable_lifecycle_map.json (if available for the target class).
* Coverage data (if available).
Outputs from the Pipeline:
* An effect_propagation.json file.
* Schema: * change_point: The method being modified.
    * propagation_vectors: RETURN_VALUE, MUTATED_PARAMETER, SHARED_STATE (which ones apply).
    * affected_methods: An ordered list of methods affected by the change, with the propagation path from the change point to each.
    * test_obligations: Methods that must have characterization tests before the change is safe.
    * existing_coverage: Which affected methods already have tests.
4. The Prompts
Divide the operation into mechanical graph traversal and semantic risk assessment.
Phase 1: Mechanical Effect Propagation
Goal: Force the agent to compute the mathematical forward-propagation of a planned change through the dependency graph.
Prompt:
"You are an Effect Propagation Engine. Your task is to compute the complete downstream impact of a planned change to [Target Method] in [File Path].
1. Use Tree-sitter to parse the target method. Classify its propagation vectors: Does it return a value? Does it mutate any parameter? Does it write to any instance variable, class variable, or global variable?
2. Write a Python script to query repository_graph.json. Perform a recursive forward traversal from [Target Method]:
   a. Follow all [CALLS] edges to find direct callers.
   b. For each caller, determine if it consumes the return value or reads the mutated parameter. If so, mark it 'affected' and continue traversal from that caller.
   c. If the changed method writes to shared state, query variable_lifecycle_map.json to find all other methods that read that state. Mark them 'affected.'
3. For each affected method, record the exact propagation path (the chain of methods from the change point to the affected method).
4. Cross-reference the affected set against coverage data to classify each as COVERED, UNCOVERED, or TRANSITIVELY_COVERED.
Output effect_propagation.json containing the affected methods, propagation paths, and test obligations."
Phase 2: Semantic Risk Assessment (LLM Native Task)
Goal: Now that the math has proven the propagation surface, use the LLM to assess the semantic severity of each affected path.
Prompt:
"You are a Change Risk Analyst. I am providing you with effect_propagation.json, which contains the mathematically computed downstream impact of a planned change.
Perform the following operations:
1. Critical Path Identification: Among the affected methods, identify any that sit on known critical business flows (payment processing, authentication, data persistence). Cross-reference against MODULE.yaml invariants if available.
2. Silent Corruption Risk: Flag any affected method where the propagation is through shared mutable state rather than direct return values. These are the highest-risk paths because the consuming method has no explicit parameter indicating it depends on the changed method.
3. Test Obligation Prioritization: Rank the UNCOVERED affected methods by risk. Methods on critical business paths or consuming shared state must be characterized first. Methods that only log or emit metrics can be deferred.
4. Minimum Safe Test Plan: Produce the exact ordered list of characterization tests that must be written before the change can proceed, referencing the Characterization Scaffolding pipeline for execution.
Output an effect_analysis_report.md containing the risk-ranked affected methods and the mandatory pre-change test plan."


* Interception Point / Pinch Point Selection: When a change impacts multiple classes in a cluster, agents compute the dominator tree of the affected subgraph to find the narrowest point where a single test covers the widest downstream surface.


Interception Point Selection is the mechanical implementation of Chapter 12 ("I Need to Make Many Changes in One Area. Do I Have to Break Dependencies for All the Classes Involved?"). The answer is no—you do not need to break every dependency. You need to find the pinch point.
A pinch point is a node in the call graph through which all execution paths from the entrypoint to the affected cluster must pass. If you place a characterization test at the pinch point, that single test exercises every downstream class without requiring you to individually isolate each one.
In graph theory, this is the Dominator Problem. A node D dominates node N if every path from the entry point to N must pass through D. The set of dominators forms a tree. The lowest dominator in the tree that still covers all affected nodes is the optimal interception point.
If you ask an LLM to "find the best place to put a test," it will guess based on naming conventions. The agent must compute the dominator tree mechanically.
Here is how to architect an agent-driven Interception Point Selection pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM intuit the best test placement. Use graph algorithms that guarantee correctness.
* The Repository Graph (Built Previously): The repository_graph.json provides the call graph edges. The subgraph of affected nodes (from Effect Analysis) is the input.
* NetworkX (Python): Provides built-in dominator tree computation (nx.immediate_dominators). This is a well-studied algorithm with O(n) complexity per query.
* The Effect Propagation Output (Built Previously): The effect_propagation.json identifies the full set of affected nodes. The interception point must dominate all of them.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that extracts the affected subgraph, computes dominators, and selects the optimal test placement.
1. The Subgraph Extractor: A script that takes the affected set from effect_propagation.json and extracts the induced subgraph from repository_graph.json—all nodes in the affected set plus all edges between them. It identifies the entrypoints of this subgraph (nodes with no incoming edges from within the affected set).
2. The Dominator Tree Builder: A script using NetworkX that computes the dominator tree of the subgraph from each entrypoint. It identifies the Immediate Dominator of every affected node.
3. The Pinch Point Selector: A script that finds the lowest common dominator of all affected nodes. This is the single node that, if tested, covers every downstream affected node. If no single pinch point exists (the affected set spans multiple independent subtrees), the script returns the minimum set of pinch points.
4. The Coverage Estimator: A script that calculates the "coverage ratio" of each candidate pinch point—how many affected nodes are downstream of it, divided by the total affected set. A pinch point covering 95% of affected nodes is actionable; one covering 40% is not.
3. Inputs and Outputs
Inputs to the Pipeline:
* effect_propagation.json (The affected node set from Effect Analysis).
* repository_graph.json.
Outputs from the Pipeline:
* A pinch_points.json file.
* Schema: * pinch_points: An ordered list of optimal test placement locations, each with: method_name, file_path, coverage_ratio (percentage of affected nodes downstream), and the list of affected nodes it dominates.
    * uncovered_nodes: Any affected nodes not dominated by any identified pinch point (indicating an independent execution branch requiring its own test).
4. The Prompts
Divide the operation into mechanical graph computation and test design.
Phase 1: Mechanical Dominator Computation
Goal: Force the agent to compute the dominator tree and select the minimum set of interception points using graph algorithms.
Prompt:
"You are a Graph Dominance Engine. Your task is to find the optimal test placement points for a cluster of affected methods identified by Effect Analysis.
1. Write a Python script to load effect_propagation.json and repository_graph.json.
2. Extract the induced subgraph containing only the affected nodes and the edges between them. Identify the entrypoints (nodes with zero in-degree within the subgraph).
3. Using NetworkX, compute the dominator tree from each entrypoint using nx.immediate_dominators.
4. Find the lowest common dominator that covers the maximum number of affected nodes. If no single dominator covers all nodes, find the minimum set of dominators that collectively cover the entire affected set.
5. For each pinch point, calculate the coverage ratio: (number of affected nodes dominated) / (total affected nodes).
Output pinch_points.json containing the ranked interception points and any uncovered nodes."
Phase 2: Test Placement Design (LLM Native Task)
Goal: Now that the math has identified the optimal interception points, use the LLM to design the actual characterization test that should be placed at each pinch point.
Prompt:
"You are a Test Placement Architect. I am providing you with pinch_points.json, which contains the mathematically optimal locations to place characterization tests for covering a cluster of affected legacy methods.
Perform the following operations:
1. Pinch Point Feasibility: For each identified pinch point, read its source code. Determine whether the method can be called directly from a test harness, or whether it requires dependency breaking first. If it requires dependency breaking, specify the exact technique from the catalog (e.g., Parameterize Constructor, Extract and Override Call).
2. Input Design: For the highest-coverage pinch point, design the input data that will exercise the maximum number of downstream branches. Use the effect propagation paths to ensure the test data flows through all affected nodes.
3. Assertion Strategy: Define what observable outputs the test should capture at the pinch point. Use the Sensing techniques from Chapter 3 (return values, fake objects, sensing variables) to specify how the test will verify downstream behavior without individually testing each affected node.
4. Fallback Plan: If the top pinch point requires too much dependency breaking to be practical under time constraints, recommend the next-best pinch point with the cost-benefit tradeoff explained.
Output a test_placement_plan.md detailing the exact pinch point, the test setup, the inputs, and the assertion strategy."

* Scratch Refactoring: Agents act as disposable processors, executing aggressive, untested structural modifications (flattening conditionals, renaming variables for domain clarity) strictly to expose underlying logic for developer comprehension.


Scratch Refactoring (from Chapter 16) is fundamentally different from the previous tasks. Dependency Graphing and Duplication Detection require the AI to be a strict orchestrator of deterministic math because LLMs hallucinate structure.
Scratch Refactoring, however, is exactly where Large Language Models shine.
The goal of a scratch refactoring is comprehension through aggressive, disposable modification. You do not care if the tests pass. You do not care if the code compiles perfectly. You want to translate a machine-optimized or organically decayed mess into a human-readable domain narrative. Once the LLM translates the logic, you keep the understanding and throw away the code.
Here is how to architect an agent-driven Scratch Refactoring pipeline.
1. The Substrate: Existing Tools to Leverage
You want tools that enforce a safe sandbox and perform localized code manipulation.
* Git: The ultimate sandbox. The agent must use Git to create a disposable branch, ensuring zero risk of accidental commits to the main working tree.
* Tree-sitter: To isolate the specific "Monster Method" or opaque class without dragging the entire 5,000-line file into the LLM's context window.
* Language-Specific Linters/Analyzers (e.g., ESLint, Vulture for Python): To mechanically identify and strip dead code or unused variables before the LLM even begins reading. Less noise equals better semantic translation.
* Frontier LLMs (GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro): This is the core engine. You are using the LLM's semantic reasoning to guess domain intent from poor variable names and convoluted control flows.
2. The Pipeline: Tools to Build in Python
The agent needs a pipeline that isolates the code, translates it, extracts the meaning, and then cleans up the sandbox.
1. The Sandbox Manager: A script that automatically executes git checkout -b scratch/agent-analysis, records the current state, and guarantees a git reset --hard and branch deletion when the task concludes.
2. The Context Isolator: A script using Tree-sitter that takes a target line number, walks up the AST to find the enclosing function/class, and extracts only that block of code, plus any local structs/types it explicitly references.
3. The Semantic Translator: A script that feeds the isolated code to the LLM for aggressive renaming and structural flattening.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target File and Line Number (or Function Name).
* Domain Glossary (Optional but highly recommended): A brief text file listing the core business terms (e.g., "In this system, a 'Ticket' is called a 'Manifest'").
Outputs from the Pipeline:
* A scratch_comprehension.md file.
* Schema: * Original_Code (truncated).
    * Refactored_Code (the highly readable, AI-translated version).
    * Logical_Summary (A plain-English explanation of the hidden business rules).
    * Crucially: The Git branch is destroyed. The output is purely documentation.
4. The Prompts
Divide the operation into mechanical pruning and aggressive semantic translation.
Phase 1: Mechanical Pruning (The Setup)
Goal: Remove the cognitive noise (dead code) before asking the LLM to understand the domain logic.
Prompt:

Prompt:

* Dead Code Purging: Agents perform static analysis prior to manual intervention, identifying and removing provably unreachable execution branches to reduce cognitive load.


Dead Code Purging is the immediate tactical application of the insight from Chapter 16: "When we delete code, we don't have to understand it anymore." Every line of dead code in a legacy system acts as a cognitive tax on the developer (and the AI agent) trying to understand the execution path. However, asking an LLM to "find unused code" by reading files is incredibly dangerous; LLMs cannot hold a whole repository in memory and will hallucinate dependencies, leading to broken builds.
Dead code elimination must be treated as a Graph Reachability Problem, orchestrated deterministically by the agent, with the LLM acting only as a safety reviewer for dynamic edge cases.
Here is the architecture for an agent-driven Dead Code Purging pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM guess what is executed. Use tools that mathematically prove reachability.
* The Repository Graph (Built Previously): The repository_graph.json we constructed earlier is the primary engine. Any node (function, class, file) that is not an explicitly defined Entrypoint and has an in-degree of 0 (no edges pointing to it) is mathematically dead.
* Language-Specific Static Analyzers: * Python: vulture
    * TypeScript/JS: ts-prune or eslint-plugin-unused-imports
    * Java/C#: Compiler warnings for unreferenced symbols.
* Ripgrep (rg): The fastest tool for string searching. Crucial for the "Safety Check" to ensure a supposedly dead class isn't being instantiated via reflection, Dependency Injection (DI) containers by its string name, or dynamic routing.
* Tree-sitter: Used not just for reading, but for AST-aware deletion. Simply deleting lines based on regex leaves dangling commas or broken brackets. Tree-sitter ensures the AST remains structurally valid after the node is excised.
2. The Pipeline: Tools to Build in Python
The agent needs a pipeline that identifies orphans, double-checks them against dynamic traps, and physically excises them while keeping the syntax tree valid.
1. The Orphan Hunter: A script that queries the repository_graph.json. It filters out known Entrypoints (routes, event listeners, public APIs exported from MODULE.yaml) and returns a list of all nodes with 0 incoming [CALLS] or [IMPORTS] edges.
2. The Dynamic Trap Scanner: A defensive script. It takes the name of every orphaned node and uses ripgrep to search the entire codebase for that exact string literal. This prevents deleting a class that is instantiated dynamically (e.g., container.resolve("PaymentService")).
3. The AST Surgeon: A script utilizing Tree-sitter's manipulation capabilities. Given a file and a specific node (e.g., a function definition), it safely removes the exact byte-range of that node from the file and saves it.
4. The Build Verifier: A script that simply runs the compiler (tsc, mvn compile, etc.) or the test suite immediately after a deletion to mathematically prove the excision did not break the system.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository.
* repository_graph.json (The structural map).
* boundaries.json (The list of valid entrypoints).
Outputs from the Pipeline:
* A tombstone_report.json containing the metadata of what was killed.
* A verifiable git diff ready for compilation/testing.
4. The Prompts
Divide the operation into mechanical harvesting and semantic safety verification.
Phase 1: Mechanical Harvesting (The Setup)
Goal: Force the agent to rely entirely on graph math and static tools to generate a list of candidates, then mechanically excise them.
Prompt:
"You are a Structural Reachability Engine. Your task is to safely identify and excise dead code from the target repository.
1. Write a Python script to query repository_graph.json. Identify all Nodes (Functions, Classes, Variables) that have an in-degree of 0 (no incoming edges).
2. Filter out any Nodes that are listed in boundaries.json as valid system Entrypoints.
3. For each remaining orphaned Node, execute a ripgrep search across the codebase for its exact string name. If the string appears anywhere outside its own definition (e.g., in a config file or DI container), flag it as 'Unsafe to Delete' and remove it from the list.
4. For the 'Safe to Delete' candidates, use Tree-sitter to safely excise the AST nodes from their respective files.
5. Run the project's build command to verify compilation.
Output tombstone_report.json detailing exactly which functions/classes were deleted and the bytes recovered."
Phase 2: The Semantic Safety Review (LLM Native Task)
Goal: In legacy systems, sometimes code is dead for a reason (a half-finished feature), and sometimes it's dead by accident (a bug disconnected a critical workflow). The LLM must review the intent of the deleted code before approving the PR.
Prompt:
"You are an Architecture Reviewer. I am providing you with tombstone_report.json and the git diff of the dead code that was just mechanically excised from our repository.
The static analysis proves this code is unreachable. However, before we finalize this deletion, perform the following semantic review:
1. Intent Analysis: Read the names and logic of the deleted functions. Do these represent a deprecated legacy feature, or do they look like a critical business rule that was accidentally disconnected (e.g., a TaxCalculator that is no longer wired to the checkout route)?
2. Dependency Debris: Did deleting these functions leave behind empty files, unused imports at the top of the file, or empty configuration blocks?
3. Database Impact: Do any of the deleted classes represent ORM entities or database schemas? If so, flag this; we cannot safely delete the code without also generating a database migration to drop the corresponding tables.
Output an 'Excision Approval Report'. Explicitly flag any deleted logic that looks like an accidental disconnection of a core business requirement."



* Execution Tracing: When legacy code produces highly illogical outputs, agents reverse-engineer the exact sequence of operations and state mutations responsible for the anomaly.


Execution Tracing bridges the most dangerous gap in legacy code analysis: the difference between what the code says it does (Static Analysis) and what it actually does at runtime (Dynamic Analysis).
Static tools (ASTs, Tree-sitter) are blind to polymorphism, dynamic Dependency Injection, reflection, and configuration-gated routing. If you ask an LLM to read a legacy controller, it might confidently tell you it calls PaymentProcessor.charge(). But at runtime, the DI container injects MockPaymentProcessor or LegacySoapPaymentProcessor based on an obscure environment variable.
To map execution reality, the AI agent cannot just read code; it must instrument, execute, and observe.
Here is how to architect an agent-driven Execution Tracing pipeline.
1. The Substrate: Existing Tools to Leverage
Do not ask the LLM to mentally simulate the call stack. Use industry-standard observability tools to capture the exact mathematical reality of the runtime.
* OpenTelemetry (OTel): The absolute gold standard. The agent can inject OTel auto-instrumentation agents into the runtime (Java, Python, Node, etc.) to capture spans, trace IDs, and database queries without heavily modifying the source code.
* Language-Specific Profilers: Tools like sys.settrace() or cProfile in Python, or JVM agents in Java. These generate exact dynamic call trees.
* eBPF (Extended Berkeley Packet Filter): For low-level or containerized environments where the agent cannot modify the application at all, eBPF can trace system calls, network egress, and file reads transparently.
* Database Query Loggers: Enabling query logging on a local sandbox database (e.g., Postgres log_statement = 'all') to capture the exact SQL generated by opaque ORMs.
2. The Pipeline: Tools to Build in Python
The AI agent must orchestrate a pipeline that safely attaches a "wiretap" to the system, triggers an execution path, and parses the resulting telemetry.
1. The Instrumentor: A script that programmatically attaches a tracing agent (like an OTel SDK or a simple decorator/wrapper) to a specific Entrypoint or "Monster Method." If standard tools fail, it injects "Sensing Variables" (from Chapter 22) to log internal state mutations.
2. The Sandbox Driver: A script that executes a Characterization Test (from Chapter 13) or a synthetic HTTP request against the instrumented code in an isolated local environment.
3. The Trace Distiller: Raw trace dumps (like Jaeger JSON or profiler outputs) are too massive for an LLM context window. This script parses the raw dump, filtering out framework noise (e.g., filtering out 500 lines of Spring Boot initialization) and leaving only the domain-specific function calls, variable states, and external I/O.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Method / Entrypoint.
* Execution Trigger: A specific test file, a cURL command, or a mock event payload.
* Noise Filter: A list of packages to ignore in the trace (e.g., exclude=["java.util.*", "org.springframework.*"]).
Outputs from the Pipeline:
* A dynamic_trace.json file.
* Schema: A chronological tree of Execution_Spans. Each span contains Invoked_Method, Concrete_Class_Executed (resolving polymorphism), Arguments_Passed, Return_Value, and Side_Effects (e.g., SQL executed).
4. The Prompts
Divide the operation into mechanical execution and semantic synthesis.
Phase 1: Mechanical Instrumentation & Execution
Goal: Force the agent to wiretap the application and trigger a real execution, proving what the machine actually does.
Prompt:
"You are a Dynamic Analysis Operator. Your task is to capture the exact runtime execution trace of [Target Method] when triggered by [Execution Trigger].
1. Write a Python script to attach a localized profiler or OpenTelemetry tracer to the target environment. Configure the tracer to filter out framework-level execution (e.g., ignore node_modules or vendor directories) and only capture application-level domain logic.
2. Execute the provided test harness or trigger command to fire the code path.
3. Capture the resulting trace output. Parse the raw trace to extract the sequential call tree, identifying the concrete implementations of any interfaces invoked, the exact arguments passed at runtime, and any SQL queries or network calls executed.
4. Shut down the sandbox.
Output the distilled call tree as dynamic_trace.json."
Phase 2: Trace Synthesis & Anomaly Detection (LLM Native Task)
Goal: Now that you have the mathematical proof of execution, use the LLM to explain the "why" behind the trace, especially when the output defies static expectations.
Prompt:
"You are an Execution Reality Analyst. I am providing you with dynamic_trace.json, representing the actual runtime behavior of a legacy workflow, alongside the static repository_graph.json of the same module.
Perform the following operations:
1. Reality Reconciliation: Compare the dynamic trace to the static graph. Identify any 'hidden' execution paths—instances where the static graph indicated a call to Interface A, but the runtime trace proves Concrete Class B was injected and executed.
2. State Mutation Mapping: Trace the lifecycle of the core domain entity passed into the entrypoint. List every function in the trace that mutated this entity's state before the final database persistence.
3. Anomaly Detection: Identify any redundant database queries (e.g., the N+1 problem) or circular executions that occurred during this trace.
4. Behavioral Summary: Write a plain-English explanation of exactly how the system processed the input, explicitly naming the configuration flags or DI bindings that dictated the execution path.
Output an execution_reality_report.md detailing these findings."



* Configuration Archaeology: Agents extract all runtime configuration reads (environment variables, config files, feature flags, DI bindings) and map them to the execution branches they gate, producing a configuration-to-behavior matrix that static analysis alone cannot reveal.


Configuration Archaeology addresses a blind spot in every static analysis pipeline described so far. The Repository Graph, the Semantic Clusters, and the Variable Lifecycle Map all represent the code as it is written. But legacy systems often contain entire execution paths that are invisible to static analysis because they are gated by runtime configuration: environment variables, feature flags, config files, DI container bindings, database-driven toggles, and command-line arguments.
A function that statically calls PaymentProcessor.charge() might, at runtime, call MockPaymentProcessor.charge() because a DI container binding was set in a YAML file three directories away. A 500-line if/else chain might have only two branches ever activated in production because the other eight are controlled by feature flags that have been OFF for three years.
If the agent does not map configuration to behavior, every other analysis (Effect Analysis, Execution Tracing, Characterization Testing) operates on an incomplete model of reality.
Here is how to architect an agent-driven Configuration Archaeology pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM read config files and guess their effects. Use deterministic extraction to map every config read to its consuming code.
* Tree-sitter / AST Scanners: To scan every source file for configuration read patterns. These are language-specific: process.env.X in Node, os.environ["X"] in Python, @Value("${X}") in Spring, ConfigurationManager.AppSettings["X"] in .NET.
* Ripgrep (rg): For high-speed scanning of config file formats (.env, .yaml, .json, .xml, .properties, .toml) to enumerate all defined configuration keys and their values per environment.
* The Repository Graph (Built Previously): To connect the config-reading code locations to the methods and modules that depend on them, revealing which execution paths are config-gated.
* DI Container Analyzers: Framework-specific tools or scripts that parse DI registration files (Spring XML/Java config, .NET Startup.cs, Angular modules, inversify bindings) to extract the concrete class bound to each interface at each profile/environment level.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that enumerates config keys, traces their consumption points, and maps the behavioral branches they control.
1. The Config Key Enumerator: A script that scans all config source files (.env, .yaml, .json, .properties, Dockerfile ENV directives, CI/CD pipeline variables) and produces a flat registry of every defined configuration key, its default value, and which environments override it.
2. The Config Read Scanner: A script using Tree-sitter that scans every source file for config read patterns (language-specific). For each read, it records: the config key being read, the file and line number, the enclosing function, and how the value is used (assigned to a variable, used in a conditional, passed to a constructor).
3. The Branch Gate Mapper: A script that identifies every conditional statement (if, switch, ternary) where the condition variable is derived from a config read. It maps the config key to the set of code branches it enables or disables.
4. The DI Binding Resolver: A script that parses DI container configuration (Spring XML, inversify modules, .NET service registrations) and resolves which concrete class is bound to which interface for each environment profile. This directly reveals the runtime class substitutions invisible to the static call graph.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository/Directory.
* Framework Profile: The config reading patterns for this language/framework.
* Environment List: The set of environments to analyze (e.g., development, staging, production).
Outputs from the Pipeline:
* A config_behavior_matrix.json file.
* Schema: * config_keys: Array of every configuration key with its values per environment.
    * consumption_points: For each key, the exact file, line, function, and usage pattern (conditional gate, constructor injection, value assignment).
    * gated_branches: For each config key used in a conditional, the code blocks that are enabled vs. disabled per environment.
    * di_bindings: For each interface, the concrete class bound per environment profile.
    * dead_config: Keys defined in config files but never read by any source file (candidates for cleanup).
    * phantom_reads: Config keys read by source code but never defined in any config file (runtime errors waiting to happen).
4. The Prompts
Divide the operation into mechanical extraction and semantic environment modeling.
Phase 1: Mechanical Config Extraction and Consumption Mapping
Goal: Force the agent to deterministically enumerate every config key and trace every consumption point in the source code.
Prompt:
"You are a Configuration Topology Engine. Your task is to map every runtime configuration key in the target repository and trace its effect on execution behavior.

2. Write a script using Tree-sitter to scan all source files for configuration read patterns matching the provided Framework Profile. For each read, record the config key, file path, line number, enclosing function, and usage context (conditional, assignment, constructor argument).
3. For each config read used in a conditional expression, extract both branches of the conditional and label them with the config value that activates each.
4. Parse any DI container configuration files. For each interface binding, record the interface name, the concrete class bound, and the environment/profile that activates the binding.
Output config_behavior_matrix.json containing the complete config-to-behavior mapping."
Phase 2: Environment Divergence Analysis (LLM Native Task)
Goal: Now that the math has mapped every config gate, use the LLM to identify dangerous environment divergences and hidden behavioral assumptions.
Prompt:
"You are a Runtime Environment Analyst. I am providing you with config_behavior_matrix.json, which maps every configuration key to the code branches it gates across all environments.
Perform the following operations:
1. Environment Divergence Report: Identify config keys where the value differs between environments in ways that change critical execution paths (e.g., a payment processor mock in development vs. a real processor in production). Rank by business criticality.

* State Machine Extraction: Agents identify implicit state machines scattered across legacy code—order statuses, workflow stages, approval chains encoded as ad-hoc if/else chains and database enum columns—and formalize them into explicit state transition diagrams with guard conditions, actions, and terminal states.


State Machine Extraction addresses one of the most dangerous forms of implicit knowledge in legacy systems. Business workflows (order processing, claim adjudication, user onboarding, approval chains) are almost always state machines. But in legacy code, the state machine is never formalized. Instead, it exists as:
* A database column (status VARCHAR(20)) with no documented set of valid values.
* Scattered if (order.status == "PENDING") checks across 15 different files.
* Transitions that happen in different layers (a controller sets status to "PROCESSING," an event handler sets it to "COMPLETED," a cron job sets it to "EXPIRED").
* Guard conditions encoded as business rules inside unrelated methods.
If you ask an LLM to "find the state machine," it will read a few files and produce a plausible but incomplete diagram. It will miss transitions that happen in event handlers, cron jobs, or database triggers. The agent must mechanically harvest every state reference and every transition from the entire codebase.
Here is how to architect an agent-driven State Machine Extraction pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM construct the state machine from a few files. Use exhaustive codebase scanning to find every state mention.
* Tree-sitter: For AST extraction of every comparison, assignment, and switch statement involving the state-carrying field.
* Ripgrep (rg): For high-speed scanning of string literals that represent state values (e.g., "PENDING", "APPROVED", "CANCELLED"). State values are almost always string or enum constants.
* The Repository Graph (Built Previously): To trace which modules and methods are responsible for each state transition.
* Database Schema / Migration Files: To identify the columns that carry state (usually status, state, phase, step columns in ORM models or migration DDL).
* The Event Mesh (Built Previously): To trace asynchronous state transitions triggered by event handlers or queue consumers.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that harvests states, harvests transitions, and constructs the formal state machine.
1. The State Field Identifier: A script that scans ORM models, database migration files, and type definitions to identify fields likely carrying state. Heuristics: field named status, state, phase, step, stage; field typed as enum; field with a constrained set of string values in validation rules.
2. The State Value Harvester: A script using ripgrep and Tree-sitter that scans the entire codebase for every comparison (==, ===, .equals(), switch case) and every assignment (=, setter) involving the identified state fields. For each, it records: the state value (string/enum), the operation (READ or WRITE), the file, the line, and the enclosing function.
3. The Transition Extractor: A script that pairs WRITE operations (transitions) with their preceding READ operations (guard conditions). If a method reads status == "PENDING" and then writes status = "APPROVED", that is a transition: PENDING -> APPROVED, guarded by the conditions in the enclosing if block, executed by this specific method.
4. The State Machine Assembler: A script using NetworkX that constructs a directed graph where nodes are state values and edges are transitions. Each edge is annotated with: the guard condition, the method that executes the transition, and the file path. The script identifies terminal states (nodes with zero outgoing transitions), initial states (nodes with zero incoming transitions from other states, only from initialization), and unreachable states (nodes with zero incoming transitions and no initialization path).
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Entity: The domain entity whose state machine is being extracted (e.g., "Order", "Claim", "User").
* State Field Name: The specific field carrying state (e.g., "status", "phase").
* Target Repository/Directory.
Outputs from the Pipeline:
* A state_machine.json file.
* Schema: * states: Array of all discovered state values with metadata (initial, terminal, intermediate, unreachable).
    * transitions: Array of edges, each containing: from_state, to_state, guard_condition (the if/switch logic), action (side effects executed during transition), triggered_by (method name and file), trigger_type (SYNCHRONOUS, EVENT, CRON, MANUAL).
    * orphan_states: State values found in comparisons but never assigned (dead states).
    * orphan_transitions: Assignments to state values that are never read anywhere (transitions to nowhere).
4. The Prompts
Divide the operation into mechanical harvesting and semantic formalization.
Phase 1: Mechanical State and Transition Harvesting
Goal: Force the agent to exhaustively scan the codebase for every mention of the state field, constructing the raw transition table from code facts.
Prompt:
"You are a State Machine Extractor. Your task is to reconstruct the implicit state machine for the [Entity Name] entity, tracked by the field [State Field Name].
1. Write a Python script to scan ORM models, type definitions, and migration files. Identify the field [State Field Name] and extract any declared enum values or validation constraints.
2. Use ripgrep to find every string literal or enum constant in the codebase that matches the known state values. Expand the state set with any previously unknown values discovered.
3. Use Tree-sitter to parse every source file. Find every comparison (==, switch case) and every assignment involving [State Field Name]. Record: the state value, the operation (READ/WRITE), the enclosing function, the file path, and the line number.
4. Pair consecutive READ-then-WRITE operations within the same method or execution path to construct transitions. Record the guard condition (the if/switch logic) and any side effects (method calls, event emissions) between the read and the write.
5. Construct a directed graph using NetworkX. Nodes = state values. Edges = transitions.
Output state_machine.json containing the complete state graph with all annotations."
Phase 2: Semantic Validation and Formalization (LLM Native Task)
Goal: Now that the math has reconstructed the state machine from code facts, use the LLM to validate it against business intent and identify anomalies.
Prompt:
"You are a Business Process Analyst. I am providing you with state_machine.json, which was mechanically reconstructed from the legacy codebase for the [Entity Name] entity.
Perform the following operations:
1. Completeness Check: Does the state machine have exactly one initial state and at least one terminal state? If not, flag the structural anomaly.
2. Impossible Transition Detection: Are there any transitions that should not logically exist? (e.g., CANCELLED -> APPROVED, REFUNDED -> PENDING). Flag transitions that violate common business sense for this domain.
3. Missing Transition Detection: Are there state pairs where a transition logically should exist but no code path was found? (e.g., no path from PROCESSING to ERROR). Flag gaps.
4. Concurrent Transition Hazard: Are there multiple methods in different modules that execute the same transition (e.g., both the API controller and the event handler can set status to COMPLETED)? Flag race condition risks.
5. Formalization: Produce a clean state transition table and, if possible, a Mermaid or PlantUML state diagram for human review.
Output a state_machine_report.md containing the validated state machine, the anomaly flags, and the visual diagram."


* Error Handling Archaeology: Agents trace every exception throw, catch, propagation, and swallowing path through the codebase, classifying handling strategies and identifying silent failure black holes where exceptions are caught and discarded without logging or recovery.


Error Handling Archaeology addresses a problem that undermines every other pipeline in this document. If the legacy code silently swallows exceptions, then Characterization Tests will pass on inputs that should produce errors, locking in broken behavior as "correct." If error handling is inconsistent, then Effect Analysis will miss propagation paths where errors are caught and converted to null returns or default values.
Legacy error handling is almost never consistent. Different developers, different eras, different frameworks. Some paths throw and let the caller deal with it. Some catch, log, and rethrow. Some catch and return null. Some catch and silently continue. Some catch a broad Exception type and swallow everything, including NullPointerExceptions that indicate real bugs.
The agent must not review error handling file by file. It must construct a global exception flow graph: where exceptions originate, where they are caught, what happens at each catch point, and where they silently vanish.
Here is how to architect an agent-driven Error Handling Archaeology pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM read catch blocks and guess whether they are dangerous. Use AST extraction to build a complete exception flow map.
* Tree-sitter: For extracting all throw statements, try/catch blocks, catch clause types, and the bodies of catch blocks across every file.
* Language-Specific Type Resolvers: TypeScript Compiler API, javac, or mypy for resolving which exception types are actually thrown by each method (checked exceptions in Java, typed throws in Swift). In dynamic languages, this must fall back to AST pattern matching.
* The Repository Graph (Built Previously): To trace the call chain from a throw site to the nearest enclosing catch block, which may be several call frames up the stack.
* Joern / Code Property Graphs: If available, Joern can model exception flow as part of its inter-procedural control flow graph, directly answering "where does this exception end up?"
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that maps every throw, every catch, and the relationship between them.
1. The Throw Harvester: A script using Tree-sitter that scans every source file and extracts all throw (or raise) statements. For each, it records: the exception type thrown, the file, the line, the enclosing method, and whether the throw is inside a try block within the same method.
2. The Catch Harvester: A script that extracts all try/catch blocks. For each catch clause, it records: the exception type caught (or bare catch/except), the body of the catch block classified as one of: RETHROW (throws the same or a different exception), LOG_AND_RETHROW, LOG_AND_SWALLOW, RETURN_NULL, RETURN_DEFAULT, CONVERT_TO_ERROR_CODE, EMPTY_CATCH (completely empty body), CUSTOM_HANDLING.
3. The Exception Flow Linker: A script that traces the path from each throw site to its nearest matching catch clause. It walks up the call stack using the repository graph. If the enclosing method has a try/catch that matches the exception type, the flow terminates there. If not, the script traverses to each caller and checks their try/catch blocks. If no catch is found at any level, the exception is marked as UNCAUGHT (will crash the process or hit the global handler).
4. The Black Hole Detector: A script that flags the most dangerous pattern: catch blocks classified as EMPTY_CATCH or LOG_AND_SWALLOW that catch broad exception types (Exception, Error, Throwable, BaseException). These are the silent failure points that corrupt system state without any observable signal.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository/Directory.
* Language Profile: The exception throwing and catching syntax for the target language.
Outputs from the Pipeline:
* An exception_flow_graph.json file.
* Schema: * throw_sites: Array of every throw/raise, with exception type, location, and enclosing method.
    * catch_sites: Array of every catch/except, with caught type, handling classification, and location.
    * flows: Array of linked paths from throw to catch, showing the call stack frames traversed.
    * black_holes: Catch blocks that swallow exceptions silently.
    * uncaught_exceptions: Throw sites with no matching catch anywhere in the call chain.
    * broad_catches: Catch blocks catching overly broad types (Exception, Error) that may accidentally intercept unrelated failures.
4. The Prompts
Divide the operation into mechanical flow mapping and semantic risk assessment.
Phase 1: Mechanical Exception Flow Mapping
Goal: Force the agent to build the complete exception flow graph using AST extraction and call graph traversal.
Prompt:
"You are an Exception Flow Engine. Your task is to map every exception origin and destination in the target codebase.
1. Write a Python script using Tree-sitter to extract all throw/raise statements. Record the exception type, file, line, and enclosing method.
2. Extract all try/catch blocks. Classify each catch body as: RETHROW, LOG_AND_RETHROW, LOG_AND_SWALLOW, RETURN_NULL, RETURN_DEFAULT, CONVERT_TO_ERROR_CODE, EMPTY_CATCH, or CUSTOM_HANDLING.
3. For each throw site, trace the exception to its nearest matching catch. Walk up the call stack using repository_graph.json. If no local catch exists, traverse to each caller and check their catch blocks. Continue until a match is found or the call stack is exhausted (UNCAUGHT).
4. Flag all catch blocks classified as EMPTY_CATCH or LOG_AND_SWALLOW that catch broad exception types.
Output exception_flow_graph.json containing the complete throw-to-catch mapping, the black holes, and the uncaught exceptions."
Phase 2: Semantic Failure Mode Analysis (LLM Native Task)
Goal: Now that the math has mapped the exception flows, use the LLM to assess the business impact of each silent failure point and prioritize remediation.
Prompt:
"You are a Failure Mode Analyst. I am providing you with exception_flow_graph.json, which maps every exception path in the legacy codebase.
Perform the following operations:
1. Black Hole Triage: For each silent swallowing point (EMPTY_CATCH or LOG_AND_SWALLOW), read the surrounding code context. Determine whether the swallowed exception represents: a recoverable condition (acceptable), a data integrity risk (critical), or a security vulnerability (urgent).
2. Error Conversion Audit: For catch blocks that RETURN_NULL or RETURN_DEFAULT, trace the callers. Do the callers check for null? If not, the error is being silently converted into a NullPointerException several frames later, making debugging nearly impossible.
3. Broad Catch Contamination: For each broad catch (catching Exception or Error), list the specific exception types it is accidentally intercepting beyond the intended type. These are bugs hiding behind overly generous error handling.
4. Characterization Test Impact: Identify any throw sites that are in the affected set of a planned change but whose exceptions are swallowed before reaching the test harness. These paths will produce false-passing characterization tests.
Output an error_handling_report.md containing the triaged black holes, the null-conversion chains, the broad catch contamination list, and the characterization test blind spots."


3. Test Harness Generation and Characterization
* Characterization Scaffolding: Agents generate immediate testing bounds for opaque methods, covering standard paths, boundary conditions, and null inputs with placeholder assertions to lock in existing operational reality.


Characterization Scaffolding is the mechanical implementation of Chapter 13. In legacy systems, you do not write tests to verify what the code should do (because requirements are lost or the code is too messy). You write tests to lock in what the code actually does.
If you prompt an LLM to "write unit tests for this legacy function," it will hallucinate. It will read the function name, guess the intended business logic, and write assertions based on that guess. Those tests will immediately fail against the messy reality of the actual code.
An AI agent must not guess. It must build a "vise." It should generate a testing harness, feed it permutations of inputs, intentionally trigger test failures to capture the runtime outputs, and then auto-patch the tests to lock in those exact outputs.
Here is the architecture for an agent-driven Characterization Scaffolding pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM execute code in its head. Use the actual compiler and test runner as the source of truth.
* Tree-sitter: To parse the target method signature and extract the exact types and number of required parameters.
* Property-Based Testing / Fuzzers: Libraries like Hypothesis (Python), fast-check (JS/TS), or jqwik (Java). Instead of the LLM guessing 3 inputs, these tools generate hundreds of edge cases (nulls, negative limits, maximum strings, empty arrays).
* Standard Test Runners: pytest, Jest, JUnit. The agent must be able to execute these programmatically and capture the stderr/stdout.
* Coverage Tools: coverage.py, Istanbul, JaCoCo. The agent uses this to mathematically prove that the generated characterization tests actually traversed the internal branches of the legacy method.
2. The Pipeline: Tools to Build in Python
The agent orchestrates a feedback loop: generate inputs, fail tests, read failures, patch tests, measure coverage.
1. The Signature Fuzzer: A script that uses Tree-sitter to read the target method's parameter types. It generates a data matrix of inputs: standard happy-path data, boundary conditions (0, -1, INT_MAX), and invalid states (null, empty strings).
2. The Harness Generator: A script that writes the physical test file. It loops through the data matrix, calls the legacy function, and asserts against an intentionally impossible value (e.g., assertEquals("GENERATED_AGENT_PLACEHOLDER", result)).
3. The Assertion Patcher: The core feedback loop. A script that runs the test suite, parses the test failure logs (e.g., AssertionError: expected 'GENERATED_AGENT_PLACEHOLDER' but got '{"status": "partial", "tax": 14.5}'), and uses regex or AST manipulation to rewrite the test file, replacing the placeholder with the actual captured output.
4. The Coverage Verifier: A script that runs the coverage tool after patching. If branch coverage is below a threshold (e.g., 85%), it flags the method as possessing hidden states requiring LLM semantic analysis.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target File and Method Name.
* Framework Profile: The specific testing and mocking framework to use (e.g., "Use Jest and Jest.mock").
* Test Location: The parallel directory where the test should be written (from Chapter 18: src/test/...).
Outputs from the Pipeline:
* A compiling, passing [MethodName]Characterization.test.js file.
* A coverage_report.json for that specific method.
4. The Prompts
Divide the operation into the mechanical generation loop and the semantic coverage review.
Phase 1: Mechanical Generation & Patching (The Feedback Loop)
Goal: Force the agent to treat the legacy code as a black box, using the test runner to extract its true behavior.
Prompt:
"You are a Characterization Test Engineer. Your task is to build a testing vise around [Target Method] in [File Path]. Do NOT attempt to guess what the method should return.
1. Use Tree-sitter to analyze the method signature and its required parameters.

3. For each input, write a test that calls the method and asserts the result equals the exact string "AGENT_PLACEHOLDER".
4. Execute the test runner. Capture the failure logs.
5. Write a script to parse the actual returned values from the failure logs and overwrite the "AGENT_PLACEHOLDER" strings in the test file with the true outputs.
6. Re-run the tests to prove they now pass.
Output the finalized, passing test file and the execution logs."
Phase 2: Semantic Coverage Review (LLM Native Task)
Goal: If the mechanical fuzzer only hits 60% of the code branches, the LLM must analyze the static code to figure out what bizarre, highly specific input is required to unlock the remaining branches.
Prompt:
"You are an Edge Case Analyzer. I am providing you with the source code of [Target Method] and its newly generated coverage_report.json.
The mechanical characterization tests achieved only [X]% branch coverage. There are conditional branches in this legacy code that our fuzzer failed to trigger.
Perform the following operations:
1. Dead End Analysis: Identify the specific if/else blocks, switch statements, or try/catch blocks that show 0% execution in the coverage report.
2. Constraint Solving: Read the logic leading up to those dead ends. Deduce the exact, specific input values, global states, or mocked return values required to force the execution path into those branches.
3. Targeted Scaffolding: Generate the exact test code required to trigger those specific missing branches. Format them as additional test cases to be appended to the existing suite.
Output the supplementary test cases."



* Variable Scope Mapping: For untestable "monster methods," agents generate Directed Acyclic Graphs (DAGs) of local variables, identifying safe extraction points or optimal locations to inject "Sensing Variables." 

Variable Scope Mapping is the mechanical implementation of the survival tactics found in Chapter 22 ("I Need to Change a Monster Method and I Can't Write Tests for It").
When you confront a 3,000-line procedural function, the barrier to refactoring is state mutation. You cannot extract lines 500–600 into a new method if you do not know exactly which of the 45 local variables are read from, written to, or ignored within that specific block.
If you ask an LLM to "extract methods from this giant function," it will hallucinate variable scopes. It will generate helper methods that forget to return mutated state, or it will pass 15 arguments into a function that only needs 2.
To solve this, the AI agent must act as a compiler frontend, building an intra-procedural Data Dependency Graph (DAG) to find mathematically provable "seams" in the monolith.
Here is the architecture for an agent-driven Variable Scope Mapping pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM simulate the call stack or track variable mutations in its context window. It will lose track. You must use deterministic static analysis.
* Tree-sitter: For precise Abstract Syntax Tree generation. You need to distinguish between variable declarations, variable reads, and variable writes (mutations).
* Language Server Protocol (LSP) / Compiler Frontends: Tools like eslint (JS/TS), javaparser (Java), or Python's native ast module. These tools inherently understand "Def-Use Chains" (where a variable is defined, and everywhere it is used).
* Joern / Code Property Graphs (CPG): Excellent for mapping data flow inside a single function, allowing you to query exactly where a specific variable travels.
* NetworkX (Python): To build the DAG. The nodes are the lines of code (or code blocks); the edges are the variables passed between them.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that dissects the monster method into discrete blocks based on variable lifecycles.
1. The Def-Use Extractor: A script utilizing Tree-sitter to parse the monster method. It creates a ledger for every local variable, logging the exact line number where it is initialized, every line where it is read, and every line where its state is mutated.
2. The Liveness Calculator: A script that calculates "Variable Liveness"—determining the active lifespan of a variable. If taxRate is declared on line 10 and last read on line 50, it is "dead" after line 50.
3. The Seam Finder: The most critical algorithm. A script that scans the method for "bottlenecks." It looks for line ranges (e.g., lines 200-300) that have minimal overlap with active variables from the rest of the function. These are your mathematically safe extraction points.
4. The Sensing Variable Injector: If no clean seams exist, this script calculates the optimal line to inject a temporary array or logger (a "Sensing Variable") to capture internal state without breaking the method's signature.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target File and Monster Method Name.
* (Optional) Target Line Range: If the developer only cares about modifying a specific 50-line chunk of the 3,000-line method.
Outputs from the Pipeline:
* A variable_lifecycle_map.json file.
* Schema: * Variables: Arrays containing name, type, declared_at, mutated_at[], read_at[].
    * Safe_Extractions: Arrays of line ranges, detailing exactly which variables must be passed in as arguments, and which must be returned.
4. The Prompts
Divide the operation into mechanical lifecycle extraction and architectural segmentation.
Phase 1: Mechanical Data Flow Extraction
Goal: Force the agent to rely on AST traversal to build an irrefutable ledger of variable mutations.
Prompt:
"You are an Intra-Procedural Data Flow Engine. Your task is to map the lifecycle of all local variables inside the monster method [Method Name] in [File Path].
1. Write a Python script using Tree-sitter to parse this specific method.
2. For every local variable declared, map its 'Def-Use Chain'. Record the line number of its declaration, every line where it is mutated (written), and every line where it is accessed (read).
3. Calculate the 'Liveness' span of each variable (the range from its first declaration to its final read).
4. Scan the method for structural bottlenecks. Identify contiguous line blocks (minimum 20 lines) that read from 3 or fewer external local variables and mutate 1 or fewer external local variables.
Output variable_lifecycle_map.json containing the Def-Use chains and the identified bottleneck blocks."
Phase 2: Architectural Segmentation (LLM Native Task)
Goal: Now that the math has proven where the code is loosely coupled internally, use the LLM to design the exact refactoring commands (Extract Method or Introduce Sensing Variable).
Prompt:
"You are a Legacy Refactoring Specialist. I am providing you with the variable_lifecycle_map.json for a monster method, alongside the raw code of the method.
We need to modify the logic around line [Target Line Number].
Perform the following operations:
1. Extraction Design: Review the 'Safe Extractions' bottlenecks identified by the static analysis around our target line. Select the safest block to extract.
2. Signature Generation: Draft the exact method signature for the newly extracted method. Explicitly list the local variables that must be passed in as arguments (because they are read) and the single variable that must be returned (because it is mutated and read later).
3. Fallback Strategy (Sensing Variables): If the data map shows that the target block mutates multiple variables that are heavily relied upon later in the monolith (making extraction impossible in many languages), do NOT extract. Instead, write the exact code to declare a 'Sensing Variable' at the top of the monolith, append the internal state to it at the target line, and return it for a test harness to read.
Output the specific refactoring instructions and the new method signatures."



* Procedural Test Seams: In non-OO languages (C/C++), agents extract core computation from global state dependencies, parameterizing functions or generating function pointer headers to enable mock injection.


1. The Substrate: Existing Tools to Leverage
Procedural languages (C, C++, Go) lack native object-oriented polymorphism. Dependencies are hardcoded via direct function calls or global memory space. An LLM cannot mock these using standard testing frameworks. The agent must manipulate the compiler, the preprocessor, or the Abstract Syntax Tree to forcibly route execution paths.
* Tree-sitter (C/C++ grammars): Mandatory for precise AST extraction. Procedural code is heavily reliant on macros and global state; regex cannot reliably parse it.
* libclang / Clang AST: The industry standard for semantic analysis in C/C++. It resolves typedefs, macros, and memory layouts that Tree-sitter might miss as a pure syntactic parser.
* Make / CMake / Bazel: The build system is the primary dependency injection framework in procedural architecture. The agent must be capable of modifying build scripts to swap object files (Link Seams).
* C Preprocessor (cpp / gcc -E): Used to evaluate the code post-macro expansion to ensure injected Preprocessor Seams do not collide with existing definitions.
2. The Pipeline: Tools to Build in Python
The agent orchestrates a pipeline to manufacture artificial seams—transforming static bindings into dynamic ones for the test harness.
1. The Global State Extractor: A script utilizing libclang or Tree-sitter to scan a target function. It identifies every variable read or mutated that is not passed explicitly via the parameter list or instantiated locally.
2. The Parameterizer (Function Pointer Injector): A script that executes the "Extract Function" or "Parameterize Method" refactoring. It rewrites a C function signature to accept function pointers for its external dependencies, shifting control inversion to the caller.
3. The Preprocessor Seam Generator: A script that wraps direct system calls (e.g., malloc, read, or external hardware I/O) in C macros. It generates a test-specific header file that redefines these macros to point to mock functions during a test build.
4. The Link Seam Orchestrator: A script that parses a Makefile or CMakeLists.txt. It duplicates a build target, strips out the production dependency object file (e.g., hardware_driver.o), and substitutes a generated mock object file (mock_hardware_driver.o) for the test executable.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target File and Function Name.
* Target Dependency: The specific global variable or external function call that must be severed to achieve isolation.
* Build Configuration: The path to the Makefile or CMakeLists.txt.
Outputs from the Pipeline:
* Modified source file containing the newly manufactured seam.
* Generated mock_[dependency].c and mock_[dependency].h files.
* Modified build script containing the new test compilation target.

* Test Organization and Code-Test Mapping: Agents map every existing test file to the production code it exercises, identify production code with zero test coverage, detect orphaned tests that verify nothing, and propose reorganization plans that align test structure with architectural boundaries.


Test Organization and Code-Test Mapping is the mechanical implementation of Chapter 18 ("My Test Code Is in the Way"). In legacy systems, the test suite is often in worse shape than the production code. Tests are placed in arbitrary directories. A single test file exercises code from five different modules. Multiple test files duplicate the same setup logic. Some tests test through the UI or database, masquerading as "unit tests." Some tests pass but verify nothing (assertions against hardcoded values that were never validated).
Before an agent can execute Characterization Scaffolding or Effect Analysis, it must understand what test coverage already exists, where it exists, what it actually exercises, and what it misses. Without this map, the agent will generate duplicate characterization tests for code that is already covered, or skip code that appears covered but is actually verified by a broken test.
Here is how to architect an agent-driven Test Organization pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM read test file names and guess what they test. Use coverage data and import analysis to mathematically prove the test-to-production mapping.
* Coverage Tools: coverage.py (Python), Istanbul/c8 (JS/TS), JaCoCo (Java), lcov (C/C++). These produce file-level and line-level coverage reports that definitively prove which production lines each test exercises.
* Tree-sitter: For parsing test files to extract: which production modules they import, which test framework assertions they use, and whether those assertions are meaningful (asserting against a variable) or trivial (asserting true === true).
* The Repository Graph (Built Previously): For cross-referencing test imports against the module boundary structure, revealing tests that violate layer boundaries (a "unit test" that imports an infrastructure adapter).
* Test Runner Metadata: Most test frameworks (Jest, pytest, JUnit) can output structured test result files (JUnit XML, JSON reporters). These provide test names, execution times, pass/fail status, and suite groupings.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that builds the test-to-production mapping, identifies gaps, and classifies test quality.
1. The Coverage Mapper: A script that runs the full test suite with coverage enabled and parses the coverage output. For each production file, it records: total lines, covered lines, coverage percentage, and which specific test files contributed to that coverage.
2. The Test Import Analyzer: A script using Tree-sitter that parses every test file and extracts all imports of production code. This produces a static mapping (independent of coverage) showing which production modules each test file references.
3. The Assertion Quality Scanner: A script using Tree-sitter that parses every test file and classifies each assertion. Categories: MEANINGFUL (asserts against a runtime-computed value), TRIVIAL (asserts true, asserts against a hardcoded constant that matches the implementation), SNAPSHOT (asserts against a serialized snapshot—fragile but functional), NONE (test function contains no assertions—always passes).
4. The Orphan Detector: A script that cross-references the test import map against the repository graph. It identifies: orphaned tests (test files that import production modules which no longer exist), shadow tests (test files whose imports have drifted so far from the current production structure that they test outdated behavior), and boundary-violating tests (tests labeled as "unit" that import infrastructure or external service modules).
5. The Coverage Gap Calculator: A script that inverts the coverage map. Instead of "which tests cover this production file," it answers "which production files have zero coverage." It ranks uncovered files by their Hotspot Score (from the Hotspot Prioritization pipeline) to prioritize the most dangerous gaps.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository/Directory.
* Test Directory Convention: Where tests are located (e.g., __tests__/, test/, src/**/*.test.ts).
* Test Framework: The specific framework and assertion library in use.
Outputs from the Pipeline:
* A test_map.json file.
* Schema: * production_to_tests: For each production file, the list of test files that exercise it, with coverage percentage.
    * tests_to_production: For each test file, the list of production files it imports/exercises.
    * coverage_gaps: Production files with zero test coverage, ranked by Hotspot Score.
    * orphaned_tests: Test files importing non-existent production modules.
    * trivial_tests: Test files where >50% of assertions are classified as TRIVIAL or NONE.
    * boundary_violations: Test files labeled as "unit" that import infrastructure or framework modules.
    * reorganization_plan: Proposed test directory structure aligning tests with architectural module boundaries.

4. Architectural Decoupling and Isolation
* API / Framework Quarantine: Agents strip business rules from UI handlers, database callbacks, or heavy 3rd-party libraries, extracting pure data transformations into isolated, framework-agnostic classes (the Humble Object pattern).


API / Framework Quarantine is the mechanical implementation of Chapters 14 and 15 ("Dependencies on Libraries Are Killing Me" and "My Application Is All API Calls").
When business logic is interleaved with framework syntax (like React hooks, Express req/res objects, or Android UI event handlers), the logic becomes trapped. You cannot test a tax calculation if it requires spinning up a web server. The AI agent's job is to enforce the Humble Object Pattern: stripping the framework-bound file of all its intelligence until it is just a dumb router, and moving the intelligence into a Plain Old Data (POD) object or pure function.
Unlike dependency graphing, separating "business logic" from "framework routing" is a highly semantic task. It requires the LLM's reasoning to understand intent.
Here is the architecture for an agent-driven Framework Quarantine pipeline.
1. The Substrate: Existing Tools to Leverage
You need tools to locate the infection and a frontier LLM to perform the surgical extraction.
* Tree-sitter: To parse the AST and identify framework-specific constructs (e.g., classes extending HttpServlet, functions utilizing @RestController decorators, or React useEffect blocks).
* Regex / AST Import Scanners: To identify exactly which files are importing the volatile third-party library (e.g., tracking every file that imports stripe-node or aws-sdk).
* Frontier LLMs (GPT-4o, Claude 3.5 Sonnet, Gemini 1.5 Pro): The primary engine. The LLM excels at recognizing pure mathematical/business operations hidden inside network or UI callbacks.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that locates the coupled code, splits it into two distinct physical files, and rewires the connections.
1. The API Locator: A script that scans the codebase for direct imports of the target framework/library. It flags files where framework types (like HTTP Requests or UI Events) are passed deep into internal private methods.
2. The Humble Splitter (LLM Router): A script that feeds the coupled file to the LLM. It commands the LLM to separate the code into a "Pure Logic" module (zero framework imports) and a "Humble API" module (framework imports, but zero branching logic).
3. The Adapter Generator: For outbound dependencies (like a third-party vendor SDK), this script prompts the LLM to generate an internal Interface that represents only the features the application actually uses, plus the concrete wrapper class that implements it.
4. The Rewirer: A script that takes the outputs from the LLM, writes them to the file system, and updates the original call sites to inject the new Adapter or call the new Pure module.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target File: The coupled controller or UI component.
* Framework Profile: Explicitly naming the framework being quarantined (e.g., "AWS DynamoDB SDK" or "Express.js").
* Direction: Is this an Inbound quarantine (separating our logic from the UI/Network) or an Outbound quarantine (separating our logic from a vendor SDK)?
Outputs from the Pipeline:
* For Inbound: [ModuleName]Core.ts (Pure logic) and [ModuleName]Controller.ts (Humble framework handler).
* For Outbound: I[LibraryName]Adapter.ts (The Interface) and [LibraryName]Wrapper.ts (The concrete implementation).

* Interface Segregation: Agents analyze bloated classes and generate narrow, role-specific interfaces based on distinct client usage patterns, halting the expansion of God classes.


Interface Segregation is the mechanical implementation of the Interface Segregation Principle (ISP) applied to legacy God Classes. A God Class is a single class that accumulates dozens of public methods serving completely unrelated clients. A PaymentManager that exposes methods for charging cards, generating invoices, sending receipts, computing tax, and managing refunds violates ISP because the InvoiceRenderer should never depend on the same class as the FraudDetector. Every time a refund method changes, the invoice rendering code must be recompiled and redeployed even though it uses none of the refund logic.
If you ask an LLM to "split this class into smaller interfaces," it will group methods by name similarity or alphabetical proximity. That is wrong. The correct grouping is determined by client usage patterns: which external callers actually invoke which subset of methods. The AI agent must treat this as a bipartite graph problem, not a naming problem.
Here is how to architect an agent-driven Interface Segregation pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM guess which methods belong together. Use the structural graph to mathematically prove which callers use which methods.
* The Repository Graph (Built Previously): The repository_graph.json is the primary engine. It contains all [CALLS] edges, meaning you can query exactly which external files invoke which methods of the God Class.
* Tree-sitter: For flawless extraction of the God Class's full public method list, parameter types, and return types.
* NetworkX (Python): For bipartite graph construction and community detection. You build a bipartite graph where one partition is "external callers" and the other partition is "public methods of the God Class." Edges connect callers to the methods they invoke. Community detection (Louvain or label propagation) on this bipartite projection reveals the natural interface clusters.
* scikit-learn: As a fallback, if community detection produces ambiguous results, Spectral Clustering on the caller-method adjacency matrix provides a second opinion.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that maps caller-method relationships, clusters them mathematically, and generates the segregated interfaces.
1. The Client Usage Mapper: A script that queries repository_graph.json. For every external file that imports or calls into the God Class, it records exactly which public methods that caller invokes. Output: a bipartite adjacency matrix (callers x methods).
2. The Interface Cluster Engine: A script using NetworkX that projects the bipartite graph onto the methods partition. Two methods are connected if they share callers. Community detection over this projected graph reveals which methods are always used together by the same clients. These communities are the natural interfaces.
3. The Interface Generator: A script using Tree-sitter that reads the God Class, extracts the method signatures belonging to each cluster, and generates a new interface file per cluster. Each interface contains only the methods used by its client community.
4. The Implements Rewriter: A script that rewrites the God Class declaration to explicitly implement all generated interfaces. It also updates each caller's import to reference the narrow interface instead of the concrete God Class.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target God Class File and Class Name.
* repository_graph.json (The structural map with [CALLS] edges).
* Minimum Client Threshold: Ignore methods called by fewer than N external callers (to filter out internal-only helpers that should remain private).
Outputs from the Pipeline:
* A set of I[RoleName].ts (or .java, .py) interface files, each containing a narrow, role-specific subset of methods.
* A modified God Class file implementing all generated interfaces.
* A caller_migration.json mapping each external caller to its new narrow interface import.
4. The Prompts
Divide the operation into mechanical graph analysis and architectural naming.

Goal: Force the agent to use graph math to determine which methods are co-used by the same clients, proving the natural interface boundaries.
Prompt:

1. Write a Python script to query repository_graph.json. For every external file that contains a [CALLS] edge into [ClassName], record which specific public methods that caller invokes.
2. Construct a bipartite graph using NetworkX. Partition A = external caller files. Partition B = public methods of [ClassName]. Edges connect a caller to every method it invokes.
3. Project the bipartite graph onto the methods partition. Apply Louvain community detection to identify clusters of methods that are always co-used by the same set of callers.
4. Output the clusters as interface_clusters.json, where each cluster contains the method names and the list of callers that exclusively depend on them."
Phase 2: Interface Naming and Architectural Review (LLM Native Task)
Goal: Now that the math has proven which methods belong together based on usage, use the LLM to assign domain-meaningful names and validate the split.
Prompt:
"You are an Interface Segregation Architect. I am providing you with interface_clusters.json, containing mathematically grouped methods from a legacy God Class, alongside the raw source code of the class.
Perform the following operations:
1. Role Naming: For each cluster, analyze the method signatures and their parameter types. Assign a domain-specific interface name that describes the single responsibility this subset serves (e.g., IPaymentCharger, IInvoiceGenerator, IRefundProcessor).
2. Orphan Detection: Identify any public methods that appear in no cluster or appear in multiple clusters. These are either dead code (candidates for removal) or shared utilities that belong in a separate, common interface.

Output the named interfaces with their method signatures, the orphan report, and the caller migration plan."


* Adapter Generation: Agents automatically design domain-specific interfaces and concrete wrapper classes to encapsulate direct vendor API calls.


Adapter Generation is the mechanical complement to the Outbound Quarantine described in the API/Framework Quarantine section. Where Quarantine established the architectural principle (separate your domain from vendor APIs), Adapter Generation provides the fully automated detection-to-generation pipeline. The goal is not to ask the LLM "please wrap this SDK." The goal is to mechanically scan the entire repository, identify every file infected with direct vendor API calls, calculate the minimal domain-specific interface required, and generate both the interface and the concrete wrapper without human enumeration.
In legacy codebases, vendor coupling is rarely concentrated. A single SDK (like aws-sdk, stripe, or twilio) is typically imported in 15-40 files across controllers, services, background jobs, and utilities. Manual adapter creation misses outliers. The agent must perform an exhaustive, graph-backed sweep.

Do not let the LLM guess which vendor methods are used. Enumerate them deterministically.

* TypeScript Compiler API / javaparser / Python ast module: Language-specific tools that resolve the exact vendor types, method signatures, and return types being consumed. Tree-sitter extracts syntax; compiler APIs resolve semantics (e.g., determining that client.send() returns a Promise<SendMessageResult>).
* Frontier LLMs: Required for the semantic translation step. Converting vendor terminology (PutItemCommand, BatchWriteItemInput) into domain terminology (saveSession, bulkPersistOrders) is a reasoning task.

The agent must orchestrate a pipeline that maps every vendor touchpoint, calculates the minimal required interface, and generates the adapter files.
1. The Infection Scanner: A script that queries repository_graph.json for all [IMPORTS] edges targeting the vendor package namespace. Output: a list of every application file that directly imports the vendor SDK, and the specific vendor symbols (classes, functions, constants) imported.
2. The Usage Enumerator: A script using Tree-sitter that parses each infected file and extracts the exact vendor method calls, the arguments passed (with types), and the return values consumed. This produces the ground truth of "what our application actually uses from this vendor."
3. The Minimal Interface Calculator: A script that deduplicates the Usage Enumerator output. If 12 files all call client.putItem(), that is one interface method, not twelve. The output is the smallest possible interface that satisfies all current application usage.
4. The Adapter Code Generator: A script that takes the minimal interface and generates two files: the domain-specific interface (port) and the concrete wrapper class (adapter) that implements it by delegating to the real vendor SDK.
5. The Rewiring Script: A script that rewrites each infected file's imports and constructor injections to reference the new domain interface instead of the vendor SDK directly.

* Sprout/Wrap Mechanical Automation: Agents execute the four emergency legacy patterns (Sprout Method, Sprout Class, Wrap Method, Wrap Class) as deterministic AST operations, selecting the correct pattern based on the structural context and applying it without modifying existing legacy logic.


Sprout/Wrap Mechanical Automation is the implementation of Chapter 6 ("I Don't Have Much Time and I Have to Change It") as an agent-executable pipeline. The four patterns are not creative design decisions. They are finite, structural operations with strict selection criteria:
* Sprout Method: The new logic can be expressed as a standalone function. The legacy method can accept a single new line calling the sprout. Use when the legacy class can be instantiated in a test harness.
* Sprout Class: Same as Sprout Method, but the legacy class cannot be instantiated in a test harness. The new logic must live in a completely separate, independently testable class.
* Wrap Method: The new logic must execute strictly before or after the existing logic, without interleaving. Use when the legacy behavior must not be modified at all.
* Wrap Class: Same as Wrap Method, but applied at the class level using the Decorator pattern. Use when multiple methods of the legacy class need the same before/after behavior (e.g., logging, authorization).
If you ask an LLM to "add this feature to the legacy method," it will inline the new code directly into the existing method. That is "Edit and Pray." The agent must select the correct pattern and apply it mechanically.
Here is how to architect an agent-driven Sprout/Wrap Automation pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM choose how to integrate new code. Use structural analysis to select the correct pattern deterministically.
* Tree-sitter / ts-morph / libcst: For AST-level manipulation. Sprout Method requires inserting a single method call into the legacy method. Wrap Method requires renaming the legacy method and creating a new method with the original name. These are mechanical AST operations.
* The Repository Graph (Built Previously): To determine whether the legacy class can be instantiated in a test harness (decides Sprout Method vs. Sprout Class). If the class has constructor dependencies on infrastructure (from the hidden dependency analysis), Sprout Class is mandatory.
* The Variable Lifecycle Map (Built Previously): To determine the insertion point for the Sprout Method call. The sprout call must be placed at a point where the required input variables are live and the output variable has not yet been consumed.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that selects the correct pattern, generates the new code container, and wires it into the legacy code with minimal insertion.
1. The Pattern Selector: A script that evaluates the structural context and selects the correct pattern:
    * Can the legacy class be instantiated in a test? (Check constructor dependencies against repository graph.) YES -> Sprout Method candidate. NO -> Sprout Class candidate.
    * Does the new logic need to interleave with existing logic, or execute strictly before/after? INTERLEAVE -> Sprout. BEFORE/AFTER -> Wrap.
    * Is the before/after behavior needed on one method or multiple? ONE -> Wrap Method. MULTIPLE -> Wrap Class.
2. The Sprout Generator: A script that creates a new method (or new class file) containing only the new logic. It generates the test file for the new code using TDD scaffolding. It determines the exact parameter list by analyzing which variables from the legacy method are needed by the new logic (using variable lifecycle data).
3. The Insertion Surgeon: A script using AST manipulation that inserts the minimal wiring into the legacy code:
    * For Sprout Method: Inserts a single method call at the determined insertion point.
    * For Sprout Class: Inserts an instantiation of the new class and a single method call.
    * For Wrap Method: Renames the legacy method (e.g., to _legacyProcess), creates a new method with the original name that calls the new logic then _legacyProcess (or vice versa).
    * For Wrap Class: Creates a Decorator class implementing the same interface, delegating all methods to the wrapped instance, adding new logic to the targeted methods.
4. The Call Site Verifier: A script that checks all call sites of the modified method/class in the repository graph. For Sprout and Wrap Method, existing callers should be unaffected (the public API is unchanged). For Wrap Class, callers that instantiate the legacy class directly may need updating if the Decorator must be injected.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Legacy File, Class, and Method.
* New Feature Description: A structured spec of the new behavior to add (input data, output data, business rule).
* Insertion Constraint: BEFORE, AFTER, or INLINE relative to the existing logic.
* repository_graph.json and variable_lifecycle_map.json.
Outputs from the Pipeline:
* The new Sprout method/class or Wrap method/class file, fully testable.
* A test file for the new code (generated via TDD scaffolding).
* The modified legacy file with the minimal insertion (one line for Sprout, rename + new method for Wrap).
* A pattern_selection_rationale.json explaining why the selected pattern was chosen over alternatives.
4. The Prompts
Divide the operation into structural analysis for pattern selection and mechanical code generation.
Phase 1: Mechanical Pattern Selection and Structural Analysis
Goal: Force the agent to evaluate the structural context and select the correct Sprout/Wrap pattern using deterministic criteria.
Prompt:
"You are a Legacy Pattern Selector. Your task is to determine the correct emergency integration pattern for adding new behavior to [Target Method] in [Target Class].
1. Query repository_graph.json. Analyze [Target Class]'s constructor dependencies. Can this class be instantiated in a test harness without triggering infrastructure (database, network, file system)? Record YES or NO.
2. Analyze the insertion constraint. Does the new behavior need to execute strictly BEFORE or AFTER the existing logic, or does it need access to intermediate state computed INSIDE the existing method?
3. If BEFORE/AFTER and the change applies to a single method: select WRAP_METHOD. If BEFORE/AFTER and the change applies to multiple methods: select WRAP_CLASS. If INLINE and the class is testable: select SPROUT_METHOD. If INLINE and the class is NOT testable: select SPROUT_CLASS.
4. If SPROUT_METHOD is selected, query variable_lifecycle_map.json to determine the optimal insertion line: the point where all required input variables are live and no output variable has been consumed yet.
Output pattern_selection.json containing the selected pattern, the rationale, and the insertion point (if applicable)."
Phase 2: Code Generation and Test Scaffolding (LLM Native Task)
Goal: Now that the pattern is selected and the insertion point is determined, use the LLM to generate the new code container and its test, then mechanically wire it into the legacy code.
Prompt:
"You are a Sprout/Wrap Code Generator. I am providing you with pattern_selection.json and the new feature specification.
The selected pattern is [PATTERN_NAME]. Execute the following:
1. Generate the New Code Container:
   * For SPROUT_METHOD: Write the new method with explicit parameters (from the variable lifecycle analysis) and a clear return type. Place it as a public method on the legacy class.
   * For SPROUT_CLASS: Write a new class file containing the new logic. The class must have zero dependencies on the legacy class. All required data must be passed via constructor or method parameters.
   * For WRAP_METHOD: Rename the legacy method to _legacy[OriginalName]. Create a new method with the original name that calls the new logic and delegates to _legacy[OriginalName].

Output the new code file, the test file, and the insertion diff."


* Programming by Difference / Subclass-and-Override Automation: Agents use inheritance to inject new behavior into legacy classes by generating subclasses that override specific methods, updating instantiation sites through factories, and leaving the original class completely untouched.


Programming by Difference is the mechanical implementation of the subclassing strategy from Chapter 8 ("How Do I Add a Feature?"). When a legacy class is too dangerous, too large, or too coupled to modify directly—but you need to change the behavior of a specific method—you subclass it and override only that method. The original class remains byte-for-byte identical. The risk of regression is near zero because no production code is edited.
The technique has a strict prerequisite: the method to override must be virtual/overridable (not final, not static, not private). And the instantiation sites must be controllable—either through a factory, a DI container, or a limited number of new ClassName() calls that can be redirected to the subclass.
If you ask an LLM to "change the behavior of this method," it will edit the method in place. The agent must instead generate a subclass, override the method, and redirect instantiation.
Here is how to architect an agent-driven Programming by Difference pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM modify the original class. Use AST analysis to verify overridability and mechanical code generation to produce the subclass.
* Tree-sitter / ts-morph / JavaParser: For analyzing the target method's access modifier (public, protected, virtual, final, static). The method must be overridable. If it is private or final, the agent must first apply "Subclass and Override Method" from the dependency-breaking catalog (making it protected).
* The Repository Graph (Built Previously): To locate every instantiation site of the target class (every new ClassName() or DI registration). These are the points that must be redirected to the subclass.
* Language-Specific Inheritance Analyzers: To verify that the target class is not already deeply nested in an inheritance hierarchy that would make further subclassing fragile (the Fragile Base Class problem).
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that verifies overridability, generates the subclass, and redirects instantiation.
1. The Override Feasibility Checker: A script that parses the target method and verifies: Is the method non-final, non-static, non-private? Is the class non-final/non-sealed? Does the method access private fields that would be inaccessible in the subclass? If any check fails, the script reports the specific blocker and recommends the prerequisite refactoring (e.g., change private to protected).
2. The Subclass Generator: A script that generates a new class file extending the target class. It overrides only the target method with the new behavior. All other methods are inherited unchanged. The constructor mirrors the parent's constructor signature exactly.
3. The Instantiation Redirector: A script that queries the repository graph for all [INSTANTIATES] edges targeting the parent class. It classifies each instantiation site: FACTORY (a factory method creates the object—redirect the factory), DI_CONTAINER (a DI binding creates the object—update the binding), DIRECT (a direct new call—must be updated at each call site), REFLECTION (created via reflection—cannot be mechanically redirected, flagged for manual review).
4. The Factory Extractor: If no factory exists and there are many direct instantiation sites, this script generates a factory method (Extract Factory from Chapter 25) and updates all call sites to use the factory. The factory then becomes the single point to redirect.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Class and Method to Override.
* New Behavior Specification: The replacement logic for the overridden method.
* repository_graph.json.
Outputs from the Pipeline:
* [ClassName]Extended.ts (or appropriate naming) containing the subclass with the overridden method.
* A test file for the subclass exercising the new behavior.
* Modified factory, DI binding, or direct instantiation sites redirecting to the subclass.
* An override_feasibility.json reporting the overridability analysis and any prerequisite refactoring required.
4. The Prompts
Divide the operation into mechanical feasibility analysis and subclass generation.

5. Mechanical Enforcement and Risk Mitigation
* Strict Pattern Execution: Agents execute finite, mechanical operations from the dependency-breaking catalog (e.g., Extract and Override Factory Method) via direct AST manipulation, severing hard coupling with near-zero hallucination risk.


Strict Pattern Execution is the mechanical implementation of Chapter 25's dependency-breaking catalog. Feathers defines 25 named, finite refactoring techniques (Extract and Override Call, Extract and Override Factory Method, Parameterize Constructor, Replace Global Reference with Getter, Subclass and Override Method, and others). Each technique is a deterministic sequence of AST transformations. They do not require semantic reasoning. They require mechanical precision.
If you ask an LLM to "break the dependency in this constructor," it will improvise. It will combine techniques, skip steps, introduce novel abstractions, or refactor beyond the requested scope. That improvisation is where regressions are born.
Instead, the AI agent must act as a pattern executor. The human (or a higher-level planning agent) selects the named technique. The execution agent applies it mechanically via AST manipulation, the same way an IDE's "Extract Method" command operates: deterministic, scope-limited, compiler-verifiable.
Here is how to architect an agent-driven Strict Pattern Execution pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM freestyle refactoring. Use AST manipulation libraries to guarantee structural correctness.
* Tree-sitter (with edit capabilities): Tree-sitter parses into a Concrete Syntax Tree (CST), preserving whitespace and comments. This is critical: refactoring must not destroy formatting, or the resulting diff becomes unreadable and unreviewable.
* Language-Specific Refactoring APIs: * TypeScript: ts-morph (a programmatic wrapper over the TypeScript Compiler API). It provides first-class methods for addParameter, extractInterface, renameSymbol, and moveDeclaration.
    * Java: JavaParser or Eclipse JDT. Provides AST-level manipulation with type resolution.
    * Python: libcst (Concrete Syntax Tree library by Instagram). Unlike Python's built-in ast module, libcst preserves formatting and comments during transformation.
    * C/C++: libclang or Clang's LibTooling with AST matchers and rewriters.
* The Technique Catalog: A machine-readable registry of the 25 named techniques. Each entry defines: Preconditions (what structural pattern must be present), Steps (the exact AST operations in order), and Postconditions (what structural pattern must be true after execution).
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that validates preconditions, executes the named steps, and verifies postconditions.
1. The Technique Registry: A JSON or YAML file defining each named dependency-breaking technique as a sequence of mechanical AST operations. Example entry for "Parameterize Constructor":
    * Precondition: Constructor body contains a new [ClassName]() expression for the target dependency.
    * Step 1: Extract the type of the instantiated dependency.
    * Step 2: Add a new parameter of that type (or its extracted interface) to the constructor signature.
    * Step 3: Replace the internal new expression with an assignment from the new parameter.
    * Step 4: At every existing call site, insert the original new [ClassName]() expression as the argument.
    * Postcondition: Constructor no longer contains a new expression for the target dependency. All existing call sites compile and pass the original concrete instance.
2. The Precondition Validator: A script that parses the target code using Tree-sitter or ts-morph and checks whether the structural preconditions for the selected technique are met. If not, it rejects execution with a diagnostic explaining why.
3. The Step Executor: A script that applies each AST transformation in strict sequence. After each step, it re-parses the modified file to validate the AST remains structurally valid.
4. The Postcondition Verifier: A script that re-parses the final result and checks that the postconditions hold. It then runs the compiler (tsc, javac, gcc) to prove the transformation did not break the build.
5. The Call Site Updater: A script that queries the repository_graph.json for all [CALLS] or [INSTANTIATES] edges targeting the modified constructor or method. It applies the corresponding argument additions or import changes at every call site.
3. Inputs and Outputs
Inputs to the Pipeline:
* Technique Name: The exact named technique from the catalog (e.g., "Extract and Override Factory Method", "Parameterize Constructor", "Replace Global Reference with Getter").
* Target File, Class, and Method/Constructor.
* Target Dependency: The specific symbol or expression to be severed (e.g., "the new DatabaseClient() call on line 45").
Outputs from the Pipeline:
* Modified source files with the technique applied.
* A technique_execution_log.json recording every AST operation performed, every call site updated, and the compiler verification result.
* A verifiable git diff restricted exclusively to the mechanical steps of the named technique.
4. The Prompts
Divide the operation into mechanical execution and deviation detection.
Phase 1: Mechanical Pattern Application
Goal: Force the agent to execute the named technique as a finite state machine, not as a creative refactoring exercise.
Prompt:
"You are a Dependency-Breaking Pattern Executor. Your task is to apply the named technique '[Technique Name]' from the legacy code dependency-breaking catalog to the target code.
You are NOT permitted to improvise, combine techniques, or refactor beyond the scope of this single named pattern.
1. Read the technique definition from the catalog. Verify that the structural preconditions are met in the target code using AST analysis.
2. If preconditions are not met, STOP. Output a diagnostic explaining which precondition failed and which alternative technique might apply.
3. If preconditions are met, write a script using [ts-morph / libcst / JavaParser] to execute each step of the technique in strict order.
4. After all steps, verify postconditions by re-parsing the AST and running the compiler.
5. Query repository_graph.json for all call sites of the modified symbol. Apply the required call site updates.
6. Run the compiler again to prove all call sites are valid.
Output the modified files, the technique_execution_log.json, and the compiler verification result."
Phase 2: Deviation Detection and Scope Guard (LLM Native Task)
Goal: After the mechanical execution, use the LLM as a safety reviewer to verify that the resulting diff contains only the expected changes and no accidental behavioral modifications.
Prompt:
"You are a Refactoring Scope Auditor. I am providing you with the git diff produced by applying the dependency-breaking technique '[Technique Name]' and the technique_execution_log.json.
The named technique guarantees behavior preservation if applied correctly.
Perform the following operations:
1. Scope Verification: Verify that every line changed in the diff corresponds to an expected step in the technique definition. Flag any line change that is not explained by the technique log.
2. Behavioral Drift Detection: Check that no conditional logic (if/else, switch, ternary) was added, removed, or reordered. The technique must only restructure wiring, not logic.
3. Signature Audit: Verify that every modified method or constructor signature change is matched by a corresponding call site update. Flag any call site that was missed.
4. Import Hygiene: Verify that no new external dependencies were introduced and no existing imports were removed unless the technique explicitly requires it.
Output a scope_audit_report.md classifying the diff as CLEAN (all changes explained by the technique) or CONTAMINATED (unexpected changes detected, with line-by-line annotations)."



* Pre-Commit Diff Analysis: Agents function as hyper-aware pair programmers, analyzing git diffs against legacy files to detect unintended side effects, variable scope overlaps, or behavioral alterations during structural refactoring.


Pre-Commit Diff Analysis is the final gatekeeper in the legacy code modification pipeline. Every technique described in the preceding sections (Sprout, Wrap, Extract, Parameterize) is designed to preserve existing behavior. But developers and agents make mistakes. A renamed variable shadows an outer scope. An extracted method forgets to return a mutated value. A moved block loses a side effect that was load-bearing. These errors are invisible in a manual code review of a 200-line diff against a 3,000-line file.
The AI agent must not review diffs the way a human does (reading top-to-bottom and pattern-matching for "looks wrong"). It must perform a structural diff—comparing the Abstract Syntax Trees before and after the change—and a behavioral diff—comparing the data flow and control flow graphs before and after.
Here is how to architect an agent-driven Pre-Commit Diff Analysis pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM read a textual diff and guess whether behavior changed. Use tools that mathematically compare program structure.
* Git: For extracting the exact pre-change and post-change file contents (git show HEAD:[file] for the base version, working tree for the modified version).
* Tree-sitter: For generating the AST of both the pre-change and post-change versions of each modified file. By comparing ASTs instead of text, the agent ignores formatting noise (whitespace, comment changes, import reordering) and focuses on structural mutations.
* GumTree: An open-source AST differencing tool. It computes the minimum edit script between two ASTs, classifying each change as INSERT, DELETE, UPDATE, or MOVE. This is dramatically more precise than textual diff, which conflates moved code with deleted-and-added code.
* Joern / Code Property Graphs: For generating Control Flow Graphs (CFG) and Program Dependence Graphs (PDG) of modified functions. If the CFG changes, control flow changed. If the PDG changes, data dependencies changed. Either change in a "behavior-preserving" refactoring is a defect.
* The Variable Lifecycle Map (Built Previously): The variable_lifecycle_map.json from the Variable Scope Mapping pipeline. Cross-referencing the diff against variable liveness ranges detects shadowing and scope corruption.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that extracts both versions, compares their structures mathematically, and flags discrepancies.
1. The Snapshot Extractor: A script that uses git show to retrieve the pre-change version of every modified file. It stores both the pre and post versions in a temporary directory for parallel AST parsing.
2. The AST Differ: A script that parses both versions of each modified file using Tree-sitter, then uses GumTree (or a custom AST comparison algorithm) to produce a structured edit script. The edit script classifies every change as: NODE_INSERTED, NODE_DELETED, NODE_UPDATED (value changed), NODE_MOVED (same node, different location in tree).
3. The Control Flow Comparator: For every function that appears in the diff, this script generates the Control Flow Graph (CFG) of both the pre and post versions. It compares the graphs. If the number of branches, the branch conditions, or the branch targets changed in a refactoring that claims to be behavior-preserving, it flags a violation.
4. The Scope Collision Detector: A script that checks every newly introduced variable declaration against the variable liveness ranges of the enclosing scope. If a new variable name matches an existing variable that is still live in an outer scope, it flags a shadowing hazard.
5. The Side Effect Tracker: A script that extracts all statements with side effects (assignments to external state, function calls with void returns, I/O operations) from both the pre and post versions of each modified function. It compares the ordered lists. If a side effect was removed, reordered, or duplicated, it flags the change.
3. Inputs and Outputs
Inputs to the Pipeline:
* The git diff (staged changes, or a specific commit range).
* The repository_graph.json (for cross-file impact analysis).
* The change_type declaration: One of REFACTORING (behavior must be identical), FEATURE (new behavior allowed, existing must be preserved), or BUGFIX (specific behavior change expected, all other behavior preserved).
Outputs from the Pipeline:
* A pre_commit_analysis.json file.
* Schema: * structural_changes: The AST edit script (inserts, deletes, updates, moves).
    * control_flow_delta: Any branches added, removed, or reordered per function.
    * scope_hazards: Any variable shadowing or liveness violations.
    * side_effect_delta: Any side effects added, removed, reordered, or duplicated.
    * cross_file_impact: Any callers of modified functions that may need updating (from the repository graph).
    * verdict: CLEAN, WARNING (non-critical deviations), or REJECT (behavioral change detected in a declared refactoring).
4. The Prompts
Divide the operation into mechanical structural comparison and semantic intent verification.
Phase 1: Mechanical Structural Comparison
Goal: Force the agent to mathematically compare the pre-change and post-change code structures, producing an irrefutable structural delta.
Prompt:
"You are a Structural Diff Engine. Your task is to analyze the staged git diff and produce a structural change report.
1. Write a Python script to extract the pre-change and post-change versions of every modified file using git show and the working tree.
2. Parse both versions of each modified file using Tree-sitter. Generate the AST for each version.
3. For every function that appears in the diff, compare the pre and post ASTs. Classify each AST node change as INSERT, DELETE, UPDATE, or MOVE. Ignore whitespace-only and comment-only changes.
4. For every modified function, extract the ordered list of side-effecting statements (assignments to variables declared outside the function, I/O calls, mutation of passed-by-reference parameters). Compare the pre and post lists for additions, removals, or reorderings.
5. For every newly declared variable in the post version, check whether its name collides with any variable that is still live in an enclosing scope.
Output pre_commit_structural_delta.json containing the AST edit scripts, side effect deltas, and scope collision flags per modified function."
Phase 2: Behavioral Intent Verification (LLM Native Task)
Goal: Now that the math has produced the structural delta, use the LLM to cross-reference the delta against the declared intent of the change, catching semantic violations that pure structure cannot detect.
Prompt:
"You are a Behavioral Preservation Auditor. I am providing you with pre_commit_structural_delta.json and the declared change_type for this commit.
The developer has declared this change as a [REFACTORING / FEATURE / BUGFIX].
Perform the following operations:
1. Behavioral Contract Check: If the change is declared as REFACTORING, verify that zero control flow branches were added or removed. Any structural INSERT or DELETE of a conditional node (if, switch, ternary, loop guard) is a violation. Flag it with the exact file, function, and line.
2. Side Effect Audit: If any side-effecting statement was reordered, verify whether the reordering could produce a different observable outcome (e.g., writing to a log after a database call instead of before). Flag any reordering where the side effects are not commutative.

* Build Dependency Severing: Agents measure the recompilation cascade of every file in the repository—how many downstream files must be rebuilt when a single file changes—and propose interface extractions to sever the physical import chains causing the longest build times.


Build Dependency Severing is the mechanical implementation of Chapter 7 ("It Takes Forever to Make a Change"). Feathers distinguishes between physical dependencies (file A imports file B's source, so modifying B forces A to recompile) and logical dependencies (class A depends on class B's behavior, but the import chain does not create a compilation cascade). Physical dependencies are the silent killers of CI/CD velocity. A single heavily-imported utility file or base class can create a recompilation cascade that touches hundreds of files, turning a 2-second build into a 45-minute rebuild.
If you ask an LLM to "speed up the build," it will suggest caching or parallelism. Those are band-aids. The structural fix is to sever the physical import chain by extracting interfaces into lightweight files. When a consumer imports an interface file instead of the heavy implementation file, modifying the implementation no longer triggers recompilation of the consumer.
The agent must not guess which files cause cascades. It must compute the cascade mathematically from the import graph.
Here is how to architect an agent-driven Build Dependency Severing pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM estimate build impact. Use the import graph to compute exact cascade sizes.
* The Repository Graph (Built Previously): The repository_graph.json contains all [IMPORTS] edges. The cascade size of a file is the transitive closure of all files that directly or indirectly import it.
* dependency-cruiser (JS/TS): Outputs the full dependency graph with cycle detection and reachability metrics.
* Include-What-You-Use (IWYU) (C/C++): Analyzes #include directives and recommends removals of unnecessary includes.
* Build System Instrumentation: Most build systems can report what was rebuilt. tsc --listFiles, make --debug=v, Webpack stats.json, Gradle --scan. These provide empirical cascade measurements.
* Tree-sitter: For analyzing what each importing file actually uses from the imported module. If file A imports module B but only uses one type definition, A can import a lightweight interface file instead.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that measures cascade sizes, identifies the worst offenders, and proposes minimal interface extractions to sever the chains.
1. The Cascade Calculator: A script that computes the transitive import closure for every file in the repository. For each file, it calculates: fan-in (how many files import it, directly or transitively), fan-out (how many files it imports), and cascade_size (total files that must be rebuilt if this file changes). Files are ranked by cascade_size descending.
2. The Usage Analyzer: For each high-cascade file, a script using Tree-sitter that analyzes what each importer actually uses from it. If 50 files import utils.ts but 40 of them only use the TypeDefinitions, those 40 can be redirected to a lightweight types-only file.
3. The Interface Extraction Planner: A script that identifies the minimum set of interfaces or type-only files that, if extracted, would sever the most cascade chains. This is a set-cover optimization: find the smallest number of interface extractions that reduce the maximum cascade size below a target threshold.
4. The Cascade Verifier: A script that simulates the import graph after the proposed extractions and recalculates the cascade sizes to prove the improvement before any code is changed.
3. Inputs and Outputs
Inputs to the Pipeline:
* Target Repository/Directory.
* repository_graph.json with [IMPORTS] edges.
* Cascade Threshold: The maximum acceptable cascade size (e.g., "no single file change should rebuild more than 20 files").
Outputs from the Pipeline:
* A build_cascade_analysis.json file.
* Schema: * cascade_ranking: Every file ranked by cascade_size, with fan_in and fan_out.
    * worst_offenders: The top N files causing the largest cascades.
    * usage_breakdown: For each worst offender, what each importer actually uses from it (types, functions, constants, classes).
    * extraction_plan: The minimum set of interface/type files to extract, with the predicted cascade reduction.
    * projected_cascade: The recalculated cascade sizes after the proposed extractions.
4. The Prompts
Divide the operation into mechanical cascade computation and extraction design.
Phase 1: Mechanical Cascade Measurement
Goal: Force the agent to compute the exact recompilation cascade for every file in the repository using graph traversal.
Prompt:
"You are a Build Cascade Analyzer. Your task is to identify the files in the repository that cause the largest recompilation cascades when modified.
1. Write a Python script to load repository_graph.json. Extract the subgraph of [IMPORTS] edges only.
2. For every file in the graph, compute the transitive closure of all files that directly or transitively import it. Record this as the cascade_size.
3. Rank all files by cascade_size descending.
4. For each of the top 10 files, use Tree-sitter to analyze what each importing file actually uses: specific types, specific functions, specific constants, or the entire module.
5. Group the importers by usage pattern. Identify importers that use only type definitions vs. importers that use runtime functions.
Output build_cascade_analysis.json containing the cascade ranking, the usage breakdown for the top offenders, and the importer groupings."
Phase 2: Interface Extraction Design (LLM Native Task)
Goal: Now that the math has identified the cascade bottlenecks and the usage patterns, use the LLM to design the minimal set of interface extractions that break the cascades.
Prompt:
"You are a Build Architecture Optimizer. I am providing you with build_cascade_analysis.json, which contains the cascade sizes and usage breakdowns for the worst-offending files in the repository.
Perform the following operations:
1. Extraction Design: For each top offender, propose the specific interface or type-only file to extract. The extracted file must contain only the types, interfaces, or abstract classes that the majority of importers use. It must NOT import the heavy implementation dependencies.
2. Migration Plan: For each importer that can be redirected to the lightweight file, specify the exact import statement change.
3. Cascade Projection: Estimate the new cascade sizes after extraction. Verify that the worst-case cascade drops below the specified threshold.
4. Incremental Execution Order: Propose the order in which the extractions should be applied. Start with the file that provides the largest cascade reduction with the smallest code change.
5. Risk Assessment: Identify any importers that use both types and runtime functions from the same heavy file. These cannot be fully severed and require a split of the heavy file into types + implementation modules.

6. Multi-Codebase Operations

* Cross-Repository Dependency Mapping: Agents build a super-graph spanning multiple repositories, connecting services through API calls, shared database tables, event topics, shared libraries, and deployment coupling to reveal the true system topology invisible from within any single codebase.


Cross-Repository Dependency Mapping extends every single-repo technique described in the preceding sections to the system level. In real legacy environments, the "system" is not one repository. It is 5, 15, or 50 repositories connected through HTTP APIs, gRPC services, message queues, shared databases, shared libraries, and deployment pipelines. No single repository's graph tells the full story.
A change to the payments service's database schema might break the reporting service's nightly batch job. A change to a shared library's interface might require synchronized releases across every consuming service. A change to an event payload might silently corrupt a downstream consumer that deserializes the old format.
The agent must construct a super-graph where nodes are services, packages, and shared resources, and edges are the inter-repository connections. Without this, Effect Analysis, Change Impact Propagation, and Migration Path Planning operate in local blindness.
Here is how to architect an agent-driven Cross-Repository Dependency Mapping pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM guess how services communicate. Use deterministic extraction from each repository plus infrastructure configuration.
* Per-Repository Graph (Built Previously): Each repository's repository_graph.json provides the internal structure. The cross-repo mapper connects these graphs at their boundary points (API endpoints, event topics, database tables).
* OpenAPI / Swagger / gRPC Proto / GraphQL Schema Files: These are machine-readable API contracts. The agent extracts the exact endpoints, methods, request/response types from each service's schema definition.
* Infrastructure-as-Code (IaC): Terraform, CloudFormation, Kubernetes manifests, Docker Compose files. These reveal: which services connect to which databases, which services share a message broker, which services are deployed together, which services share network access.
* Package Manager Lock Files: package-lock.json, Pipfile.lock, go.sum, pom.xml. These reveal shared library dependencies and version pinning across repositories.
* CI/CD Pipeline Definitions: GitHub Actions, GitLab CI, Jenkins pipelines. These reveal deployment coupling: services deployed together, services with shared build steps, services with integration test dependencies.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that extracts boundary information from each repository and stitches it into a unified system graph.
1. The Boundary Extractor (Per Repo): A script that runs against each repository and extracts all external-facing edges: HTTP endpoints exposed (from route definitions or OpenAPI specs), HTTP clients consuming external services (from HTTP client configurations, base URLs, SDK imports), event topics published (from producer code), event topics consumed (from consumer/listener code), database connection strings and table references (from ORM models, migration files, connection configs), shared library versions (from lock files).
2. The Super-Graph Assembler: A script that takes the boundary extractions from all repositories and constructs the unified system graph. Nodes: services (repositories), databases, message brokers, shared libraries. Edges: CALLS_API (service A calls service B's endpoint), SHARES_DATABASE (services A and B both read/write table X), PUBLISHES_TO / CONSUMES_FROM (event topic connections), DEPENDS_ON_LIBRARY (shared library with version), DEPLOYS_WITH (deployment coupling).
3. The Version Conflict Detector: A script that analyzes shared library dependencies across all repositories. It flags: diamond dependencies (two services depend on the same library at incompatible versions), stale dependencies (a service depends on a library version that is 3+ major versions behind the latest consumer), and unversioned shared code (local path references or git submodules without version tags).
4. The Shared Database Mapper: A script that identifies tables accessed by multiple services. For each shared table, it records: which services READ, which WRITE, whether they use the same ORM model or divergent models, and whether any service has exclusive ownership.
3. Inputs and Outputs
Inputs to the Pipeline:
* Repository List: The set of repositories comprising the system, with their local paths or clone URLs.
* Infrastructure Config Paths: Paths to IaC files, Docker Compose, Kubernetes manifests.
Outputs from the Pipeline:
* A system_graph.json file.
* Schema: * services: Array of service nodes with metadata (repository path, language, framework, deployment target).
    * databases: Array of database nodes with connection info and table lists.
    * brokers: Array of message broker nodes with topic lists.
    * shared_libraries: Array of shared dependency nodes with version-per-service.
    * edges: All inter-service edges typed as CALLS_API, SHARES_DATABASE, PUBLISHES_TO, CONSUMES_FROM, DEPENDS_ON_LIBRARY, DEPLOYS_WITH.
    * version_conflicts: Flagged library version mismatches.
    * shared_tables: Tables accessed by multiple services with READ/WRITE classification.
4. The Prompts
Divide the operation into mechanical boundary extraction and architectural topology analysis.
Phase 1: Mechanical Boundary Extraction and Graph Assembly
Goal: Force the agent to extract inter-service edges from each repository using deterministic tools and assemble the system-level graph.
Prompt:
"You are a System Topology Engine. Your task is to construct a cross-repository dependency graph for a multi-service system.
For each repository in the provided list:
1. Extract all exposed API endpoints from route definitions, OpenAPI specs, or gRPC proto files. Record method, path, request/response types.
2. Extract all outbound API calls from HTTP client configurations, SDK imports, and base URL references. Record the target service and endpoint.
3. Extract all event topic publications and consumptions from message broker client code.
4. Extract all database table references from ORM models, migration files, and raw SQL.
5. Extract shared library dependencies and versions from package lock files.

6. Construct the system_graph.json by linking service boundaries: match outbound API calls to exposed endpoints, match event publishers to consumers via topic strings, identify shared database tables across services, and flag library version conflicts.

Phase 2: System Topology Analysis (LLM Native Task)
Goal: Now that the graph exists, use the LLM to identify architectural risks, coupling hotspots, and ownership gaps at the system level.
Prompt:
"You are a System Architecture Analyst. I am providing you with system_graph.json, which maps the complete inter-service dependency topology.
Perform the following operations:
1. Coupling Hotspot Identification: Identify services with the highest number of incoming edges (most depended-upon). These are system-level God Services. Rank by total inbound edge count.
2. Shared Database Risk Assessment: For each table accessed by multiple services, assess the coordination risk. If multiple services WRITE to the same table, flag the data integrity hazard. If services use divergent ORM models for the same table, flag the schema drift risk.

* API Contract Drift Detection: Agents extract the actual API schema from the provider codebase and the actual consumption patterns from each consumer codebase, detecting field-level mismatches, type coercions, and undocumented assumptions that will break at runtime.


API Contract Drift Detection is the cross-repository extension of the Adapter Generation and Effect Analysis pipelines. Within a single codebase, the agent can verify that a method caller matches the method signature. Across repositories, this verification does not exist. Service A exposes an endpoint returning { orderId, total, status }. Service B consumes it, but only reads orderId and total. Six months later, Service A renames total to totalAmount. Service B's code compiles fine (it has no compile-time link to Service A). The failure only manifests at runtime in production.
The agent must perform cross-repository contract verification by extracting the actual schema from the provider and the actual consumption pattern from each consumer, then comparing them field by field.
Here is how to architect an agent-driven API Contract Drift Detection pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM compare API docs. Extract the actual code-level contract from both sides.
* OpenAPI / Swagger / gRPC Proto / GraphQL Schema: If the provider maintains a machine-readable schema, this is the authoritative contract. But in legacy systems, schemas are often outdated or missing.
* Tree-sitter / AST Scanners: For extracting the actual response structure from the provider's route handler code (what fields does it actually serialize?), and the actual deserialization pattern from the consumer's client code (what fields does it actually read?).
* JSON Schema Validators: For formalizing the extracted schemas into a comparable format.
* The System Graph (Built Previously): To identify which services consume which endpoints, providing the complete list of provider-consumer pairs to verify.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that extracts both sides of every API contract and performs field-level comparison.
1. The Provider Schema Extractor: A script that, for each API endpoint in the provider service, extracts the actual response payload structure. Source priority: (1) OpenAPI/proto/GraphQL schema if present and recently updated, (2) the return statement of the route handler, tracing through serialization code to determine the exact fields and types, (3) integration test fixtures or snapshot files that contain example responses.
2. The Consumer Usage Extractor: A script that, for each consumer service that calls a provider endpoint, extracts the actual fields accessed from the response. This requires parsing the HTTP client call, tracing the response variable to wherever it is destructured or accessed (e.g., response.data.total, order['status']).
3. The Contract Comparator: A script that performs field-level comparison between the provider's actual output and each consumer's actual usage. It classifies each field as: MATCHED (provider sends it, consumer reads it, types align), UNUSED (provider sends it, consumer ignores it—safe), MISSING (consumer reads it, provider does not send it—runtime error), TYPE_MISMATCH (both sides reference the field but expect different types), RENAMED (field exists under a different name—possible recent rename).
4. The Drift Tracker: A script that compares the current contract analysis against a previous snapshot (if available) to detect fields that were recently added, removed, or changed. This provides the temporal dimension: "this field was present in last month's analysis but is now missing."
3. Inputs and Outputs
Inputs to the Pipeline:
* system_graph.json (to identify all API edges).
* Provider repository path and consumer repository paths.
* Previous contract snapshot (if available, for drift detection).
Outputs from the Pipeline:
* A contract_drift_report.json file.
* Schema: * contracts: For each provider endpoint, the extracted schema (fields and types).
    * consumers: For each consumer of each endpoint, the fields actually accessed and the types expected.
    * mismatches: Array of MISSING, TYPE_MISMATCH, and RENAMED field issues with exact file/line references in both provider and consumer code.
    * drift_deltas: Fields that changed since the last snapshot.
    * undocumented_assumptions: Consumer code that accesses fields not present in the official schema (relying on undocumented behavior).
4. The Prompts
Divide the operation into mechanical schema extraction and semantic compatibility analysis.
Phase 1: Mechanical Schema Extraction and Comparison
Goal: Force the agent to extract the actual API contract from code on both sides and perform field-level comparison.
Prompt:
"You are a Contract Verification Engine. Your task is to detect API contract drift between provider and consumer services identified in system_graph.json.
For each API edge in the system graph:
1. Provider Side: Parse the provider's route handler for the endpoint. Trace the response object through serialization. Extract the exact field names, types, and nesting structure of the actual response payload. If an OpenAPI/proto schema exists, extract it as a secondary reference.
2. Consumer Side: Parse the consumer's HTTP client code that calls this endpoint. Trace the response variable to every point where fields are accessed. Record the exact field names and the types the consumer expects.
3. Compare: For each field the consumer accesses, verify it exists in the provider's output with a compatible type. Classify every field as MATCHED, UNUSED, MISSING, TYPE_MISMATCH, or RENAMED.
Output contract_drift_report.json containing the extracted schemas, consumer usage patterns, and all mismatches."
Phase 2: Semantic Drift Impact Assessment (LLM Native Task)
Goal: Now that the math has identified the mismatches, use the LLM to assess the runtime impact and prioritize fixes.
Prompt:
"You are a Contract Compatibility Analyst. I am providing you with contract_drift_report.json, which contains field-level mismatches between API providers and consumers across the system.
Perform the following operations:
1. Severity Classification: For each MISSING field, determine the runtime impact. Does the consumer use it in a conditional (will cause incorrect branching)? In a display (will show undefined/null)? In a calculation (will produce NaN or incorrect results)? In a database write (will persist corrupt data)?
2. Silent Failure Detection: Identify consumers that destructure the response with default values (e.g., const { total = 0 } = response). These will silently produce incorrect behavior rather than throwing an error when the field is missing.
3. Backward Compatibility Assessment: For fields classified as RENAMED, determine whether the provider is sending both the old and new field names (backward-compatible) or only the new name (breaking change).
4. Contract Test Generation: For each MISSING or TYPE_MISMATCH field, generate the exact contract test that should exist in the consumer's test suite to catch this at CI time rather than runtime.
Output a contract_drift_remediation.md containing the severity-ranked mismatches, silent failure risks, and the generated contract tests."


* Database Schema Archaeology: Agents map which codebases read and write which database tables, detect shared-table coupling across services, identify ownership boundaries, and flag schema migration risks where multiple services depend on the same table structure.


Database Schema Archaeology addresses the most dangerous shared state in any multi-service legacy system: the database. In properly architected systems, each service owns its tables exclusively. In legacy systems, three, five, or ten services read and write to the same tables. The database becomes a hidden coupling mechanism that defeats every API boundary, every event bus, and every service separation effort.
A schema migration in service A that adds a NOT NULL column will crash service B's inserts. A service C that directly reads service A's tables bypasses all business rules, invariants, and access control that service A enforces in its application layer. An ORM model in service D that maps to the same table but with different column names or types will silently produce corrupt reads.
The agent must map every table to every service that touches it, classify the access pattern, and identify the ownership boundaries and conflicts.
Here is how to architect an agent-driven Database Schema Archaeology pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM guess which services use which tables. Extract the evidence from code and configuration.
* ORM Model Scanners: Tree-sitter or framework-specific parsers to extract model definitions (Django models, SQLAlchemy models, TypeORM entities, Hibernate entities, ActiveRecord models). Each model maps to a table with specific column definitions.
* Migration File Parsers: Scripts that parse migration files (Alembic, Flyway, Liquibase, Knex, Rails migrations) to extract the full DDL history: tables created, columns added/removed/altered, constraints modified, indexes added.
* Raw SQL Extractors: Ripgrep and Tree-sitter to find raw SQL strings in application code (SELECT, INSERT, UPDATE, DELETE, JOIN statements) and extract the table names referenced.
* Database Connection Config Scanners: Scripts that extract connection strings, database names, and schema names from each service's configuration to determine which services share a physical database.
* The System Graph (Built Previously): To overlay database edges onto the existing inter-service topology.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that extracts table references from every repository and constructs a database ownership and access map.
1. The Table Reference Harvester (Per Repo): A script that scans each repository and extracts every reference to a database table: ORM model definitions (table name, columns, types, constraints), migration operations (CREATE TABLE, ALTER TABLE, DROP TABLE, ADD COLUMN), raw SQL statements (table names extracted via SQL parsing), stored procedure references (if applicable).
2. The Access Pattern Classifier: A script that classifies each service's relationship to each table as: OWNER (has migration files for this table, defines the canonical ORM model), READER (only SELECT queries, no migrations), WRITER (INSERT/UPDATE/DELETE queries, no migrations), FULL_ACCESS (both reads and writes, no migrations—the most dangerous pattern), MIGRATION_OWNER (has migrations but different ORM model from another service—ownership conflict).
3. The Schema Divergence Detector: A script that compares ORM model definitions for the same table across different services. It detects: column name mismatches (service A calls it "total", service B calls it "amount"), type mismatches (service A maps it as integer, service B maps it as string), missing columns (service A's model includes a column that service B's model does not), constraint mismatches (service A expects NOT NULL, service B allows NULL).
4. The Migration Risk Analyzer: A script that takes a proposed migration (add column, alter type, drop column) and cross-references it against all services' access patterns. It reports which services will break and why.
3. Inputs and Outputs
Inputs to the Pipeline:
* Repository List with their local paths.
* Database connection configurations (to determine which services share a physical database).
Outputs from the Pipeline:
* A database_archaeology.json file.
* Schema: * tables: Array of every discovered table with its full column schema.
    * service_access: For each table, which services access it and with what classification (OWNER, READER, WRITER, FULL_ACCESS).
    * ownership_conflicts: Tables where multiple services have migration files or conflicting ORM models.
    * schema_divergences: Field-level mismatches between ORM models in different services for the same table.
    * shared_write_hazards: Tables where multiple services perform WRITE operations (data integrity risk without coordination).
    * migration_impact_matrix: For each table, the list of services that would be affected by a schema change.
4. The Prompts
Divide the operation into mechanical table-service mapping and semantic ownership analysis.
Phase 1: Mechanical Table Reference Extraction and Classification
Goal: Force the agent to exhaustively scan every repository for every database table reference and classify the access pattern.
Prompt:
"You are a Database Topology Engine. Your task is to map every database table to every service that accesses it across the multi-service system.
For each repository in the provided list:
1. Parse ORM model definitions. Extract table names, column names, column types, and constraints.
2. Parse migration files. Extract the DDL operations and the tables they modify.
3. Scan for raw SQL strings. Extract table names from SELECT, INSERT, UPDATE, DELETE, and JOIN statements.
4. Classify each service's relationship to each table as OWNER (has migrations), READER (SELECT only), WRITER (INSERT/UPDATE/DELETE), or FULL_ACCESS (reads and writes without owning migrations).
After processing all repositories:
5. Compare ORM model definitions across services for the same table. Flag column name mismatches, type mismatches, and missing columns.
6. Identify tables accessed by multiple services with WRITE permissions.
Output database_archaeology.json containing the complete table-to-service mapping, access classifications, ownership conflicts, and schema divergences."
Phase 2: Ownership and Migration Risk Assessment (LLM Native Task)
Goal: Now that the math has mapped the database topology, use the LLM to assess ownership boundaries and migration risks.
Prompt:
"You are a Database Ownership Analyst. I am providing you with database_archaeology.json, which maps every database table to every service that accesses it.
Perform the following operations:
1. Ownership Resolution: For each table with multiple services accessing it, determine the rightful owner. The owner is the service that: has migration files, defines the canonical ORM model, and performs the majority of writes. If ownership is ambiguous, flag it as CONTESTED.
2. Access Violation Detection: Identify services that WRITE to tables they do not own. These are boundary violations—the writing service is bypassing the owning service's business rules and invariants.
3. Migration Risk Scenario: For each shared table, simulate the impact of common schema changes (add NOT NULL column, rename column, change column type, drop column). For each change, list which services would break and what the failure mode would be (compile error, runtime error, silent data corruption).
4. Decoupling Recommendations: For each shared table, recommend the decoupling strategy: API-mediated access (consumers call the owner's API instead of reading the table directly), event-driven synchronization (owner publishes changes, consumers maintain local read models), or database view/synonym (owner provides a stable read-only view while retaining control of the underlying schema).
Output a database_ownership_report.md containing the resolved ownership, access violations, migration risk scenarios, and decoupling recommendations."


* Cross-Codebase Event Flow Tracing: Agents unify event topic registries across repositories and trace end-to-end business flows that span multiple services, bridging the asynchronous gaps where static analysis within any single codebase loses the thread.


Cross-Codebase Event Flow Tracing extends the single-repo Event & Data Flow Mapper (from Section 2) to the multi-service level. Within a single codebase, the agent can trace a function call from entrypoint to database sink. But when the function emits an event to a message broker and a completely different service consumes that event, the static analysis chain breaks. The consuming service has no import, no function call, no compile-time link to the producing service. The only connection is a shared string: the topic name.
In legacy systems, these event chains can span five or more services. An order placement in the commerce service emits an ORDER_CREATED event. The inventory service consumes it and emits STOCK_RESERVED. The shipping service consumes that and emits SHIPMENT_SCHEDULED. The notification service consumes that and sends a customer email. No single repository contains this end-to-end flow. Without cross-repo tracing, Effect Analysis at the system level is impossible.
Here is how to architect an agent-driven Cross-Codebase Event Flow Tracing pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM guess which services react to which events. Use deterministic extraction from each repository plus broker configuration.
* The Event Mesh (Built Previously Per Repo): Each repository's event_mesh.json identifies local publishers and consumers with their topic strings. The cross-repo tracer connects these across repositories.
* Message Broker Configuration: Kafka topic configs, RabbitMQ exchange/queue bindings, SNS/SQS subscriptions, Redis Pub/Sub channel patterns. These are the authoritative source of topic-to-consumer routing.
* Schema Registries: Confluent Schema Registry, AWS Glue Schema Registry, or local Avro/Protobuf schema files. These define the expected payload structure for each topic.
* OpenTelemetry / Distributed Tracing (If Available): Production trace data that shows the actual end-to-end flow with real timing, real service hops, and real payload sizes.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that extracts event registries from every repository, matches publishers to consumers across repo boundaries, and constructs end-to-end flow chains.
1. The Event Registry Unifier: A script that aggregates all event_mesh.json files from every repository into a single unified event registry. For each topic, it records: all publishers (with service name, file, method), all consumers (with service name, file, method), and the payload schema (if available from schema registries or code inspection).
2. The Flow Chain Builder: A script that constructs end-to-end event chains by recursive link following. Starting from an initial trigger (e.g., an HTTP endpoint in service A), it follows: service A's internal call graph to a publish() call, the published topic to all consumers in other services, each consumer's internal call graph to any further publish() calls, and so on until the chain terminates (no further events emitted).
3. The Payload Schema Comparator: A script that compares the serialized payload structure at the publish site against the deserialized payload structure at each consume site. This detects schema drift between producer and consumer, identical to API Contract Drift but for event payloads.
4. The Dead Event Detector: A script that identifies: orphaned topics (published but no consumer registered anywhere), phantom subscriptions (consumer code listening to a topic that no service publishes), and one-way topics (published and consumed but the consumer never acknowledges or processes the event—swallowed events).
3. Inputs and Outputs
Inputs to the Pipeline:
* Per-repository event_mesh.json files.
* Broker configuration files (Kafka topic configs, RabbitMQ bindings, etc.).
* Schema registry contents (if available).
* system_graph.json (for context on service relationships).
Outputs from the Pipeline:
* A cross_repo_event_flows.json file.
* Schema: * unified_registry: Every event topic with all publishers and all consumers across the system.
    * flow_chains: End-to-end business flows traced through event hops, with service sequence, topic sequence, and payload transformations at each hop.
    * payload_drift: Schema mismatches between publisher serialization and consumer deserialization for each topic.
    * orphaned_topics: Topics published but never consumed.
    * phantom_subscriptions: Topics consumed but never published.
    * flow_gaps: Points in a flow chain where the consumer does not emit any further event and does not persist data (the flow vanishes—likely a bug or dead code).
4. The Prompts
Divide the operation into mechanical registry unification and semantic flow analysis.
Phase 1: Mechanical Event Registry Unification and Chain Construction
Goal: Force the agent to aggregate event data from all repositories and construct the end-to-end flow chains using topic string matching.
Prompt:
"You are a Cross-Service Event Flow Engine. Your task is to trace end-to-end business flows that span multiple services via asynchronous event channels.
1. Load all per-repository event_mesh.json files. Aggregate into a single unified event registry. For each topic string, collect all publishers (service, file, method) and all consumers (service, file, method).
2. Load broker configuration files. Verify that every topic referenced in code actually exists in the broker config. Flag any topic referenced in code but not configured in the broker.
3. Construct flow chains: Starting from each HTTP entrypoint that eventually leads to a publish() call, recursively trace the event chain: publisher -> topic -> consumer -> consumer's internal logic -> next publish() call -> next topic -> next consumer. Continue until the chain terminates.
4. For each topic, extract the payload structure from the publisher's serialization code and compare it to the consumer's deserialization code. Flag field-level mismatches.
Output cross_repo_event_flows.json containing the unified registry, flow chains, payload drift, orphaned topics, and phantom subscriptions."
Phase 2: Business Flow Validation (LLM Native Task)
Goal: Now that the math has constructed the cross-service event chains, use the LLM to validate them against business intent and identify failure modes.
Prompt:
"You are a Distributed Flow Analyst. I am providing you with cross_repo_event_flows.json, which contains end-to-end business flows traced across multiple services via event channels.
Perform the following operations:
1. Flow Completeness Validation: For each business flow (e.g., order placement to delivery notification), verify that the chain is complete. Does it start at a user-facing entrypoint and terminate at a persistence or notification sink? If the chain has gaps (a consumer that neither persists data nor emits further events), flag the gap as a potential data loss point.
2. Ordering and Idempotency Analysis: For each flow chain, assess whether out-of-order event delivery could cause incorrect behavior. If service B must process EVENT_A before EVENT_B but both arrive on separate topics with no ordering guarantee, flag the race condition.
3. Failure Propagation Assessment: If a consumer fails to process an event (throws an exception, times out), what happens? Does the message go to a dead-letter queue? Is it retried? Is it silently dropped? Identify flows where failure at any hop results in permanent data loss.
4. Payload Evolution Risk: For each topic with payload drift, assess the impact. Does the consumer use the mismatched field in a critical path (calculation, persistence) or a non-critical path (logging, display)?
Output a cross_service_flow_report.md containing the validated flows, completeness gaps, ordering hazards, failure propagation risks, and payload evolution impact."


* Strangler Fig / Migration Path Planning: Agents analyze the dependency structure to determine the order in which legacy components can be incrementally replaced by new implementations, identifying facade insertion points, rollback boundaries, and the minimum set of changes required at each migration phase.


Strangler Fig / Migration Path Planning is the strategic capstone of the multi-codebase operations toolkit. Every preceding pipeline—structural graphing, semantic grouping, effect analysis, cross-repo mapping, database archaeology—produces diagnostic intelligence. Migration Path Planning consumes that intelligence and produces an executable plan for incrementally replacing legacy components with new implementations while keeping the system running.
The Strangler Fig pattern (named by Martin Fowler) works by: placing a facade in front of the legacy component, routing new requests to the new implementation while old requests still flow to the legacy, gradually migrating all callers to the new implementation, and finally decommissioning the legacy component. The agent must determine: where to place the facade, which components to strangle first (dependency order matters—you cannot strangle a component that is depended upon by another component you have not yet strangled), and what rollback mechanisms must exist at each phase.
Here is how to architect an agent-driven Migration Path Planning pipeline.
1. The Substrate: Existing Tools to Leverage
Do not let the LLM propose a migration sequence based on intuition. Use the dependency graph to compute the topologically valid order.
* The System Graph (Built Previously): The system_graph.json provides the inter-service dependency edges. The migration order must respect dependency direction: strangle leaf services (services with few or no dependents) before core services (services depended upon by many others).
* The Repository Graph (Built Previously): Each repository_graph.json provides the intra-service module structure. Within a service, the migration order must respect internal module dependencies.
* The Database Archaeology (Built Previously): The database_archaeology.json reveals shared-table coupling, which constrains migration sequence. A service cannot be strangled independently if it shares write access to tables with other services.
* The Cross-Repo Event Flows (Built Previously): The cross_repo_event_flows.json reveals event coupling. A service that publishes events consumed by 5 other services cannot be strangled without ensuring the new implementation publishes the same events with the same payload schema.
* Feature Flag Systems: LaunchDarkly, Unleash, or custom feature flag infrastructure. The agent must integrate with the flag system to control traffic routing between legacy and new implementations.
2. The Pipeline: Tools to Build in Python
The agent must orchestrate a pipeline that computes the migration order, identifies facade insertion points, and generates the phased migration plan.
1. The Dependency Order Calculator: A script that performs topological sort on the system graph (and internal module graphs). Leaf nodes (services/modules with no dependents) can be migrated first. The script identifies: migration-ready components (zero or minimal dependents, no shared database writes, no critical event publishers), blocked components (cannot be migrated until their dependents are migrated first or until shared state is decoupled), and circular dependencies (mutually blocking components requiring simultaneous migration or prior decoupling).
2. The Facade Point Identifier: A script that, for each component to be strangled, identifies the optimal facade insertion point. This is the narrowest interface between the legacy component and its callers. Ideally, this is an existing API boundary. If no clean boundary exists, the script identifies the minimum set of methods/endpoints that must be proxied through the facade.
3. The Migration Phase Generator: A script that groups the topologically sorted components into migration phases. Each phase contains: the components to migrate in this phase, the facade to insert, the feature flags to create, the data migration required (if any), the rollback mechanism, and the verification criteria (how to prove the new implementation matches legacy behavior).
4. The Rollback Boundary Analyzer: A script that, for each migration phase, determines whether the phase is independently rollbackable. A phase is rollbackable if: the facade can route 100% of traffic back to legacy, no irreversible data migration has occurred, and no downstream consumers have been updated to depend on new-implementation-only features.
3. Inputs and Outputs
Inputs to the Pipeline:
* system_graph.json.
* Per-repository repository_graph.json files.
* database_archaeology.json.
* cross_repo_event_flows.json.
* Target Architecture Description: A structured specification of what the new system should look like (which components are being replaced, which are being retained, what new boundaries are being drawn).
Outputs from the Pipeline:
* A migration_plan.json file.
* Schema: * phases: Ordered array of migration phases. Each phase contains: components (what is being migrated), facades (where the routing proxy is inserted), feature_flags (the flags controlling traffic split), data_migration (schema changes, data backfills), rollback_mechanism (how to revert this phase), verification_criteria (tests and metrics that prove equivalence), dependencies (which prior phases must be complete), estimated_blast_radius (how many services are affected if this phase fails).
    * blocked_components: Components that cannot be migrated in any phase until prerequisite decoupling is completed, with the specific blocker identified.
    * circular_dependencies: Mutual dependencies that require simultaneous migration or prior structural decoupling.
    * critical_path: The longest dependency chain determining the minimum total migration timeline.
4. The Prompts
Divide the operation into mechanical dependency ordering and strategic phase design.
Phase 1: Mechanical Dependency Analysis and Migration Ordering
Goal: Force the agent to compute the topologically valid migration order from the dependency graphs, respecting database and event coupling constraints.
Prompt:
"You are a Migration Sequencing Engine. Your task is to compute the valid order in which legacy system components can be incrementally replaced.
1. Load system_graph.json, database_archaeology.json, and cross_repo_event_flows.json.
2. Construct a combined constraint graph. Add edges for: service-to-service API dependencies, shared database write coupling (if services A and B both write to table X, they are mutually constrained), event publishing dependencies (a service publishing events consumed by others cannot be migrated until consumers can handle the new implementation's event format or both old and new implementations publish the same events).
3. Perform topological sort on the constraint graph. Identify leaf components (migration-ready), blocked components (with their specific blockers), and circular dependencies.
4. Group components into phases such that all components in a phase can be migrated independently of each other, and all dependencies of a phase are satisfied by prior phases.
5. For each phase, identify the narrowest interface boundary where a facade/proxy can be inserted.
Output migration_plan.json containing the phased migration order, facade insertion points, blockers, and circular dependencies."
Phase 2: Strategic Migration Phase Design (LLM Native Task)
Goal: Now that the math has determined the valid migration order, use the LLM to design the operational details of each phase including rollback mechanisms, verification criteria, and risk mitigation.
Prompt:
"You are a Migration Strategist. I am providing you with migration_plan.json, which contains the topologically valid phased migration order for the legacy system.
Perform the following operations:
1. Facade Design: For each phase, design the routing facade. Specify whether it is an HTTP reverse proxy, a library-level interface swap, a feature-flagged code path, or a database view/synonym. The facade must support: 0% new (pure legacy), gradual rollout (percentage-based traffic split), and 100% new (legacy fully decommissioned).
2. Verification Strategy: For each phase, define the equivalence tests. The new implementation must produce identical observable behavior to the legacy for all existing inputs. Specify: characterization tests to port from legacy, contract tests between facade and new implementation, shadow traffic comparison (run both legacy and new, compare outputs, alert on divergence).
3. Rollback Plan: For each phase, specify the exact rollback procedure: flip the feature flag, revert the data migration (if reversible), restore the legacy deployment. Identify any phases where rollback is impossible (irreversible data migrations) and flag these as requiring extra verification before execution.
4. Risk-Ordered Execution: Reorder the phases to minimize risk. Prefer phases that: affect fewer services, have rollbackable data migrations, have comprehensive existing test coverage, and serve lower-traffic endpoints.
5. Timeline Dependencies: Identify the critical path—the longest chain of dependent phases that determines the minimum total migration duration.
Output a migration_strategy.md containing the phase-by-phase facade designs, verification strategies, rollback plans, risk ordering, and critical path timeline."
